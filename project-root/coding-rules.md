
## 1. 适用范围与优先级

本文件用于约束本项目的代码生成、人工编写、代码修改、代码审查、联调、测试和文档同步行为。

适用范围：
- 前端代码
- 后端代码
- 数据导入与数据处理脚本
- 单元测试代码
- 与开发直接相关的配置文件
- 接口文档、数据库设计文档、模块说明文档中的实现约束

优先级规则：
1. 本文件与 `security-rules.md` 为仓库级最高优先级规则
2. 当前模块文档高于全局泛化约定
3. 已确认的项目文档高于外部参考资料
4. 若与课程要求冲突，以课程要求为准
5. 若与已冻结的数据库/API设计冲突，以已确认项目文档为准

目标：
- 保证代码可读、可测、可维护
- 保证 AI 生成代码稳定、不乱发挥
- 保证前后端、数据库、接口、文档之间的一致性
- 保证项目能按模块推进并顺利联调
- 保证核心算法要求真正落地

---

## 2. 项目总体开发原则

1. 不允许一次性生成整个系统，必须按模块逐步实现。
2. 必须先有文档上下文，再生成或修改代码。
3. 开发顺序遵循：工程骨架 → 最薄主链路 → 按模块扩展 → 联调 → 测试 → 交付。
4. 每次只实现一个模块或一个明确子功能。
5. 当前模块若依赖未完成，只生成骨架、接口定义、TODO 和待确认项，不跨模块乱写。
6. 所有课程要求中的核心功能最终都必须可演示。
7. 所有核心算法必须基于自己设计的数据结构实现，不直接把黑箱算法服务当作最终实现。
8. 算法实现必须与模块文档、数据结构说明、测试用例保持一致。
9. 修改代码后，必须检查是否需要同步数据库、API、模块、测试和进度文档。
10. 禁止为了“先跑通”而把临时代码直接留在正式实现中。

## 3. 项目结构规范

### 3.1 仓库结构

项目目录默认包含：
- `frontend/`
- `backend/`
- `scripts/`
- `data/`
- `docs/`
- `assets/`

### 3.2 后端结构规范

后端按分层组织：
- `controller`：参数接收、权限入口、调用 service、返回响应
- `service`：业务逻辑
- `service.impl`：业务实现
- `mapper` / `repository`：数据访问
- `entity`：数据库实体
- `dto` / `request`：请求对象
- `vo` / `response`：响应对象
- `config`：配置类
- `exception`：异常类与全局异常处理
- `common`：通用返回体、枚举、常量
- `util`：纯工具类
- `test`：测试代码

约束：
1. Controller 不写复杂业务逻辑。
2. Service 不直接将 Entity 返回给前端。
3. Mapper 只做数据访问，不承载业务决策。
4. DTO、VO、Entity 禁止混用。
5. 公共逻辑优先抽到公共服务层。

### 3.3 公共能力层

以下公共能力优先复用，避免重复实现：
- `QueryService`
- `RankService`
- `MapService`
- `SearchService`
- `AIService`
- `FileService`

### 3.4 前端结构规范

前端按以下方式组织：
- `pages`：页面
- `components`：复用组件
- `api`：接口请求封装
- `store`：状态管理
- `router`：路由
- `types`：类型定义
- `constants`：常量与枚举
- `utils`：工具函数

约束：
1. 页面只保留页面级逻辑。
2. 复用逻辑优先拆组件或抽公共函数。
3. 接口请求统一放在 `api`。
4. 页面字段命名与后端文档保持一致，不擅自改字段含义。
5. 页面联调前优先用 mock 数据验证结构。

### 3.5 脚本规范

1. 数据采集、清洗、导入脚本统一放在 `scripts/`。
2. 临时一次性脚本不得直接混入业务目录。
3. 每个脚本必须说明：用途、输入、输出、运行方式、依赖环境。
4. 脚本如会改数据库或覆盖文件，必须显式提示风险。

## 4. 命名规范

### 4.1 Java 命名
- 类名：`UpperCamelCase`
- 方法名：`lowerCamelCase`
- 变量名：`lowerCamelCase`
- 常量名：`UPPER_SNAKE_CASE`
- 包名：全小写，按功能分层
- 接口、类、方法命名必须清晰表达职责，不使用模糊缩写

### 4.2 后端后缀约定
- Controller：`*Controller`
- Service：`*Service`
- ServiceImpl：`*ServiceImpl`
- Mapper：`*Mapper`
- DTO / Request：`*DTO` / `*Request`
- VO / Response：`*VO` / `*Response`
- Enum：`*Enum`

### 4.3 前端命名
- Vue 组件：`UpperCamelCase`
- Store：`useXxxStore`
- API 文件：按模块命名，如 `recommend.ts`、`route.ts`
- 页面文件按业务语义命名，不使用模糊名称

### 4.4 数据库与接口命名
- 表名：小写下划线
- 主键统一：`id`
- 外键统一：`xxx_id`
- 时间字段统一：`created_at`、`updated_at`
- REST 路径优先使用复数名词，例如：
  - `/destinations`
  - `/routes`
  - `/facilities`
  - `/foods`
  - `/diaries`

## 5. 返回体、异常与错误码

1. 所有接口统一返回 `ApiResponse`。
2. 错误码统一由 `docs/04_api/error-codes.md` 管理。
3. 使用全局异常处理，不允许吞异常。
4. 业务异常与系统异常分开处理。
5. 用户可见信息应简洁明确，不返回堆栈。
6. 日志中保留排错所需上下文，但必须脱敏。
7. Controller 不直接拼装杂乱错误结构。

---

## 6. 注释与文档化规范

1. Public 类和核心方法必须有简洁注释。
2. 注释至少说明：用途、关键参数、返回结果、异常场景。
3. 不写“翻译式”废话注释。
4. 复杂算法逻辑必须说明：
   - 使用的数据结构
   - 核心步骤
   - 时间复杂度/空间复杂度
   - 适用范围
5. 核心模块的实现必须能从代码追溯到模块文档。
6. 删除或修改旧实现时，要同步清理过时注释。

---

## 7. 数据访问规范

1. 不允许字符串拼接 SQL。
2. 优先参数化查询或 ORM。
3. 分页、排序、过滤条件必须显式可读。
4. 排序字段、筛选字段必须来自白名单，不允许随意透传。
5. 复杂查询要解释用途和性能影响。
6. 批量操作、导入操作要明确事务边界和失败处理方式。
7. 查询层不要无节制返回大对象，默认限制分页大小。

---

## 8. 日志规范

1. 日志必须帮助定位问题，不写无意义日志。
2. 禁止打印 `password`、`token`、`secret`、`cookie`、手机号全量、身份证号等敏感信息。
3. 错误日志应包含模块名、操作名、关键上下文，但必须脱敏。
4. 不允许把调试日志长期留在主流程中。
5. 第三方接口失败日志要能区分：请求失败、超时、解析失败、业务拒绝。

---

## 9. 测试规范

1. 核心 service 至少提供最小单元测试样例。
2. 每个核心模块至少有：
   - 正常输入样例
   - 边界输入样例
   - 异常输入样例
3. 修复 bug 后，优先补回归测试。
4. 算法模块测试必须覆盖：
   - 典型输入
   - 极端输入
   - 空输入或非法输入
   - 性能或规模边界
5. 接口联调前，先保证最小可运行单测或 mock 验证通过。
6. 提交前至少保证本次改动相关测试通过。

---

## 10. 代码质量要求

1. 方法尽量只做一件事。
2. 避免过长方法和过深嵌套。
3. 避免复制粘贴式重复逻辑。
4. 避免魔法值，必要时提取为常量或枚举。
5. 不允许空 catch。
6. 不允许保留无用变量、无用代码、长时间注释掉的旧实现。
7. 不允许把测试数据硬编码进正式业务逻辑。
8. 不允许为了省事绕过课程要求的数据结构与算法。

---

## 11. 前端实现补充规则

1. 页面按“主任务优先”设计，不堆无关内容。
2. 推荐页、导航页、日记页应优先突出用户主任务。
3. 地图相关页面必须保留清晰的路径或节点展示能力。
4. 前端校验不能代替后端校验。
5. 组件 props、事件、状态命名要语义化。
6. 表单提交、加载中、错误提示、空状态都要有基本处理。

---

## 12. 依赖管理与提交门槛

1. 新增依赖必须说明用途、版本和作用域。
2. 不随意引入大型、过时或重复依赖。
3. 构建失败、测试失败不得合并主分支。
4. 新增工具链或框架前，先确认是否与当前技术路线冲突。
5. AI 生成代码若引入新依赖，必须显式说明原因。

---

## 13. 文档同步要求

修改以下内容时，必须同步文档：

- 表结构 → `docs/03_data/schema.md`、`data-dictionary.md`、`er-model.md`
- 接口 → `docs/04_api/api-spec.md`、必要时同步 `swagger-draft.yaml`
- 模块职责/依赖 → `docs/02_architecture/module-map.md` 与 `dependency-map.md`
- 算法方案 → `docs/09_decisions/ADR-004-algorithm-choice.md`
- 测试策略/用例 → `docs/06_testing/*`
- 进度 → `docs/00_project/progress.md`

---

## 14. 禁止事项

1. 禁止一次性生成整个系统。
2. 禁止跳过依赖模块直接实现上层功能。
3. 禁止未同步文档就修改接口或表结构。
4. 禁止为了“跑通”而写死测试数据到正式业务逻辑中。
5. 禁止把外部参考资料直接当作项目事实。
6. 禁止绕过算法要求，用简单遍历或直线距离替代规定的数据结构与算法实现。
7. 禁止把临时调试代码、临时脚本、临时文件混入正式提交。
