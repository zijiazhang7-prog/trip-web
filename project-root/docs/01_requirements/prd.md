# 产品需求文档（PRD）

## 1. 文档信息

- 项目名称：个性化旅游系统的设计与实现
- 文档名称：PRD（产品需求文档）
- 当前版本：V1.1（基础功能 + 创新需求纳入版）
- 当前状态：初稿
- 适用阶段：需求分析、概要设计与创新功能收口阶段
- 说明：本版本在 V1.0 基础功能版的基础上，补充多人协同决策、AI 智能手账、用户偏好输入、轨迹记录与推荐增强等创新需求。当前仍以基础主线闭环为优先，创新需求按当前实现、后续迭代、待评估三个层级管理。

---

## 2. 项目背景

当前用户在旅游活动中，通常会经历三个连续场景：

1. 旅游前，需要选择目的地，并根据个人兴趣、热度、评价等信息做筛选；
2. 旅游中，需要在景区或校园内部规划路线、查询场所与周边设施；
3. 旅游后，希望根据所拍照片和游览经历生成旅游日记，并进行浏览与交流。

本项目拟设计并实现一个面向旅游活动全流程的 Web 个性化旅游系统，覆盖“推荐—规划—查询—日记”主线，并在系统中体现数据结构与算法课程的核心内容。课程要求中明确包括旅游推荐、旅游路线规划、场所查询和旅游日记相关功能，且核心算法需要基于自己设计的数据结构实现。

---

## 3. 产品目标

### 3.1 总体目标
实现一个可远程访问、可演示、可继续扩展的个性化旅游系统，满足课程设计的功能要求与文档要求。

### 3.2 具体目标
1. 提供旅游前的目的地推荐与搜索能力；
2. 提供旅游中的路径规划与场所查询能力；
3. 提供旅游后的日记发布、浏览与基础检索能力；
4. 支持多用户使用、数据持久化、基础数据导入与管理；
5. 将排序、查找、图、路径规划等核心算法真正落到系统实现中；
6. 提供用户偏好输入能力，支持用户通过标签选择和自由输入表达旅游偏好；
7. 提供多人协同决策能力，辅助多人出行场景下的路线方案讨论与选择；
8. 提供 AI 辅助日记 / 手账能力，帮助用户更完整地记录一次旅行；
9. 为后续接入轨迹记录、外部地图 API、多模态 AI 生成等增强能力预留扩展空间。

## 5. 目标用户

### 5.1 普通用户
普通用户是系统的主要使用者，核心需求包括：
- 寻找适合自己的目的地
- 快速规划参观路线
- 查询周边设施和美食
- 记录并分享旅游经历

### 5.2 管理员
管理员负责维护系统基础数据，保障系统正常运行，核心需求包括：
- 管理目的地、场所、设施和美食数据
- 维护日记内容和基础信息
- 进行数据导入和基础配置管理

### 5.3 创新功能重点用户
创新功能主要面向年轻和中年旅游游客，尤其是具有自主规划能力、希望提高出行效率、重视旅游记录和多人协商体验的用户。

重点覆盖以下场景：
- 单人出行：用户希望快速获得符合个人偏好的路线和记录方式；
- 多人出行：用户希望在不同成员偏好不完全一致时，通过系统辅助达成较合理的出行方案；
- 旅行记录：用户希望将一次完整旅行整理成图文结合、具有纪念意义的手账式日记。

老年和幼年用户友好性当前暂不作为本阶段重点实现目标，后续可根据项目进度和使用反馈继续研究。

## 6. 核心用户场景

### 6.1 场景一：旅游前选择目的地
用户希望在出发前通过热度、评价、兴趣、关键词等条件快速找到合适的景点或学校，并查看推荐结果和基本信息。

### 6.2 场景二：旅游中规划路线
用户进入景区或校园后，希望输入目标场所，快速获得一条较优路线，并查看距离、预计时间和路径展示。

### 6.3 场景三：旅游中查询周边
用户在游览过程中，希望根据当前位置查找厕所、食堂、商店、超市、咖啡馆等周边设施，并优先看到可达距离较近的结果。

### 6.4 场景四：旅游后记录与浏览日记
用户在结束游览后，希望发布图文日记，浏览其他用户的内容，并按目的地找到相关日记。

### 6.5 场景五：用户填写个性化偏好

用户在使用推荐或规划功能前，可以通过预设选项和自由输入的方式填写个人偏好。  
系统根据用户选择的兴趣标签和补充描述，为推荐和路线规划提供基础输入。

典型偏好包括：
- 美食探索型
- 人文建筑型
- 自然景观型
- 繁华商城型
- 用户自定义补充偏好

### 6.6 场景六：多人出行协同决策

当用户选择多人出行时，系统可根据不同出行成员的偏好，调用多个智能体模拟讨论过程。  
每个智能体代表一位出行成员的利益和偏好，对不同路线或景点方案进行优缺点讨论。  
系统最终提取核心讨论记录、方案理由和路线建议，呈现给用户作为决策参考。

当前阶段采用“多智能体辩论 + 总结输出”的基础方案。  
可视化对比矩阵、偏好权重融合、分层决策机制作为后续增强方向。

### 6.7 场景七：旅游中实时记录与手账编辑

用户在旅游过程中到达某个地点后，可以点击“完成本地点”，并上传图片、输入文字或补充评价。  
系统将用户在多个地点产生的记录整理到同一次旅行手账中，形成比普通碎片化分享更完整的旅游记录。

当前阶段优先支持：
- 文字记录
- 图片上传
- 地点关联
- 手账式内容组织

AI 装饰、自动排版、多模态内容生成等作为后续增强方向。

### 6.8 场景八：旅游后路线回顾与手账完善

旅游结束后，用户可以继续完善本次旅行手账。  
系统可基于路线记录、地点记录和用户上传内容，生成本次旅行的路线回顾说明或轨迹图展示。

当前阶段优先在设计上预留轨迹记录和路线回顾能力，具体接入外部地图 API 或多模态生成模型时，需另行评估成本、配置、安全和稳定性。

## 7. 当前需求范围

### 7.1 当前版本覆盖范围

本版本 PRD 覆盖当前已确定的基础功能与已纳入设计范围的创新功能。

基础功能包括：

- 旅游推荐
- 目的地搜索与详情
- 路线规划
- 周边设施查询
- 美食推荐基础版
- 旅游日记基础版
- 基础后台支撑

创新功能包括：
- 用户偏好输入增强
- 多人协同决策基础版
- AI 智能手账 / 日记增强
- 轨迹记录与路线回顾预留
- 推荐和路线规划的增强维度预留

---

### 7.2 当前版本重点实现内容

当前阶段仍以基础主线闭环为优先，即：

> 旅游前推荐与搜索 → 旅游中路线规划与设施查询 → 旅游后日记记录与浏览

在此基础上，优先纳入以下创新能力的基础版本：

1. 用户偏好输入：支持预设标签选择和自由输入；
2. 多人协同决策：采用多智能体辩论并输出讨论摘要；
3. 手账日记基础形态：支持一次旅行形成较完整的图文记录；
4. 路线规划增强输入：在设计中预留时间、地点、人数、偏好等输入字段。

---

### 7.3 当前版本不展开或后续评估的内容

以下内容当前不作为基础主线阻塞项：

- 老年和幼年用户友好性专项设计；
- 用户行为记录分析；
- 完整社区互动机制；
- 评论恶意内容过滤机制；
- 复杂隐私政策与用户数据导出机制；
- 完整商业化推广与运营功能；
- AI 装饰升级、自动排版、高级多模态生成；
- 高德地图 API 或其他外部地图 API 的正式接入；
- 外部多模态 AI 模型 API 的正式接入。

以上内容可作为后续迭代或技术评估项，在基础功能稳定后再决定是否纳入实现。

## 9. 详细功能需求

## 9.1 账户与基础能力

### 9.1.1 用户注册
**功能描述**  
用户可以创建账号，进入系统使用个性化功能。

**输入**
- 用户名
- 密码
- 昵称（可选）

**输出**
- 注册成功 / 失败提示
- 用户创建结果

**基本要求**
- 用户名唯一
- 密码不能为空
- 输入格式校验明确

---

### 9.1.2 用户登录
**功能描述**  
用户使用账号密码登录系统。

**输入**
- 用户名
- 密码

**输出**
- 登录结果
- 当前用户信息
- 认证信息

**基本要求**
- 支持登录状态校验
- 对无效账号、密码错误、未登录状态有清晰处理

---

### 9.1.3 获取当前用户信息
**功能描述**  
登录后获取当前用户的基础资料，用于页面展示和个性化能力调用。

---

## 9.2 旅游推荐

### 9.2.1 目的地推荐
**功能描述**  
系统根据热度、评分、兴趣等因素，为用户推荐景点或学校。

**输入**
- 用户偏好（可为空）
- 推荐条件（热度、评分、主题、类型等的最小子集）

**输出**
- 推荐目的地列表
- 每个目的地的基础信息（名称、类型、热度、评分、标签等）

**基本要求**
- 推荐不能只支持单一维度
- 当前基础版优先支持热度、评分、兴趣的基础组合
- 推荐结果应优先输出前若干项，而不是简单对全量完全排序

**设计依据**
课堂讨论中明确提到推荐不能单一化，且建议采用“检索后二次排序”的方式，用户通常不会翻很多页。

---

### 9.2.2 目的地搜索与筛选
**功能描述**  
用户可输入名称、类别、关键字等信息查找景点或学校，并按热度或评价排序。

**输入**
- 关键字
- 类型
- 类别
- 排序方式

**输出**
- 搜索结果列表

**基本要求**
- 支持基础模糊查询
- 支持结果筛选
- 支持按热度和评分排序

**设计依据**
课程要求中明确指出，用户可以输入景点和学校名称、类别、关键字进行查询，并可对结果按热度和评价排序。

---

### 9.2.3 目的地详情
**功能描述**  
展示某个目的地的基础信息，为用户继续路线规划、查看相关日记等提供入口。

**展示内容建议**
- 名称
- 类型
- 简介
- 热度
- 评分
- 标签
- 相关操作入口

---

### 9.2.4 用户偏好输入

**功能描述**  
用户可通过预设标签和自由输入的方式表达个人旅游偏好，系统后续可将偏好信息用于目的地推荐、路线规划和多人协同决策。

**输入**
- 预设偏好标签
- 自定义偏好描述

**预设标签示例**
- 美食探索型
- 人文建筑型
- 自然景观型
- 繁华商城型

**输出**
- 用户偏好保存结果
- 当前用户偏好信息

**基本要求**
- 支持多选标签
- 支持用户补充自由文本
- 偏好信息应能与用户账号关联
- 第一阶段以用户主动填写为主

**后续增强**
- 可根据用户浏览、收藏、点击、评分等行为逐步优化偏好模型
- 行为记录分析当前不作为本阶段必须实现项

---

## 9.3 路线规划

### 9.3.1 单目标路径规划（F-MAP-001）
**功能描述**
用户输入当前位置和目标场所后，系统生成一条最优旅游路线。

**输入**
- 当前节点
- 目标节点
- 路径策略（当前基础版优先支持最短距离 F-MAP-002）

**输出**
- 路径节点列表
- 总距离
- 预计时间
- 路径可视化结果（通过 F-MAP-003 提供数据）

**基本要求**
- 使用图结构建模
- 至少支持单目标最短路径
- 结果可展示、可解释
- 支持基础异常处理（F-MAP-005）：不可达/节点不存在等场景

**设计依据**

**关联功能**
- F-MAP-002：最短距离策略（以距离为边权的 Dijkstra 算法）
- F-MAP-003：路径结果数据输出（为前端地图展示提供路径数据）

## 9.8 多人协同决策

### 9.8.1 多智能体讨论与方案总结

**功能描述**  
当用户选择多人出行时，系统可根据不同出行成员的偏好，生成或比较多个旅游方案，并通过多个智能体模拟不同成员的讨论过程。  
每个智能体代表一位出行成员的偏好和利益，对方案的优点、缺点、冲突点和折中方案进行讨论。  
系统最终整理讨论摘要、推荐方案和决策依据，供用户查看并自行决定是否采纳。

**输入**
- 出行人数
- 每位成员的偏好标签或偏好描述
- 目的地或候选景点
- 出行时间
- 用户已选择或系统推荐的路线方案

**输出**
- 多智能体讨论摘要
- 各方案优缺点说明
- 推荐方案或折中方案
- 决策依据说明

**基本要求**
- 当前采用“多智能体辩论 + 总结输出”的基础方案；
- 输出内容应可读、可解释，不能只给最终路线；
- 用户保留最终决策权；
- 该功能属于创新增强能力，不应阻塞基础路线规划和推荐主线。

**后续增强**
- 可视化对比矩阵；
- 偏好权重融合算法；
- 分层决策机制；
- 成员偏好冲突检测。

## 13. 创新需求

本项目在基础课程要求之上，补充以下创新需求方向。  
当前阶段创新需求按照“当前可纳入设计 / 后续迭代 / 待评估”进行管理，避免创新需求影响基础主线闭环。

---

### 13.1 创新需求方向一：多人协同决策

**需求背景**  
多人出行时，不同成员可能存在兴趣偏好、时间安排、预算和路线选择上的冲突。传统旅游工具通常只提供单人视角的推荐和路线规划，缺少对多人协商过程的支持。

**功能目标**  
系统通过多个智能体模拟不同成员的观点，对旅游方案进行讨论、比较和总结，帮助用户理解不同方案的优缺点，并辅助做出最终决策。

**当前方案**
- 每个智能体代表一位出行成员的偏好；
- 智能体围绕候选路线或景点方案进行讨论；
- 系统总结讨论过程、关键冲突点、折中方案和推荐理由；
- 用户查看结果后自行决定是否采纳。

**当前优先级**
- P2：创新增强能力；
- 当前先保留基础方案设计，具体实现可在基础主线稳定后推进。

---

### 13.2 创新需求方向二：AI 智能手账 / 旅行日记增强

**需求背景**  
用户在旅行结束后往往希望记录完整经历，但传统朋友圈、小红书等记录方式偏碎片化，难以完整保存一次旅行的路线、地点、图片和感受。

**功能目标**  
系统将一次旅行组织成一本手账式记录，支持用户在旅行中实时记录，在旅行后继续完善，并可借助 AI 生成标题、摘要、正文草稿和内容标签。

**当前基础形态**
- 用户到达地点后可记录文字、图片和评价；
- 系统将多个地点记录组织为一次旅行手账；
- 用户可继续编辑和完善；
- 支持公开 / 私有状态设计。

**后续增强**
- AI 日记草稿生成；
- AI 摘要与标签；
- 图片理解与多模态内容生成；
- 手账装饰、自动排版和成就反馈。

**当前优先级**
- P1：手账式记录基础形态；
- P2：AI 生成与多模态增强。

---

### 13.3 创新需求方向三：个性化推荐增强

**需求背景**  
仅依赖热度和评分的推荐容易同质化，无法充分体现用户的兴趣差异。

**功能目标**  
系统通过用户主动填写偏好标签和自由描述，为推荐、路线规划和多人协同决策提供个性化输入。

**当前方案**
- 第一阶段采用用户主动填写偏好；
- 预设偏好标签包括美食探索型、人文建筑型、自然景观型、繁华商城型等；
- 用户可多选，也可补充自定义偏好描述。

**后续增强**
- 根据用户浏览、收藏、点击、评分等行为进行偏好分析；
- 引入更复杂的推荐理由生成和路线解释。

**当前优先级**
- P0：基础偏好标签输入；
- P1：自由偏好描述；
- P2：行为分析与 AI 个性化推荐增强。

---

### 13.4 创新需求方向四：轨迹记录与路线回顾

**需求背景**  
用户在旅行结束后希望回顾自己实际走过的路线，而不仅仅是查看系统原先规划的路线。

**功能目标**  
系统后续可接入地图 API 或定位能力，记录用户实际移动轨迹，并生成路线回顾图或简笔画轨迹地图。

**当前方案**
- 当前阶段仅进行需求和架构预留；
- 后续如需接入高德地图 API，应单独评估接入成本、调用限制、定位权限、数据安全和失败降级方案。

**当前优先级**
- P2：后续增强能力。

## 14. 当前版本优先级说明

### P0（当前必须优先完成）
- 用户注册 / 登录
- 获取当前用户信息
- 用户偏好标签输入
- 推荐目的地
- 搜索目的地
- 单目标路线规划
- 周边设施查询
- 发布图文日记
- 浏览日记
- 按目的地查看日记
- 基础数据库、接口、上传与异常支撑

### P1（基础闭环跑通后优先补充）
- 目的地详情完善
- 多目标路径规划
- 美食推荐基础版
- 日记检索与排序
- 自由偏好描述
- 手账式旅行记录基础形态
- 管理端完善
- 景点评分与评论基础能力
- 路线规划增强维度预留

### P2（后续创新与增强）
- 多智能体旅游规划协商
- AI 日记草稿生成
- 多模态日记生成
- AI 手账装饰与自动排版
- 用户行为记录分析
- AI 个性化推荐增强
- 轨迹记录与路线图回顾
- 外部地图 API 接入
- 外部多模态 AI 模型 API 接入
- 复杂社区互动与恶意评论过滤
