# 术语表（Glossary）

## 1. 文件用途
本文件用于统一项目中常见术语、缩写、模块名称和关键概念的含义，减少需求分析、架构设计、接口定义、测试和协作过程中的理解偏差。

## 2. 使用原则
1. 本文件关注的是“项目中会反复出现、容易混淆或需要统一”的术语。
2. 定义尽量简洁、清楚、适合课程设计场景。
3. 如果某个术语在项目中有特定含义，应以本文件定义为准。
4. 新增模块、接口风格、算法名或协作概念时，可继续补充。

---

## 3. 基础项目术语

### 3.1 个性化旅游系统
指本项目拟开发的 Web 系统，目标是覆盖旅游前推荐、旅游中规划与查询、旅游后日记生成与交流等完整场景。

### 3.2 课程设计
指本次《数据结构》课程中的综合实践项目，要求结合软件开发过程和数据结构与算法知识，实现一个可运行的系统，并提交完整文档、代码、测试和演示材料。

### 3.3 Web 远程系统
指通过浏览器访问、支持远程使用的系统形态，不是只能在单机本地运行的桌面程序。课堂要求建议采用 Web/App 远程系统，而不是单机版。

### 3.4 B/S 架构
Browser / Server 架构，即浏览器 / 服务器架构。前端通过浏览器访问后端服务，适合本项目的 Web 远程系统形态。课堂讨论中也明确建议采用 B/S 模式。

### 3.5 前后端分离
前端负责页面与交互，后端负责业务逻辑、数据访问和接口提供，二者通过 API 通信。

### 3.6 单体后端
指后端作为一个整体应用进行开发和部署，而不是拆成多个微服务。当前项目规模更适合单体后端。

### 3.7 模块化
指系统按功能划分为若干相对独立的模块，例如认证、推荐、路线规划、日记等模块，并通过清晰接口进行协作。

## 4. 功能相关术语

### 4.1 旅游推荐
系统根据用户偏好、热度、评分、类别等因素，对景点或学校进行推荐的功能。

### 4.2 目的地
指用户计划前往旅游或参观的整体对象，例如景区或学校。

### 4.3 场所
指目的地内部的具体位置或功能点，例如景点、教学楼、食堂、办公楼、厕所、商店等。

### 4.4 周边设施查询
指用户在当前位置附近查找厕所、食堂、超市、咖啡馆等设施的功能。

### 4.5 美食推荐
指根据位置、热度、评价、菜系或用户偏好，推荐餐饮相关信息的功能。

### 4.6 旅游日记
指用户在旅游结束后，以文字、图片等形式记录和分享旅行经历的内容。

### 4.7 多模态日记
指结合文字、图片、路线信息等多种数据生成或展示的旅游日记形式。

### 4.8 多人旅游规划
指多个用户或多个偏好参与同一份路线或行程方案的规划过程，可结合 AI 协助模拟协商与决策。

## 5. 数据结构与算法术语

### 5.1 排序算法
用于将景点、学校、日记、美食等对象按热度、评分或综合得分排序的算法。课程要求推荐功能需体现排序能力。

### 5.2 查找算法
用于根据名称、类别、关键字等条件查询景点、学校、日记等对象的算法。课程要求查询功能需体现查找能力。

### 5.3 Top-K
指不对全部数据做完全排序，而是优先得到前 K 个最优结果的方法。课堂讨论中提到，用户通常不会翻很多页，推荐不应简单做全量排序。

### 5.4 图（Graph）
用于表示点与点之间关系的数据结构。在本项目中，目的地内部道路网络可抽象为图。

### 5.5 有向图
边有方向的数据结构。课堂讨论中提到，本项目地图更适合建模为有向图，因为双向道路的拥挤度或通行情况可能不同。

### 5.6 节点（Node）
图中的顶点。在本项目中可对应路口、建筑、设施点位等。

### 5.7 边（Edge）
图中连接两个节点的关系。在本项目中可对应道路、通道或可通行路径。

### 5.8 最短路径算法
用于从起点到终点寻找代价最小路径的算法，是路线规划模块的核心。课程要求中明确包含最短路径算法。

### 5.9 多点路径规划
指从当前位置出发，依次访问多个目标点并生成整体路线的算法问题。课程要求中明确包含途经多点的最短路径问题。

### 5.10 A*
一种启发式路径搜索算法。课堂讨论中提到，可将 A* 作为路线规划的候选算法之一。

### 5.11 全文检索
指按日记正文、说明文字等内容进行关键词搜索的能力。

### 5.12 压缩存储
指对日记内容、图片或相关资源进行更节省存储空间的处理。课堂讨论中提到，日记平台可能涉及压缩存储。

## 6. 系统设计与接口术语

### 6.1 API
Application Programming Interface，即应用程序接口。前端通过 API 调用后端能力。

### 6.2 RESTful
一种常见接口设计风格，通常按资源组织路径，使用 GET、POST、PUT、DELETE 等方法表示操作。

### 6.3 返回体（ApiResponse）
指后端对接口请求返回的统一数据结构，用于统一成功、失败、错误码和消息格式。

### 6.4 DTO
Data Transfer Object，请求或传输过程中使用的数据对象，一般用于接收前端参数。

### 6.5 VO
View Object，返回给前端展示使用的数据对象。

### 6.6 Entity
与数据库表结构映射的数据对象。

### 6.7 Mapper / Repository
负责数据访问的层，用于操作数据库，不直接承载复杂业务逻辑。

### 6.8 Service
负责业务逻辑处理的层。

### 6.9 Controller
负责接收请求、调用业务层并返回结果的接口入口层。

## 7. 测试与协作术语

### 7.1 单元测试
对单个类、方法或模块进行的小范围测试，用于验证最基本的正确性。

### 7.2 黑盒测试
只关注输入和输出，不关心内部实现过程的测试方式。课堂要求中明确建议覆盖黑盒测试。

### 7.3 白盒测试
结合程序内部逻辑、分支和路径进行的测试方式。课堂要求中明确建议覆盖白盒测试。

### 7.4 回归测试
在修复问题或修改功能后，为确认原有功能未被破坏而进行的测试。

### 7.5 Bug
系统中的缺陷或错误，包括功能错误、逻辑错误、界面问题、性能问题等。

### 7.6 联调
前端、后端、数据库或多个模块之间进行联合验证和问题修正的过程。

### 7.7 风险
指可能影响项目进度、质量、协作或验收结果的不确定因素。

### 7.8 周报
用于记录每周已完成工作、产出、问题和下周计划的项目管理文档。

## 8. AI 与协作相关术语

### 8.1 AI 辅助开发
指使用大模型、智能体或相关工具辅助完成需求分析、系统设计、代码生成、测试整理和文档润色等工作方式。课堂要求中也鼓励使用智能体进行系统设计与开发。

### 8.2 智能体（Agent）
能够围绕特定目标进行任务拆解、上下文读取、工具调用和结果输出的 AI 工作单元。

### 8.3 Prompt
给 AI 的任务说明或提示语，用于约束其输出目标、格式、风格和边界。

### 8.4 Context
AI 在当前任务中读取和依赖的项目上下文，包括需求文档、架构文档、模块文档、规则文件等。

### 8.5 项目级规则文件
指`project-root/`根目录中的 `coding-rules.md`、`security-rules.md`、`AGENTS.md`、`CLAUDE.md` 等仓库级说明文件，用于约束 AI 和人工协作行为。
