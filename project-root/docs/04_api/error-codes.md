# 错误码说明（Error Codes）

## 1. 文件用途
本文件用于统一说明个性化旅游系统项目中各类接口错误码的命名方式、使用原则、返回格式和具体错误码定义，为以下工作提供依据：

- 后端接口开发
- 前后端联调
- 测试用例编写
- 缺陷记录与定位
- 日志分析
- 验收与答辩说明

本文件关注的是“接口出错时怎么统一表达”，不替代接口文档本身，也不替代全局异常处理实现。

相关文档：
- `docs/04_api/api-spec.md`
- `docs/06_testing/test-cases.md`
- `docs/06_testing/bug-log.md`
- `coding-rules.md`
- `security-rules.md`

---

## 2. 设计目标

本项目错误码设计的目标如下：

1. 让前端能够根据错误码区分错误类型；
2. 让后端在不同模块中保持统一返回口径；
3. 让测试和缺陷记录时能精确引用错误场景；
4. 让日志排查时能快速定位问题来源；
5. 保证错误信息对用户友好、对开发可定位、对安全不过度泄露内部实现。

---

## 3. 使用原则

## 3.1 统一返回，不直接抛裸异常
所有可预期的业务错误，应尽量转换为统一错误码和统一错误消息返回给前端，不应直接把原始异常栈暴露给前端。

## 3.2 错误码稳定，错误消息可优化
- 错误码应尽量保持稳定
- 错误消息文案可以优化
- 前端逻辑判断优先依赖 `code`，不要依赖 `message`

## 3.3 一类问题一个明确错误码
不要把很多不同问题都混成同一个“系统错误”。

例如：
- 用户名已存在
- 密码错误
- token 无效
- 节点不可达
- 文件类型非法

这些都应有不同错误码。

## 3.4 用户可见信息与内部日志分离
返回给前端的 `message` 应简洁、可理解；  
更细的异常细节应写入后端日志，而不是直接返回给用户。

## 3.5 安全优先
对于认证、权限、上传、外部调用等敏感场景：
- 不返回敏感配置
- 不返回堆栈
- 不返回数据库表名、SQL 语句、密钥等内容

---

## 4. 统一返回格式建议

建议所有接口统一使用如下返回结构：

```json id="api_error_response_example"
{
  "success": false,
  "code": "AUTH_003",
  "message": "未登录或登录状态已失效",
  "data": null,
  "requestId": "20260419-abc123"
}
````

字段说明：

|字段|说明|
|---|---|
|`success`|是否成功，失败时固定为 `false`|
|`code`|错误码，供前后端和测试定位使用|
|`message`|面向前端和用户的简洁错误信息|
|`data`|失败场景下通常为 `null`|
|`requestId`|可选，用于日志排查与链路追踪|

## 5. 错误码命名规范

## 5.1 命名格式

建议统一采用：

`模块前缀_三位数字编号`

例如：

- `AUTH_001`
- `ROUTE_003`
- `DIARY_002`

## 5.2 模块前缀建议

|模块|前缀|
|---|---|
|通用|`COMMON`|
|认证|`AUTH`|
|推荐|`REC`|
|路线规划|`ROUTE`|
|周边设施|`FAC`|
|美食|`FOOD`|
|日记|`DIARY`|
|管理端|`ADMIN`|
|文件上传|`FILE`|
|数据导入|`IMPORT`|
|AI|`AI`|

## 5.3 编号规则

- `001 ~ 099`：参数 / 输入类错误
- `100 ~ 199`：权限 / 状态类错误
- `200 ~ 299`：资源不存在 / 业务冲突
- `300 ~ 399`：计算 / 处理失败
- `900 ~ 999`：未知或兜底系统错误

这个规则不是强制，但建议尽量遵守，方便后期维护。

## 6. HTTP 状态码与业务错误码的关系

建议同时使用：

- **HTTP 状态码**：表示请求的大类结果
    
- **业务错误码 `code`**：表示具体错误原因
    

### 建议映射关系

|HTTP 状态码|含义|典型场景|
|---|---|---|
|400|请求参数错误|参数缺失、格式非法、排序字段非法|
|401|未登录 / token 无效|token 缺失、token 过期、token 无效|
|403|权限不足|普通用户访问管理端|
|404|资源不存在|日记不存在、节点不存在、目的地不存在|
|409|资源冲突|用户名已存在、重复评分|
|422|业务校验失败|节点不可达、文件类型不允许|
|500|服务器内部错误|未预期异常|
|502 / 503|外部服务异常|外部 AI 服务调用失败|

说明：

- 项目内部也可以全部统一返回 200，再靠 `success/code` 区分；  
    但从接口规范和调试清晰度来看，更推荐同时正确设置 HTTP 状态码。

## 18. 错误码使用建议

## 18.1 前端使用建议

前端应根据 `code` 做以下处理：

1. `AUTH_003 / AUTH_004`
    - 跳转登录页或提示重新登录
2. `AUTH_005 / ADMIN_001`
    - 提示权限不足，不展示后台内容
3. `COMMON_001 / COMMON_002`
    - 高亮表单错误或参数错误提示
4. `ROUTE_003 / FAC_004`
    - 提示不可达，不要显示系统崩溃页
5. `FILE_002 / FILE_003 / FILE_004`
    - 在上传组件附近直接提示具体原因
6. `AI_002 ~ AI_007`
    - 提示 AI 生成失败，但保留人工操作入口

---

## 18.2 后端使用建议

后端应尽量做到：

1. 模块内部先抛业务异常，再由全局异常处理统一转换；
2. 不要在 Controller 里到处手写错误返回；
3. 相同场景尽量复用相同错误码；
4. 不要一个问题多套错误码；
5. 兜底错误统一收敛到各模块 `_999` 或通用 `COMMON_006` 也可以，但尽量少用。

---

## 18.3 测试使用建议

测试文档和 bug 记录中建议直接引用错误码，例如：

- `TC-AUTH-006` 预期返回 `AUTH_003`
- `TC-ROUTE-003` 预期返回 `ROUTE_003`
- `TC-IMPORT-002` 预期返回 `FILE_002`

这样测试和 bug 追踪会更清晰。
