# 技术栈说明（Tech Stack）

## 关联决策记录

- `project-root/docs/09_decisions/ADR-002-mvp-scope.md`

## 1. 文档用途
本文件用于说明本项目当前采用的技术栈、各技术的职责、选型理由和当前使用边界，为系统设计、开发协作、环境搭建、后续答辩说明提供统一依据。

本文件主要回答以下问题：

- 当前项目采用了哪些核心技术
- 每项技术在系统中负责什么
- 为什么选择这些技术
- 当前阶段不采用哪些替代方案
- 技术栈与课程要求、项目规模是否匹配

---

## 2. 技术栈选型原则

本项目技术选型遵循以下原则：

### 2.1 先满足课程要求
课程设计要求系统真实可用，并覆盖旅游推荐、旅游路线规划、场所查询、旅游日记等核心功能，同时鼓励使用智能体参与设计和开发。

### 2.2 先保证可实现
当前项目为 3 人协作的课程设计项目，因此技术栈必须控制复杂度，优先选择：
- 学习资料多
- 社区成熟
- 环境好搭
- 易于联调
- 便于 AI 工具理解和生成代码

### 2.3 先支撑最小闭环
技术栈要能先支撑“推荐—规划—查询—日记”最小闭环，而不是为了炫技引入过重技术。

### 2.4 先稳定，后扩展
优先选择成熟技术，避免在课程周期内承担过高框架试错成本。

> 本文件中的系统形态、前后端技术、数据库、数据处理工具和 AI 接入方式，
> 以 `ADR-001-tech-stack.md` 为正式决策依据。
> 当前技术栈服务于 `ADR-002-mvp-scope.md` 中冻结的基础主线范围。

### 2.5 运行时外部 API 后置接入
对于多模态模型 API、地图 API 这类运行时外部服务，当前阶段先在架构和模块上保留统一接入位置，不提前绑定具体厂商。

原因是：
- 5.10 前优先目标是跑通 MVP 主线；
- 当前尚未最终确定具体服务商；
- 过早绑定外部 API 会增加调试、配置和降级复杂度；
- 当前内部 `MapService` 和 `AIService` 已足够承接后续外部能力接入。

## 4. 前端技术栈

## 4.1 Vue 3

### 作用
Vue 3 用于实现系统前端页面，包括：
- 首页
- 推荐页
- 路线规划页
- 周边设施页
- 美食页
- 日记页
- 管理端页面

### 选型理由
1. 组件化开发清晰，适合页面和模块拆分；
2. 学习和上手成本相对适中；
3. 与 Element Plus、Pinia、Axios 组合成熟；
4. 适合课程设计项目快速搭建可运行页面；
5. 适合 AI 工具生成基础页面结构。

### 当前定位
Vue 3 是本项目前端主框架，不考虑更换为 React 或其他框架。

---

## 4.2 Vite

### 作用
Vite 用于前端工程初始化、开发服务器和构建流程。

### 选型理由
1. 启动快，开发体验较轻量；
2. 与 Vue 3 官方生态结合自然；
3. 对课程项目来说配置成本低。

### 当前定位
Vite 作为前端默认构建工具，不额外引入更复杂的前端工程体系。

---

## 4.3 Element Plus

### 作用
Element Plus 作为前端 UI 组件库，提供：
- 表单
- 表格
- 菜单
- 对话框
- 分页
- 上传
- 标签页等基础组件

### 选型理由
1. 可快速搭建管理端和用户端基础页面；
2. 组件丰富，适合课程设计项目节省 UI 手写成本；
3. 与 Vue 3 兼容良好；
4. 更适合当前“功能优先、视觉其次”的项目目标。

### 当前定位
Element Plus 是当前前端 UI 主组件库，不单独引入复杂设计系统。

---

## 4.4 Pinia

### 作用
Pinia 用于管理前端共享状态，例如：
- 当前用户信息
- 登录状态
- 当前选中的目的地
- 页面共享筛选条件

### 选型理由
1. 与 Vue 3 配套；
2. 结构相对清晰；
3. 适合当前规模项目；
4. 上手成本低于复杂状态管理方案。

---

## 4.5 Axios

### 作用
Axios 用于前端调用后端 REST API。

### 选型理由
1. 社区成熟；
2. 使用方式清晰；
3. 便于统一封装请求拦截、错误处理和 token 注入；
4. 适合课程项目快速联调。

---

## 5. 后端技术栈

## 5.1 Java 17

### 作用
Java 17 作为后端主要开发语言，用于实现：
- 业务逻辑
- 接口服务
- 数据访问
- 安全校验
- 算法模块封装

### 选型理由
1. 当前项目已经按 Java / Spring Boot 路线规划；
2. 与 Spring Boot 3 兼容；
3. 生态成熟、文档多；
4. 更适合写清晰的分层后端结构；
5. 便于后续围绕后端岗位方向继续积累。

### 当前定位
Java 17 为当前后端默认语言，不考虑中途切换为 Python 后端或 Node.js 后端。

---

## 5.2 Spring Boot 3

### 作用
Spring Boot 3 是后端主框架，用于：
- 启动后端应用
- 管理依赖注入
- 提供 REST API
- 组织 Controller / Service / Config / Exception 等分层
- 提供统一配置入口

### 选型理由
1. 适合快速搭建单体后端；
2. 生态成熟，适合课程项目；
3. 便于组织清晰的分层结构；
4. 与 Java 17 和 MyBatis-Plus 搭配成熟；
5. 便于 AI 工具理解和生成常见工程结构。

### 当前定位
Spring Boot 3 作为后端核心框架，不引入微服务体系。

---

## 5.3 MyBatis-Plus

### 作用
MyBatis-Plus 用于辅助数据库访问，承担：
- 基础 CRUD
- 条件查询
- 分页
- 简化 Mapper 编写

### 选型理由
1. 比纯 JDBC 更高效；
2. 比从零手写 MyBatis 配置更省时间；
3. 对课程项目的表结构和查询需求足够；
4. 仍然保留较强的 SQL 可控性；
5. 适合做结构化数据为主的单体后端项目。

### 当前定位
MyBatis-Plus 用于数据库访问层，不上升到复杂数据访问框架组合。

---

## 5.4 Maven

### 作用
Maven 用于后端项目的：
- 依赖管理
- 构建
- 测试运行
- 打包

### 选型理由
1. Java 项目标准构建工具之一；
2. 与 Spring Boot 项目配合成熟；
3. 有利于规范依赖与构建流程。

---

## 6. 数据与存储技术栈

## 6.1 MySQL 8

### 作用
MySQL 8 用于存储系统中的结构化数据，包括：
- 用户
- 目的地
- 场所
- 地图节点和边
- 周边设施
- 美食
- 日记
- 评分
- 路线历史等

### 选型理由
1. 课程项目明确要求数据持久化和批量导入能力；
2. 结构化数据为主，适合关系型数据库；
3. 学习资料和使用经验多；
4. 与 Spring Boot + MyBatis-Plus 结合成熟；
5. 便于后续画 ER 图、写数据字典和接口文档。

### 当前定位
MySQL 8 作为主数据库，不引入 Redis、Elasticsearch 等额外中间件作为首发依赖。

---

## 6.2 文件存储

### 作用
文件存储用于保存：
- 日记图片
- 视频（如后续需要）
- 地图 JSON / CSV
- 导入文件
- 其他附件

### 选型理由
1. 日记模块天然需要媒体文件支持；
2. 地图与导入数据文件不适合直接全部塞入数据库；
3. 对课程项目来说，本地统一文件目录比对象存储更简单、更易演示。

### 当前定位
当前阶段使用本地文件目录或统一文件存储方案，不引入云对象存储作为基础依赖。

---

## 6.3 外部地图服务（后续评估）

### 作用
后续如需接入地图 API，可用于：
- 真实导航增强
- 轨迹记录
- 路线图回顾
- 地图展示增强

### 当前边界
1. 当前路线规划与周边设施查询仍以内部图数据和 `MapService` 为主；
2. 外部地图服务不替代当前主路径算法；
3. 在未确定具体地图 API 前，只保留统一接入位置，不直接绑定某一家服务商；
4. 若后续正式接入，应在外部 API 管理文档中记录配置变量、限流、失败降级和调用方式。

### 当前定位
属于 P2 / 后续增强，不纳入当前 MVP 的硬性依赖。

## 7. 数据处理与导入技术栈

## 7.1 Python

### 作用
Python 主要用于离线数据处理层，承担：
- 爬取 / 抽取原始数据
- 清洗与格式转换
- 生成可导入文件
- 辅助批量导入

1. Python 在脚本与数据处理场景中更高效；
2. 更适合快速写清洗脚本和导入脚本；
3. 与 pandas 搭配适合处理中小规模结构化数据；
4. 将离线数据处理和线上业务后端分开，更清晰。

## 9. AI 与智能体工具

## 9.1 Claude Code / OpenClaw / ChatGPT / Codex

### 作用
这些工具主要用于辅助：
- 需求分析
- 架构设计
- 模块拆分
- 代码生成
- 测试用例整理
- 文档润色与统一格式

### 选型理由
1. 课程明确鼓励使用智能体参与系统设计、开发、测试和文档整理。
2. 当前项目文档驱动开发方式适合 AI 工具参与；
3. 可提升文档整理和基础代码搭建效率。

AI 工具是“辅助开发工具”，不是“替代课程要求的黑箱实现”。  
特别是在核心算法部分，AI 只能辅助理解、生成草稿和整理文档，最终实现仍需符合课程对数据结构和算法的要求。

## 9.2 运行时 AI 模型能力（后续评估）

### 作用
运行时外部模型 API 后续可用于：
- AI 日记草稿生成
- 图片摘要与标签提取
- 推荐理由解释
- 多人协同决策说明
- 路线回顾说明

3. 外部模型 API 属于 P2 增强能力，不阻塞当前 MVP 主线；
4. 若后续正式接入，应在独立的外部 API 管理文档中记录变量名、调用限制、降级策略和安全边界。

## 10. 当前不采用的方案及原因

## 10.1 不采用微服务
### 原因
- 项目规模较小
- 团队仅 3 人
- 联调和部署成本会显著增加
- 不利于课程周期内快速形成闭环

## 10.2 不采用复杂中间件体系
例如：
- 消息队列
- 分布式配置中心
- 分布式注册发现
- 复杂权限平台

### 原因
- 当前功能不需要
- 维护成本高
- 会分散课程重点

## 10.3 不采用 Redis / Elasticsearch 作为首发依赖
### 原因
- 当前阶段 MySQL + 统一查询能力基本可支撑 MVP
- 引入额外中间件会提升环境复杂度
- 当前目标是先跑通基础闭环，而不是追求大型系统架构

## 10.4 不采用桌面端 / 单机程序形态
### 原因
- 课堂建议优先采用 Web / App 远程形态
- Web 更利于演示和协作
- 更符合 B/S 架构路线
