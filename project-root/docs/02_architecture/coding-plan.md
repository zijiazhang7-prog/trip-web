# 开发执行方案（Coding Plan）

## 1. 文档用途
本文件用于说明本项目的具体开发推进方式，包括：

- 整体开发策略
- 开发阶段划分
- 模块实现顺序
- 三人协作方式
- AI 工具使用方式
- 每个阶段的输入、输出和验收标准
- 当前阶段应优先完成的内容

本文件不负责描述详细需求本身，而是回答：

> “我们应该怎么把这个项目真正做出来？”

---

## 2. 开发总原则

本项目开发遵循以下原则：

### 2.1 先文档，后实现
先明确需求、架构、模块边界、数据库和接口，再开始写代码。  
避免边写边改方向，导致前后端和文档全部返工。

### 2.2 先最小闭环，后增强扩展
优先完成“推荐—规划—查询—日记”这条最小可运行主线，再补美食增强、多目标路径、检索增强和创新功能。

### 2.3 先底层能力，后页面联调
优先完成数据库、统一返回、认证、查询、排序、图建模、文件上传等底层能力，再做业务模块和前端页面联调。  
这样更符合当前系统依赖关系。

### 2.4 一次只推进一个明确子目标
不一次性生成整个系统，不同时并行推进过多高耦合模块。  
每次推进一个模块或一个明确子功能，保证问题可定位、可回滚、可复盘。

### 2.5 AI 辅助但不替代设计判断
AI 工具用于辅助需求分析、模块设计、代码草稿、测试整理和文档润色；  
但数据库、接口、模块职责、核心算法实现，必须以项目文档和课程要求为准。

### 2.6 先 P0，后 P1，再 P2
当前开发必须严格遵循已冻结的优先级：

- **P0**：当前必须先实现并联调的主线能力
- **P1**：基础闭环跑通后补充的增强能力
- **P2**：创新与外部能力增强

不得把 P1 / P2 能力默认提升为当前主线阻塞项。

- 日记评分
- 日记检索与排序

形成一套：
- 可运行的 Web 系统

## 4. 当前推荐开发路线

本项目建议采用以下路线：

> 文档收口  
> → 工程骨架  
> → 数据与底层能力  
> → 公共能力层  
> → P0 核心业务模块  
> → 前后端联调  
> → P1 增强模块  
> → 测试修复  
> → P2 创新扩展与交付整理

这一路线和当前系统依赖关系是一致的，能够减少后期返工。

#### UserPreference

- 推荐目的地

- 查询附近设施

#### Diary
- 发布图文日记

### 阶段验收标准
- 至少一条完整用户主线可跑通

4. 每周至少一次同步当前状态和阻塞点。
5. 公共能力改动必须先同步相关协作人。

## 8. AI 工具使用计划

## 8.1 适合用 AI 的任务
- 需求整理
- 架构文档初稿
- 模块说明文档初稿
- 接口草稿
- 基础页面骨架
- 基础 CRUD 代码草稿
- 测试用例整理
- 周报和进度文档整理
- 答辩材料润色

## 8.2 不应直接交给 AI 决定的内容
- 最终数据库结论
- 最终接口字段定义
- 最终模块边界
- 核心算法是否符合课程要求
- 创新功能是否纳入当前阶段
- 文档冲突时的最终裁决

## 8.3 AI 使用原则
1. 使用 AI 前先给足项目文档上下文。
2. 一次只让 AI 处理一个明确任务。
3. AI 输出必须回到项目规则文件下审查。
4. 涉及数据库、接口、算法和模块职责时，必须人工复核。
5. AI 更适合生成草稿，不直接当最终事实。

## 11. 风险控制建议

在开发执行过程中，建议重点关注以下风险：

1. 需求迟迟不收敛，导致后面数据库和接口反复改；
2. 一开始就并行写太多模块，导致依赖断裂；
3. 页面先写太快，后端和数据库跟不上；
4. AI 一次性生成太多代码，后续无法定位问题；
5. 创新功能过早插入，拖垮基础主线；
6. 测试和答辩准备被压缩到最后阶段。

这些风险应同步记录到 `risk-log.md` 中持续跟踪。
