# 模块划分说明（Module Map）

## 关联决策记录

- `docs/09_decisions/ADR-003-module-split.md`

## 1. 文档用途
本文件用于说明系统的模块划分方式、各模块职责、模块边界、输入输出和依赖关系，为后续开发分工、接口设计、数据库设计和测试设计提供统一依据。

本文件关注的问题包括：
- 系统当前包含哪些核心模块
- 每个模块负责什么，不负责什么
- 模块之间如何协作
- 哪些模块属于当前 MVP 优先实现范围
- 哪些模块属于后续增强或创新扩展范围

---

## 2. 模块划分原则

本项目模块划分遵循以下原则：

1. **围绕业务主线划分**  
   按“推荐—规划—查询—日记”主线拆分核心模块。

2. **职责单一、边界清晰**  
   每个模块负责一个相对独立的业务域，避免职责重叠。

3. **公共能力抽取**  
   查询、排序、图建模、检索、文件访问等能力优先抽取为公共服务层，而不是分散在多个模块中重复实现。

4. **先主线后增强**  
   优先支持课程要求的核心功能，再逐步补充增强能力和创新功能。课程要求中的核心条目包括旅游推荐、路线规划、场所查询和旅游日记。

5. **便于三人协作**  
   模块划分需要服务于小组分工，保证每个成员能围绕若干模块形成相对完整的小闭环。

> 本文件中的模块划分、模块边界和优先级，以以下决策记录为准：
> - MVP 范围：`ADR-002-mvp-scope.md`
> - 模块拆分：`ADR-003-module-split.md`
> - 核心算法归属：`ADR-004-algorithm-choice.md`

## 3. 模块总览

当前系统按以下模块进行划分：

### 3.1 业务模块
- Auth 模块（认证与用户基础）
- Recommend 模块（推荐与目的地搜索）
- Route 模块（路线规划）
- Facility 模块（周边设施查询）
- Food 模块（美食推荐）
- Diary 模块（旅游日记）
- Admin 模块（管理端数据维护）

### 3.2 公共能力模块
- QueryService（统一查询）
- RankService（排序 / Top-K）
- MapService（图建模 / 路径规划）
- SearchService（全文检索）
- FileService（文件上传 / 文件访问）
- AIService（AI 辅助能力）
- ImportService（数据导入）

### 3.3 基础支撑模块
- Security / JWT
- Config / Exception / Response
- GraphEngine / IndexEngine / CompressionEngine / ImportEngine

## 5. 核心业务模块说明

## 5.1 Auth 模块

### 5.1.1 模块目标
为系统提供多用户基础能力，包括账号注册、登录、当前用户信息获取和最小权限区分。

### 5.1.2 主要职责
- 用户注册
- 用户登录
- 获取当前用户信息
- 基础 token 校验
- 管理端基础角色校验入口

### 5.1.3 不负责的内容
- 不负责复杂用户画像建模
- 不负责复杂权限中心
- 不负责社交关系管理

### 5.1.4 主要输入
- 用户名
- 密码
- 昵称等基础资料

### 5.1.5 主要输出
- 登录状态
- 用户基础信息
- 认证结果

### 5.1.6 依赖关系
- 依赖 `UserMapper`
- 依赖 `Security / JWT`
- 依赖统一异常处理和统一返回体

### 5.1.7 当前优先级
P0

### 5.1.8 与课程要求的关系
系统需要支持多用户使用，这是后续推荐、日记、管理能力的基础。

---

## 5.2 Recommend 模块

### 5.2.1 模块目标
为用户提供旅游前的目的地推荐、搜索和筛选能力。

### 5.2.2 主要职责
- 推荐景点 / 学校
- 按热度、评分等进行排序
- 关键字搜索
- 类型、主题等基础筛选
- 输出推荐结果列表

### 5.2.3 不负责的内容
- 不直接负责路径规划
- 不直接负责日记生成
- 不直接负责复杂用户画像训练

### 5.2.4 主要输入
- 关键字
- 类型 / 类别
- 排序方式
- 用户偏好（基础版可选）

### 5.2.5 主要输出
- 推荐列表
- 搜索结果列表
- 目的地基础信息

### 5.2.6 依赖关系
- 依赖 `QueryService`
- 依赖 `RankService`
- 读取目的地与场所基础数据

### 5.2.7 当前优先级
P0

### 5.2.8 与课程要求的关系
课程要求中明确包含旅游推荐，并要求能够按热度、评价等进行排序和查询。课堂讨论还强调推荐不能单一化，应采用检索后二次排序。

---

## 5.3 Route 模块

### 5.3.1 模块目标
为用户提供景区或校园内部的路径规划能力。

### 5.3.2 主要职责
- 单目标最短路径规划
- 多目标路径规划（后续增强）
- 路径节点、距离、时间结果输出
- 为前端路径展示提供结果数据

### 5.3.3 不负责的内容
- 不负责目的地推荐
- 不负责设施和美食数据本体管理
- 不负责日记内容管理

### 5.3.4 主要输入
- 当前节点
- 目标节点 / 目标节点列表
- 路径策略
- 当前目的地图数据

### 5.3.5 主要输出
- 路径节点列表
- 总距离
- 预计时间
- 路径展示相关结果

### 5.3.6 依赖关系
- 依赖 `MapService`
- 依赖 `MapNodeMapper`、`MapEdgeMapper`
- 依赖图建模与路径算法能力

### 5.3.7 当前优先级
P0

### 5.3.8 与课程要求的关系
课程要求中明确包含最短路径和多点路径规划，是数据结构与算法考查重点之一。

---

## 5.4 Facility 模块

### 5.4.1 模块目标
为用户提供当前位置附近设施的查询能力。

### 5.4.2 主要职责
- 查询附近设施
- 按设施类型过滤
- 基于图上可达距离 / 时间返回结果
- 输出设施列表

### 5.4.3 不负责的内容
- 不负责完整路线规划主流程
- 不负责设施基础数据维护（由管理端负责）

### 5.4.4 主要输入
- 当前节点
- 设施类型
- 过滤条件

### 5.4.5 主要输出
- 设施列表
- 可达距离 / 可达时间
- 设施基本信息

### 5.4.6 依赖关系
- 依赖 `QueryService`
- 依赖 `RankService`
- 依赖 `MapService`

### 5.4.7 当前优先级
P0

### 5.4.8 与课程要求的关系
课程要求中明确包含场所查询；课堂讨论中进一步强调设施结果应基于图上可达距离或时间，而不是直线距离。

---

## 5.5 Food 模块

### 5.5.1 模块目标
为用户提供当前位置或目的地相关的美食信息和基础推荐能力。

### 5.5.2 主要职责
- 输出美食列表
- 支持按热度、评分、距离排序
- 支持基础菜系或关键字过滤
- 支持与目的地或当前位置联动

### 5.5.3 不负责的内容
- 不负责复杂餐饮订单或交易
- 不负责复杂图片识别能力的最终实现

### 5.5.4 主要输入
- 当前目的地 / 当前位置
- 排序条件
- 菜系 / 关键字

### 5.5.5 主要输出
- 美食列表
- 基础评价与分类信息

### 5.5.6 依赖关系
- 依赖 `QueryService`
- 依赖 `RankService`

### 5.5.7 当前优先级
P1

### 5.5.8 与课堂讨论的关系
课堂讨论中明确提到美食推荐、模糊查询、按口味和特色推荐等方向。

---

## 5.6 Diary 模块

### 5.6.1 模块目标
为用户提供旅游后日记的发布、浏览、详情和关联查看能力。

### 5.6.2 主要职责
- 发布图文日记
- 浏览日记列表
- 查看日记详情
- 按目的地查看相关日记
- 后续支持检索、排序和 AI 增强

### 5.6.3 不负责的内容
- 不负责复杂社交互动系统
- 不负责复杂富文本平台化能力

### 5.6.4 主要输入
- 标题
- 正文
- 图片
- 关联目的地
- 检索关键字（后续增强）

### 5.6.5 主要输出
- 日记列表
- 日记详情
- 发布结果
- 日记与目的地关联结果

### 5.6.6 依赖关系
- 依赖 `SearchService`
- 依赖 `FileService`
- 可依赖 `AIService` 做后续增强
- 依赖日记主表、评分表、媒体表

### 5.6.7 当前优先级
P0

### 5.6.8 与课程要求的关系
课程要求中明确提出旅游后可根据照片和游览经历生成旅游日记。课堂讨论中还提到日记与推荐打通、支持全文检索和压缩存储。

---

## 5.7 Admin 模块

### 5.7.1 模块目标
提供最小可用的后台支撑能力，维护系统运行所需基础数据。

### 5.7.2 主要职责
- 维护目的地、场所、设施、美食、日记等基础数据
- 提供基础导入入口
- 支持管理端查看和修正数据

### 5.7.3 不负责的内容
- 不做复杂运营后台
- 不做复杂权限平台
- 不做完整审计平台

### 5.7.4 主要输入
- 管理端表单数据
- 导入文件
- 查询与筛选条件

### 5.7.5 主要输出
- 数据维护结果
- 导入结果
- 管理端查询结果

### 5.7.6 依赖关系
- 依赖 `QueryService`
- 依赖 `ImportService`
- 依赖多个 Mapper 和统一返回机制

### 5.7.7 当前优先级
P1

## 6. 公共能力模块说明

## 6.1 QueryService
### 职责
为推荐、设施、美食、管理端等多个模块提供统一查询能力。

### 主要作用
- 统一条件查询
- 降低重复 SQL 和重复查询逻辑
- 为排序、筛选、联动展示提供统一基础

### 当前优先级
P0

---

## 6.2 RankService
### 职责
提供排序与 Top-K 能力。

### 主要作用
- 目的地排序
- 设施结果排序
- 美食结果排序
- 为课堂讨论中提到的“用户通常只看前若干项”提供实现基础

### 当前优先级
P0

---

## 6.3 MapService
### 职责
提供图建模、最短路径、多点路径等能力。

### 主要作用
- 路线规划
- 周边设施可达距离计算
- 后续交通策略、最短时间策略扩展

### 当前优先级
P0

---

## 6.4 SearchService
### 职责
提供全文检索与模糊检索能力。

### 主要作用
- 日记正文检索
- 可能扩展到更丰富的文本搜索场景

### 当前优先级
P1

---

## 6.5 FileService
### 职责
提供文件上传与文件访问能力。

### 主要作用
- 日记图片上传
- 附件与资源访问
- 后续多媒体内容支撑

### 当前优先级
P0

---

## 6.6 AIService
### 职责
作为创新功能和 AI 辅助能力的统一接入点。

### 主要作用
- 日记草稿生成
- 图片摘要
- 标签提取
- 多人规划协商说明

### 当前优先级
P2

### 当前处理原则
当前阶段保留模块位置，但不让 AI 创新功能阻塞基础主线闭环。

---

## 6.7 ImportService
### 职责
负责批量导入和数据初始化支撑。

### 主要作用
- 接收 Python 清洗后的数据
- 写入数据库和文件存储
- 支撑样例数据和后续真实数据导入

### 当前优先级
P1

---

## 7. 基础支撑模块说明

## 7.1 Security / JWT
提供认证校验、token 处理和最小权限支撑能力。

## 7.2 Config / Exception / Response
提供基础配置、统一异常处理、统一返回结构等系统级支撑。

## 7.3 GraphEngine
作为图结构和路径算法的底层实现基础，服务于 `MapService`。

## 7.4 IndexEngine
作为查找、检索相关的数据结构支撑，服务于查询和检索能力。

## 7.5 CompressionEngine
作为日记等内容的压缩能力预留支撑。

## 7.6 ImportEngine
作为导入格式处理和转换的底层支撑。

## 8. 模块边界与协作关系

## 8.1 主线协作关系
系统当前业务主线可概括为：

- Auth  
  → 用户登录进入系统

- Recommend  
  → 用户选择或搜索目的地

- Route / Facility / Food  
  → 用户在目的地内规划路线、查询周边、查看美食

- Diary  
  → 用户旅游后发布和浏览日记

## 8.2 关键协作说明
- Recommend 依赖 QueryService 和 RankService
- Route 强依赖 MapService
- Facility 同时依赖 QueryService、RankService 和 MapService
- Diary 依赖 FileService，后续依赖 SearchService 和 AIService
- Admin 为多个业务模块提供数据维护支撑
