# 推荐模块设计说明（Recommend Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“推荐模块”的设计目标、职责边界、输入输出、算法思路、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：
- 推荐模块负责什么，不负责什么
- 推荐模块如何支撑“旅游前”场景
- 推荐模块依赖哪些数据和公共能力
- 推荐模块采用什么推荐与排序思路
- 推荐模块如何与查询、日记、路线规划等模块协作

---

## 2. 模块定位

### 2.1 模块目标
推荐模块用于支持用户在旅游前选择目的地，帮助用户根据热度、评价和个人兴趣，从景区和校园中筛选并获得前若干个推荐结果。

### 2.2 对应课程要求
课程要求中，旅游推荐功能明确包括：
1. 用户可以根据自己的喜好选择不同的景点和学校作为旅游目的地；
2. 系统可按照旅游热度、评价和个人兴趣进行推荐；
3. 推荐算法基础要求为排序算法；
4. 用户通常只看前 10 个结果，因此要求不经过完全排序也能排好前 10；
5. 用户可以按名称、类别、关键字等进行查询，查询结果有多项时可继续按热度和评价排序。

因此，推荐模块既包含“推荐”，也包含“搜索 + 结果排序”两部分能力。

### 2.3 模块在系统中的位置
推荐模块位于用户端主链路的最前端，是系统“旅游前”阶段的主要入口之一。  
用户通常会先通过推荐模块确定目的地，再进入：
- 目的地详情
- 路线规划模块
- 周边设施模块
- 美食模块
- 与该目的地相关的日记浏览

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
推荐模块负责：
1. 目的地推荐列表生成
2. 目的地搜索与筛选
3. 推荐结果排序
4. Top-K 结果输出
5. 推荐页和搜索页的数据返回
6. 为前端提供推荐理由或推荐标签的基础信息
7. 为目的地详情页提供入口数据

## 3.2 本模块不负责的内容
推荐模块不负责：
1. 用户登录与权限校验（由 Auth 模块负责）
2. 路线规划（由 Route 模块负责）
3. 周边设施查询（由 Facility 模块负责）
4. 美食数据维护（由 Food / Admin 模块负责）
5. 日记发布与内容管理（由 Diary 模块负责）
6. 复杂画像训练或机器学习模型训练
7. 地图数据建模

## 3.3 模块边界原则
- 推荐模块只负责“找目的地、排结果、返回前若干项”；
- 不在推荐模块内直接实现路线、设施、日记等下游逻辑；
- 共用查询和排序逻辑优先复用公共能力层，不在模块内部重复实现。

## 4. 用户场景与功能范围

## 4.1 核心用户场景
### 场景一：用户不知道去哪
用户进入系统后，希望看到系统推荐的一批目的地，快速找到“值得去”的景区或学校。

### 场景二：用户有模糊目标
用户可能知道自己想去“校园”“文化类景点”或者“适合散步的地方”，因此希望通过关键字、类型、主题等缩小范围。

### 场景三：用户已有候选结果
用户搜索得到多个结果后，希望按热度、评价重新排序，只看前若干个最值得看的结果。

---

## 4.2 当前阶段功能范围

### P0（当前必须优先实现）
1. 目的地推荐
2. 关键字搜索
3. 类型 / 类别筛选
4. 按热度排序
5. 按评分排序
6. Top-K 输出前若干结果

### P1（基础闭环后补充）
1. 用户偏好更完整参与推荐
2. 推荐理由生成
3. 更复杂的综合评分模型
4. 与日记热度、浏览量联动
5. 个性化排序增强

### P2（后续创新）
1. AI 解释式推荐
2. 多用户共同偏好推荐
3. 与多人旅游规划协商联动

## 5. 输入与输出设计

## 5.1 输入
推荐模块的输入可分为四类：

### 5.1.1 用户上下文输入
- 当前用户 ID（可为空）
- 用户偏好信息（若已登录）

### 5.1.2 检索输入
- 关键字
- 名称
- 类型（景区 / 校园）
- 类别
- 标签 / 主题

### 5.1.3 排序输入
- 排序字段：热度 / 评分 / 推荐
- 排序方向：升序 / 降序（一般默认降序）

### 5.1.4 分页与 Top-K 输入
- pageNum
- pageSize
- topK

---

## 5.2 输出
推荐模块输出统一为目的地结果列表，单个目的地结果至少包含：
- id
- name
- type
- category
- city
- description（摘要）
- heatScore
- ratingScore
- coverUrl
- tags

在增强版本中，可继续输出：
- recommendReason
- matchScore
- userPreferenceMatchTag

## 7. 模块依赖关系

## 7.1 依赖的系统模块
推荐模块直接依赖：

### 7.1.1 QueryService
负责统一查询逻辑，包括：
- 关键字过滤
- 类型筛选
- 类别筛选
- 条件组合查询

### 7.1.2 RankService
负责排序与 Top-K 输出，包括：
- 按热度排序
- 按评分排序
- 推荐综合分排序
- 不完全排序下的前 K 个结果输出

### 7.1.3 Auth 模块（弱依赖）
用于获取当前用户身份和偏好上下文。未登录时，推荐模块也应能正常工作。

- 搜索页
- 目的地详情页
- 路线规划模块的目的地入口
- 日记模块中“按目的地查看相关日记”的入口

## 8. 核心算法设计

## 8.1 设计目标
课程对推荐模块的核心要求不是“复杂模型”，而是：
1. 体现排序算法；
2. 体现查找能力；
3. 体现 Top-K 思想；
4. 考虑数据动态变化；
5. 尽量避免全量完全排序。

因此，本模块优先采用“**检索 + 排序 + Top-K**”的组合策略。

---

## 8.2 整体算法思路

### 第一步：候选召回（Recall）
根据用户输入条件，从 `destination` 中召回候选结果。
可用条件：
- 名称匹配
- 类别匹配
- 关键字匹配
- 类型匹配
- 标签匹配（后续增强）

### 第二步：规则打分（Scoring）
对召回结果计算综合得分。当前基础版可采用简单加权模型，例如：

`recommendScore = a * heatScore + b * ratingScore + c * preferenceMatchScore`

其中：
- `heatScore`：目的地热度
- `ratingScore`：目的地评分
- `preferenceMatchScore`：用户偏好匹配程度（可选）

### 第三步：Top-K 排序输出
不对所有结果做完全排序，而优先获取前 K 个结果。  
可采用：
- 小根堆维护 Top-K
- 快速选择思想
- 局部排序

当前阶段建议先实现：
- 小根堆 Top-K
- 作为最稳妥的基础版本

---

## 8.3 当前推荐算法版本划分

### V1：基础排序版（P0）
- 候选集过滤
- 按热度或评分单字段排序
- 支持 Top-K 返回

### V2：简单综合评分版（P1）
- 将热度、评分、偏好匹配做简单加权
- 支持推荐排序

### V3：增强推荐版（P2）
- 引入浏览、评分、日记热度等多源特征
- 加入可解释推荐理由

---

## 8.4 可用数据结构建议
为满足课程要求，推荐模块建议至少体现以下数据结构：

1. **数组 / 列表**
   - 保存候选结果集

2. **哈希结构**
   - 用于关键字、类别、标签等快速过滤辅助

3. **堆（优先队列）**
   - 用于 Top-K 推荐

4. **可选：Trie / 倒排结构**
   - 用于后续更高效的名称/关键字检索增强

---

## 8.5 时间复杂度说明
以候选结果数为 `n`，返回前 `k` 个结果为例：

- 过滤召回：约 `O(n)`
- 使用小根堆维护 Top-K：约 `O(n log k)`

相比全量完全排序 `O(n log n)`，当 `k << n` 时更适合当前推荐场景。

## 9. 业务流程设计

## 9.1 推荐流程
1. 用户进入推荐页
2. 前端传入筛选条件和排序方式
3. RecommendController 接收请求
4. RecommendService 调用 QueryService 查询候选目的地
5. RecommendService 调用 RankService 计算排序 / Top-K
6. 返回推荐结果列表
7. 前端展示卡片结果

---

## 9.2 搜索流程
1. 用户输入关键字
2. RecommendController 接收搜索请求
3. QueryService 根据关键字和类型做召回
4. RankService 对结果按热度或评分排序
5. 返回搜索结果列表

---

## 12. 异常与边界场景

推荐模块至少要考虑以下情况：

1. 无结果
   - 返回空列表，不报系统异常

2. 关键字为空
   - 允许退化为默认推荐

3. 排序字段非法
   - 返回参数错误

4. 数据字段缺失
   - 对热度、评分为空的记录做兜底处理

5. 用户未登录
   - 推荐功能仍可用，只是忽略偏好项

6. 分页参数异常
   - 使用默认分页值或返回参数错误

## 13. 测试要点

## 13.1 功能测试
1. 按热度推荐是否正确
2. 按评分推荐是否正确
3. 关键字搜索是否正确
4. 类型筛选是否正确
5. 排序切换是否正确
6. Top-K 数量是否正确
7. 无结果时是否正常返回

## 13.2 算法测试
1. Top-K 输出是否正确
2. 数据动态变化后结果是否仍正确
3. 小数据量与大数据量下排序结果是否稳定

## 13.3 边界测试
1. 空关键字
2. 非法排序字段
3. 极大 pageSize
4. 热度 / 评分为空
5. 同名目的地
6. 全部记录评分相同或热度相同

## 14. 实现约束

推荐模块在实现时，必须遵守以下约束：

1. Controller 不写复杂业务逻辑
2. 查询逻辑优先抽到 QueryService
3. 排序和 Top-K 逻辑优先抽到 RankService
4. 不允许字符串拼接 SQL
5. 分页、排序、筛选条件必须显式可读
6. 所有接口统一返回 ApiResponse
7. 公共方法和核心算法必须写简洁注释
8. 复杂算法需说明数据结构和复杂度
9. 不能在日志中输出敏感信息
10. 推荐模块修改后，应同步更新：
   - `api-spec.md`
   - `module-map.md`
   - `dependency-map.md`
   - 相关测试文档
