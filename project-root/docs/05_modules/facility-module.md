# 周边设施查询模块设计说明（Facility Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“周边设施查询模块”的设计目标、职责边界、输入输出、核心算法、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：
- 周边设施查询模块负责什么，不负责什么
- 周边设施查询模块如何支撑“旅游中”场景
- 周边设施查询模块依赖哪些数据和公共能力
- 为什么设施排序必须按图上可达距离 / 时间，而不是直线距离
- 周边设施查询模块如何与路线规划和前端周边页协作

---

## 2. 模块定位

### 2.1 模块目标
周边设施查询模块用于支持用户在景区或校园内部游览时，根据当前位置快速查找厕所、食堂、商店、图书馆、超市、咖啡馆等设施，并按可达距离或可达时间返回结果。

### 2.2 对应课程要求
课程题面在“旅游中”场景中明确提出，用户在景点（包括学校）内部游览时，除了规划最优线路，还需要进行场所查询。课程数据要求里也明确提出服务设施不少于 10 种、总数量不少于 50 个，包括商店、饭店、洗手间、图书馆、食堂、超市、咖啡馆等。:contentReference[oaicite:6]{index=6}

### 2.3 当前项目中的设计强化
在你们当前方案中，周边设施查询不是简单返回“附近有什么”，而是强调：
- 按设施类别过滤
- 按图上可达距离排序
- 必要时按可达时间排序
- 明确不是按直线距离返回  
这是当前系统区别于很多简单地图查询实现的关键点之一。:contentReference[oaicite:7]{index=7}

### 2.4 模块在系统中的位置
周边设施查询模块位于用户端“旅游中”主链路中，通常在以下场景触发：
- 用户规划完路线后，继续查询附近设施
- 用户游览中途需要找厕所、食堂、商店等
- 用户从某景点 / 建筑出发，查看周边服务点
- 用户查看美食模块前，先筛选附近餐饮类设施

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
周边设施查询模块负责：
1. 根据当前位置查询可用设施
2. 按设施类别过滤
3. 按图上可达距离排序
4. 按可达时间排序（后续增强）
5. 返回设施的基础信息和可达信息
6. 为前端周边页和路线页提供设施查询结果
7. 为美食模块提供设施层入口（如食堂、店铺）

## 3.2 本模块不负责的内容
周边设施查询模块不负责：
1. 用户登录与权限校验（由 Auth 模块负责）
2. 目的地推荐与搜索（由 Recommend 模块负责）
3. 完整路径主流程规划（由 Route 模块负责）
4. 美食具体菜品推荐（由 Food 模块负责）
5. 设施基础数据维护（由 Admin 模块负责）
6. 地图原始数据采集和清洗（由数据导入流程负责）

## 3.3 模块边界原则
- 本模块只负责“从当前位置查设施并按可达性排序”；
- 设施数据查询与可达性计算分层实现；
- 设施结果的“查”和“排”应分别复用公共能力层；
- 不允许在模块内部退化为“只按经纬度直线距离排序”。

## 4. 用户场景与功能范围

## 4.1 核心用户场景
### 场景一：找厕所 / 食堂 / 商店
用户在校园或景区内，希望从当前位置快速找到最近的厕所、食堂、商店等高频设施。

### 场景二：按类别筛选
用户只想看某类设施，例如“只看咖啡馆”或“只看图书馆”。

### 场景三：与路线规划联动
用户已经规划完路线，但临时想顺路找一个设施点，系统需要给出图上更合理的可达结果。

### 场景四：作为美食入口
用户先查到“食堂 / 餐饮设施”，再进入美食推荐模块查看具体菜品或窗口。

---

## 4.2 当前阶段功能范围

### P0（当前必须优先实现）
1. 根据当前位置查询设施
2. 按设施类型过滤
3. 按图上可达距离排序
4. 返回设施基础信息
5. 返回可达距离
6. 与路线规划模块共享图数据

### P1（基础闭环后补充）
1. 按可达时间排序
2. 返回预计到达时间
3. 返回简单路径摘要
4. 支持更多设施类别组合过滤

### P2（后续增强）
1. 设施热度 / 评分排序
2. 排队情况 / 拥挤度展示
3. 室内设施查询
4. 与多人行程协商联动

## 6. 数据依赖

## 6.1 主要数据表
周边设施查询模块主要依赖以下数据表：

### 6.1.1 `facility`
保存厕所、食堂、商店、图书馆、咖啡馆等设施信息，是本模块的核心对象表。当前表设计中，`facility` 明确用于支撑周边设施查询与可达排序，并建议建立 `destination_id`、`facility_type`、`name` 索引。:contentReference[oaicite:8]{index=8}

### 6.1.2 `place`
用于辅助表达设施所在场所，便于前端展示“设施属于哪个建筑 / 场所”。

### 6.1.3 `map_node`
用于把设施和当前位置映射到图节点。当前设计中，`map_node` 可以表示交叉口、建筑物、设施点，并通过 `node_type + ref_id` 映射到场所或设施。:contentReference[oaicite:9]{index=9}

### 6.1.4 `map_edge`
用于计算从当前位置到各设施的可达距离和时间。当前设计中，`map_edge` 支持距离、理想速度、拥挤度、交通方式和边类型，适合可达距离 / 时间计算。:contentReference[oaicite:10]{index=10}

### 6.1.5 `destination`
用于限定当前设施查询发生在哪个目的地下，避免跨目的地误查。

## 7. 模块依赖关系

## 7.1 依赖的系统模块
周边设施查询模块直接依赖：

### 7.1.1 `QueryService`
负责：
- 按目的地筛选设施
- 按设施类型过滤
- 返回候选设施集合

### 7.1.2 `RankService`
负责：
- 根据可达距离对候选设施排序
- 后续支持可达时间排序
- 限制返回前若干项

### 7.1.3 `MapService`
负责：
- 设施点与当前位置的图上距离计算
- 后续可达时间计算
- 提供必要的最短路径摘要

### 7.1.4 `Route` 模块（弱依赖）
Route 模块不直接作为查询依赖，但两者共享图结构与 `MapService`。  
你们当前方案里已经明确：周边设施查询与路线规划都应建立在同一张有向带权图之上。:contentReference[oaicite:11]{index=11}

当前系统架构图中，周边设施 / 美食模块位于同一业务区，并依赖 `QueryService`、`RankService`；同时由于可达性计算，设施模块还应复用 `MapService`。:contentReference[oaicite:12]{index=12}

## 7.2 被哪些模块依赖
周边设施查询模块的结果会被以下模块复用：
- 周边设施页
- 地图导航页
- 行程规划页中的“顺路查询”
- 美食模块（餐饮类设施入口）
- 后续 Diary 模块中的“路线回顾点位说明”

## 8. 核心算法设计

## 8.1 设计目标
本模块的核心不是“把设施查出来”，而是“**查出来后，按用户真正走得到的代价排序**”。  
因此它的算法重点在：
1. 设施候选集过滤
2. 当前点到设施点的图上最短路径代价计算
3. 代价排序与前若干项输出

---

## 8.2 为什么不能按直线距离
如果仅按经纬度直线距离排序，会出现：
- 楼之间被围栏或建筑阻挡
- 实际道路绕行较远
- 楼层或通道限制
- 单向边、楼梯、电梯等通行约束被忽略

因此，当前系统要求周边设施按**图上可达距离 + 时间**排序，而不是直线距离。这个点在你们当前方案中已经明确写出，也是答辩时很重要的亮点。:contentReference[oaicite:13]{index=13}

---

## 8.3 整体算法思路

### 第一步：候选设施召回
从 `facility` 表中按：
- `destinationId`
- `facilityType`
进行过滤，得到候选设施集合。

### 第二步：设施映射到节点
把每个候选设施映射到对应的 `map_node`：
- 直接通过 `node_type = facility` + `ref_id`
- 或通过场所 / 位置映射

### 第三步：计算可达代价
从 `sourceNodeId` 出发，使用图算法计算到各设施节点的最短代价：
- P0：最短距离
- P1：最短时间

### 第四步：排序与 Top-K
根据可达代价排序，并返回前若干个最优结果。

---

## 8.4 当前推荐实现

### P0：图上可达距离版
- 候选设施过滤
- 设施点映射到图节点
- Dijkstra 计算当前点到候选设施点的距离
- 按 `reachableDistance` 升序排序

### P1：可达时间版
- 在 P0 基础上切换边权
- 使用：
  - `realSpeed = crowdFactor * idealSpeed`
  - `timeCost = distance / realSpeed`
- 按 `estimatedTime` 升序排序

### P2：约束扩展版
- 不同交通工具约束
- 室内设施可达
- 拥挤度感知推荐

---

## 8.5 可用数据结构建议
为体现课程的数据结构要求，本模块建议至少体现以下数据结构：

1. **数组 / 列表**
   - 保存候选设施集

2. **哈希映射**
   - 设施 ID → 节点 ID 映射
   - 设施类型 → 候选集过滤

3. **优先队列**
   - 用于最短路径搜索
   - 用于结果 Top-K 输出（可选）

4. **有向带权图**
   - 复用 `MapService / GraphEngine` 的图结构

---

## 8.6 时间复杂度说明
设：
- 候选设施数为 `m`
- 图节点数为 `V`
- 图边数为 `E`

若先做一次单源最短路径，再查询候选设施点结果，则：
- 图上最短路径计算约为 `O((V + E) log V)`
- 候选设施排序约为 `O(m log m)`

若只取前 `k` 个结果，后续可进一步优化为：
- `O((V + E) log V + m log k)`

## 9. 业务流程设计

## 9.1 周边设施查询流程
1. 用户进入周边设施页
2. 当前页面拿到 `destinationId` 和 `sourceNodeId`
3. 用户选择设施类型
4. 前端调用周边设施查询接口
5. `FacilityController` 接收请求
6. `FacilityService` 组织查询流程
7. `QueryService` 过滤候选设施
8. `MapService` 计算可达距离
9. `RankService` 按可达距离排序
10. 返回设施列表给前端
11. 前端展示结果并支持“设为目标点”

---

## 9.2 与路线规划联动流程
1. 用户在导航页完成路径规划
2. 选择“查看附近设施”
3. 当前终点或当前位置作为 `sourceNodeId`
4. 调用设施查询接口
5. 结果页继续显示“从当前位置可达的设施”

## 12. 异常与边界场景

至少需要考虑以下情况：

1. 当前节点不存在
2. 当前节点不属于指定目的地
3. 设施类型非法
4. 当前目的地下无该类设施
5. 设施存在但节点映射缺失
6. 图数据缺失，无法计算可达距离
7. 图上不可达
8. 候选设施数为 0
9. 排序字段非法

这些情况应返回清晰的业务提示，而不是直接向前端暴露底层异常。当前项目安全规范和代码规范都要求统一异常处理与安全日志。:contentReference[oaicite:14]{index=14} :contentReference[oaicite:15]{index=15}

## 13. 测试要点

## 13.1 功能测试
1. 指定设施类型后是否能正确返回设施
2. 返回结果是否属于同一目的地
3. 按可达距离排序是否正确
4. 结果数量限制是否生效
5. 切换不同设施类型是否正确

## 13.2 算法测试
1. 图上可达距离是否正确
2. 直线距离更近但图上更远时，结果是否仍正确
3. 某设施节点不可达时是否正确跳过或提示
4. 多个设施距离相近时排序是否稳定

## 13.3 边界测试
1. 当前点本身就是设施点
2. 当前点附近没有同类设施
3. 设施缺失 `place_id`
4. 节点与设施映射不完整
5. 极大 `limit` 参数
6. 空类型 / 非法类型输入

按照当前代码规范，核心模块至少应覆盖正常输入、边界输入和异常输入样例。:contentReference[oaicite:16]{index=16}

## 14. 实现约束

周边设施查询模块在实现时，必须遵守以下约束：

1. `FacilityController` 不写复杂业务逻辑
2. 候选设施过滤优先交给 `QueryService`
3. 距离 / 时间计算统一放在 `MapService`
4. 排序逻辑优先交给 `RankService`
5. 不允许直接按经纬度直线距离替代图上可达距离
6. 不允许字符串拼接 SQL
7. 所有接口统一返回 `ApiResponse`
8. 涉及复杂查询和排序的地方必须写注释
9. 结果字段必须与前端保持一致
10. 修改设施表、接口或依赖关系后，必须同步更新：
   - `schema.md`
   - `api-spec.md`
   - `module-map.md`
   - `dependency-map.md`

这些实现约束与当前项目的分层和公共能力抽取规则一致。:contentReference[oaicite:17]{index=17}
