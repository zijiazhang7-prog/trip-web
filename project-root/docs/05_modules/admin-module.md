# 管理端模块设计说明（Admin Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“管理端模块”的设计目标、职责边界、输入输出、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：
- 管理端模块负责什么，不负责什么
- 管理端模块如何支撑整个系统运行
- 管理端模块依赖哪些数据和公共能力
- 管理端模块应提供哪些最小维护能力
- 管理端模块如何与导入流程、用户权限和前台业务模块协作

---

## 2. 模块定位

## 2.1 模块目标
管理端模块用于提供系统运行所需的最小后台支撑能力，负责基础数据维护、批量导入入口、内容管理和部分用户管理能力，保证推荐、导航、周边设施、美食和日记等前台模块拥有可用数据。

## 2.2 模块在系统中的位置
管理端模块属于系统的后台支撑模块。  
它不是用户主链路的一部分，但它决定了前台功能能否正常运行，尤其影响：
- 目的地推荐是否有数据
- 地图导航是否有节点和边
- 周边设施查询是否可用
- 美食模块是否有数据
- 日记内容和媒体是否可维护

## 2.3 当前模块角色
在当前项目里，管理端模块属于：
- **P1 基础支撑模块**
- 不一定要在最开始做完整
- 但至少要先有最小可用的数据维护和导入能力
- 适合作为演示环境初始化和后续扩展的重要支撑

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
管理端模块负责：
1. 目的地数据管理
2. 场所 / 建筑物数据管理
3. 服务设施数据管理
4. 美食数据管理
5. 地图节点与边数据管理（最小可用）
6. 日记内容基础管理
7. 用户基础管理
8. 批量导入入口
9. 导入结果和错误信息查看（后续增强）
10. 部分状态管理（启用 / 禁用 / 上下架等）

## 3.2 本模块不负责的内容
管理端模块不负责：
1. 前台推荐逻辑本体
2. 前台路线规划算法本体
3. 前台日记发布流程本体
4. 外部原始数据抓取和清洗（由离线数据处理脚本负责）
5. 复杂运营报表
6. 复杂审核流
7. 复杂权限平台
8. 商业化后台能力

## 3.3 模块边界原则
- 管理端负责“维护”和“导入”，不负责底层算法实现；
- 原始数据抓取、清洗和标准化优先在 `scripts/` 和离线流程中完成；
- 管理端可作为“最后入库入口”和“少量人工修正入口”；
- 首版不做复杂后台平台化能力，先保证最小可用。

## 4. 用户场景与功能范围

## 4.1 核心用户场景
### 场景一：初始化系统数据
管理员需要把景区、校园、建筑、设施、地图、美食等数据导入系统，供前台模块使用。

### 场景二：修正数据错误
管理员发现某个目的地名称、设施类别、地图节点、图片地址有问题，需要在后台修改。

### 场景三：维护内容状态
管理员需要禁用某条错误数据、下架某篇不合适的日记、停用某个用户账号。

### 场景四：查看导入结果
管理员执行批量导入后，希望知道哪些成功、哪些失败、失败原因是什么。

---

## 4.2 当前阶段功能范围

### P0（最小可用后台支撑）
1. 管理员登录后访问管理端
2. 目的地数据增删改查
3. 设施数据增删改查
4. 美食数据增删改查
5. 基础导入入口
6. 用户基础列表查看
7. 日记基础列表查看

### P1（基础版后补充）
1. 场所 / 建筑物数据管理
2. 地图节点与边管理
3. 日记状态管理
4. 用户状态管理
5. 导入批次查看
6. 错误信息查看

### P2（后续增强）
1. 更细粒度权限
2. 批量审核能力
3. 导入预校验页面
4. 数据质量看板
5. 操作日志

说明：
当前已在接口文档中预留场所管理、地图节点管理和地图边管理相关接口，但实现优先级仍按 P1 执行，不阻塞当前最小后台支撑能力。

## 7. 模块依赖关系

## 7.1 依赖的系统模块
管理端模块直接依赖：

### 7.1.1 `Auth` 模块
负责：
- 管理员登录
- 管理端访问权限校验
- 当前用户角色识别

### 7.1.2 `QueryService`
负责：
- 各类数据列表查询
- 条件过滤
- 分页结果组织

### 7.1.3 `ImportService`
负责：
- 接收标准化导入文件
- 执行批量导入
- 返回导入结果
- 记录导入批次（后续）

### 7.1.4 `FileService`
负责：
- 封面图、媒体资源、导入文件访问
- 文件路径维护

### 7.1.5 统一异常处理与统一返回体
负责：
- 后台接口统一输出
- 导入失败、权限不足、数据不存在等异常的统一处理

---

## 7.2 被哪些模块依赖
管理端模块虽然不直接被前台页面调用，但它维护的数据会被以下模块使用：
- Recommend 模块
- Route 模块
- Facility 模块
- Food 模块
- Diary 模块
- 前台首页 / 详情页 / 地图页 / 周边页 / 美食页

因此，管理端的本质是：
> 为所有前台模块提供可靠、可修正、可扩展的数据底座。

## 8. 核心设计思路

## 8.1 当前设计目标
管理端模块当前目标不是做复杂后台系统，而是做一个：
- 能维护核心数据
- 能做最小导入
- 能支持演示环境初始化
- 能辅助后续调试和修正数据

的轻量后台。

---

## 8.2 最小可用后台原则
首版后台只做三件最重要的事：
1. 让基础数据可以被维护
2. 让批量导入有入口
3. 让管理员能看见和修正明显问题

不建议一开始就做：
- 多角色复杂后台
- 多级审核流
- 大量统计图表
- 复杂日志体系

---

## 8.3 数据导入设计思路
管理端中的导入能力应基于你们已经定好的数据导入计划来实现：

### 当前建议方式
- 外部数据先经过 Python + pandas 清洗
- 生成标准化 CSV / JSON
- 管理端或导入脚本调用 `ImportService`
- `ImportService` 再写入 MySQL 和文件存储

这样管理端不承担原始数据清洗，而只承担“导入入口 + 结果回显”的角色。

---

## 8.4 状态管理设计思路
当前后台管理的状态控制建议保持简单：

### 用户状态
- 启用
- 禁用

### 目的地状态
- 启用
- 禁用 / 下架

### 日记状态
- 可见
- 隐藏 / 下架

不建议首版引入复杂的多状态工作流。

---

## 9. 业务流程设计

## 9.1 管理员登录流程
1. 管理员进入后台登录页
2. 输入用户名和密码
3. 前端调用 Auth 登录接口
4. 后端返回 token 和用户角色
5. 前端判断角色为 `admin`
6. 进入后台页面

---

## 9.2 目的地维护流程
1. 管理员进入目的地管理页
2. 查看目的地列表
3. 新增 / 编辑 / 删除某个目的地
4. 后端写入 `destination` 表
5. 返回操作结果
6. 前端刷新列表

---

## 9.3 场所维护流程
1. 管理员进入场所管理页
2. 选择当前目的地
3. 查看该目的地下的场所列表
4. 新增 / 编辑 / 删除某个场所
5. 后端写入 `place` 表
6. 返回操作结果
7. 前端刷新列表

---

## 9.4 地图节点 / 边维护流程
1. 管理员进入地图管理页
2. 选择当前目的地
3. 切换“节点管理”或“边管理”
4. 新增 / 编辑 / 删除节点或边
5. 后端写入 `map_node` 或 `map_edge` 表
6. 返回操作结果
7. 前端刷新列表

---

## 9.5 设施 / 美食维护流程
1. 管理员进入设施或美食管理页
2. 按目的地筛选数据
3. 修改类别、名称、描述、状态等信息
4. 后端写入相应数据表
5. 返回更新结果

---

## 9.6 批量导入流程
1. 管理员进入导入页面
2. 上传标准化导入文件
3. 选择导入类型和批次名称
4. 前端调用导入接口
5. `AdminController` 调用 `ImportService`
6. `ImportService` 执行导入
7. 返回成功 / 失败结果
8. 后续可记录到 `import_batch`

---

## 9.7 日记与用户管理流程
1. 管理员进入用户或日记管理页
2. 查看列表
3. 选择禁用 / 启用 / 隐藏 / 删除（按当前策略）
4. 后端更新状态字段
5. 返回操作结果

---

## 12. 异常与边界场景

至少需要考虑以下情况：

1. 普通用户访问后台
2. token 缺失或失效
3. 要编辑的数据不存在
4. 删除对象存在下游依赖
5. 导入文件格式不合法
6. 导入文件为空
7. 同一批次重复导入
8. 状态字段非法
9. 文件资源路径失效

这些情况应返回清晰的业务错误信息，不直接暴露底层异常。

## 14. 实现约束

管理端模块在实现时，必须遵守以下约束：

1. `AdminController` 不写复杂业务逻辑
2. 基础查询优先走 `QueryService`
3. 导入优先走 `ImportService`
4. 权限校验依赖 Auth 与安全层
5. 不允许字符串拼接 SQL
6. 所有接口统一返回 `ApiResponse`
7. 导入失败必须能返回可理解的错误信息
8. 修改后台维护的表、接口或状态字段后，必须同步更新：
   - `schema.md`
   - `api-spec.md`
   - `module-map.md`
   - `dependency-map.md`
   - `import-plan.md`
