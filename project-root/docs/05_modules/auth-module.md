# 认证模块设计说明（Auth Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“认证模块”的设计目标、职责边界、输入输出、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：
- Auth 模块负责什么，不负责什么
- Auth 模块如何为其他模块提供基础用户能力
- Auth 模块依赖哪些数据和公共能力
- Auth 模块应提供哪些最小接口
- Auth 模块如何支撑权限、登录状态和用户上下文

---

## 2. 模块定位

## 2.1 模块目标
认证模块用于提供系统的基础用户能力，包括：
- 用户注册
- 用户登录
- 当前用户信息获取
- 登录状态校验
- 基础角色区分
- 为其他模块提供用户上下文

## 2.2 模块在系统中的位置
Auth 模块属于系统底层基础业务模块。  
它不直接参与“推荐—规划—查询—日记”的业务逻辑，但几乎所有需要用户身份的功能都依赖它，例如：
- 发布日记
- 日记评分
- 管理端访问
- 个性化偏好读取
- 路线历史保存

## 2.3 当前模块角色
Auth 模块在当前项目里属于：
- **P0 基础模块**
- 必须尽早实现
- 是前后端联调的关键起点之一

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
Auth 模块负责：
1. 用户注册
2. 用户登录
3. 当前用户信息获取
4. token 校验
5. 基础角色区分
6. 为其他模块提供登录用户上下文
7. 管理端接口的最小权限前置判断

## 3.2 本模块不负责的内容
Auth 模块不负责：
1. 用户复杂画像建模
2. 用户社交关系
3. 推荐算法
4. 路线规划
5. 日记内容管理
6. 复杂权限系统
7. OAuth、多端单点登录等重型能力

## 3.3 模块边界原则
- Auth 模块只做“身份”和“最小权限”；
- 不把用户偏好、日记、路线等业务逻辑塞进认证模块；
- 权限粒度当前保持简单，先满足课程项目的基本需要。

---

## 4. 用户场景与功能范围

## 4.1 核心用户场景
### 场景一：注册账号
用户第一次进入系统，希望注册一个账号。

### 场景二：登录系统
用户使用用户名和密码登录系统，进入推荐、日记、个人中心等功能。

### 场景三：获取当前用户信息
前端页面刷新后，希望知道当前是否已登录，以及当前用户是谁。

### 场景四：管理员访问后台
管理员登录后进入管理端维护数据。

---

## 4.2 当前阶段功能范围

### P0（当前必须优先实现）
1. 用户注册
2. 用户登录
3. 获取当前用户信息
4. token 基础校验
5. 普通用户 / 管理员基础角色区分

### P1（基础版后补充）
1. 修改个人资料
2. 修改密码
3. 退出登录
4. 用户状态禁用控制

### P2（后续增强）
1. 多端登录管理
2. 更细粒度权限控制
3. 登录日志
4. 更复杂账号安全机制

---

## 5. 输入与输出设计

## 5.1 输入

### 5.1.1 注册输入
- `username`
- `password`
- `nickname`

### 5.1.2 登录输入
- `username`
- `password`

### 5.1.3 当前用户查询输入
- 请求头中的 `Authorization` token

---

## 5.2 输出
Auth 模块统一输出：
- `userId`
- `username`
- `nickname`
- `avatarUrl`
- `role`
- `token`（登录时）
- `loginStatus`

增强版中可继续输出：
- `status`
- `lastLoginTime`
- `preferenceProfileSummary`

---

## 6. 数据依赖

## 6.1 主要数据表
Auth 模块主要依赖以下数据表：

### 6.1.1 `user`
保存系统用户账号信息，包括：
- 用户名
- 加密密码
- 昵称
- 头像
- 角色
- 状态
- 创建时间
- 更新时间

### 6.1.2 `user_preference`（弱依赖）
严格来说不属于认证逻辑本体，但当前用户信息返回或个性化扩展中，后续可能会读取用户偏好。

---

## 6.2 数据字段重点
当前认证模块最关键的数据字段包括：
- `user.id`
- `user.username`
- `user.password_hash`
- `user.nickname`
- `user.avatar_url`
- `user.role`
- `user.status`

---

## 7. 模块依赖关系

## 7.1 依赖的系统模块
Auth 模块直接依赖：

### 7.1.1 `Security / JWT`
负责：
- token 生成
- token 解析
- token 校验
- 基础登录态判断

### 7.1.2 统一异常处理
负责：
- 登录失败
- 未登录
- token 无效
- 权限不足等异常的统一返回

### 7.1.3 统一返回体
负责：
- 所有认证接口统一返回 `ApiResponse`

---

## 7.2 被哪些模块依赖
Auth 模块被以下模块依赖：
- Recommend（后续个性化偏好读取）
- Diary（发布日记、评分）
- Route（保存路线历史）
- Admin（后台权限）
- 个人中心页面
- 前端导航栏登录状态展示

---

## 8. 核心设计思路

## 8.1 当前设计目标
Auth 模块当前的目标不是做复杂账号体系，而是做一个**足够稳、足够简单、能支撑全项目**的最小用户系统。

重点保证：
1. 用户能注册
2. 用户能登录
3. 前端能判断当前是否登录
4. 管理端能区分普通用户和管理员

---

## 8.2 密码处理原则
- 不保存明文密码
- 数据库存储 `password_hash`
- 登录时进行加密比对
- 日志中绝不输出原始密码

---

## 8.3 token 处理原则
当前建议采用：
- 登录成功后签发 token
- 前端存储 token
- 后续请求通过 `Authorization: Bearer <token>` 传递
- 后端统一拦截校验

---

## 8.4 角色控制原则
当前角色建议至少分两类：
- `user`
- `admin`

当前权限控制建议保持简单：
- 普通用户：使用前台功能
- 管理员：可进入管理端

不建议当前阶段就做很细粒度的 RBAC。

---

## 9. 业务流程设计

## 9.1 注册流程
1. 用户进入注册页
2. 输入用户名、密码、昵称
3. 前端调用注册接口
4. `AuthController` 接收请求
5. 校验用户名是否重复
6. 对密码进行加密
7. 写入 `user` 表
8. 返回注册成功结果

---

## 9.2 登录流程
1. 用户进入登录页
2. 输入用户名和密码
3. 前端调用登录接口
4. `AuthController` 接收请求
5. 查找用户
6. 校验密码
7. 生成 token
8. 返回 token 和用户信息

---

## 9.3 获取当前用户流程
1. 前端页面初始化时读取本地 token
2. 请求当前用户接口
3. 后端解析 token
4. 查找用户信息
5. 返回当前用户结果

---

## 10. 接口设计范围

## 10.1 注册接口
### `POST /api/v1/auth/register`
用途：
- 注册新用户

请求体建议：
- `username`
- `password`
- `nickname`

返回：
- `userId`
- 注册结果

---

## 10.2 登录接口
### `POST /api/v1/auth/login`
用途：
- 用户登录

请求体建议：
- `username`
- `password`

返回：
- `token`
- 当前用户信息

---

## 10.3 当前用户接口
### `GET /api/v1/auth/me`
用途：
- 获取当前登录用户信息

返回：
- `id`
- `username`
- `nickname`
- `avatarUrl`
- `role`

---

## 10.4 退出登录接口（可选）
### `POST /api/v1/auth/logout`
用途：
- 前后端约定退出登录  
首版可由前端清理本地 token 实现，后续再增强。

---

## 11. 前端页面支持说明

## 11.1 登录页
应至少支持：
- 用户名输入
- 密码输入
- 登录按钮
- 登录失败提示

## 11.2 注册页
应至少支持：
- 用户名输入
- 密码输入
- 昵称输入
- 注册按钮
- 注册成功提示

## 11.3 页面级登录状态
前端应支持：
- 导航栏展示当前用户昵称
- 未登录时展示登录入口
- 已登录时显示退出或个人中心入口

---

## 12. 异常与边界场景

至少需要考虑以下情况：

1. 用户名为空
2. 密码为空
3. 用户名已存在
4. 用户不存在
5. 密码错误
6. 用户被禁用
7. token 缺失
8. token 无效或过期
9. 普通用户访问管理端

这些情况应返回清晰的业务错误信息，不直接暴露底层细节。

---

## 13. 测试要点

## 13.1 功能测试
1. 注册是否成功
2. 重复用户名是否拦截
3. 登录是否成功
4. 密码错误是否提示
5. 获取当前用户信息是否正确
6. 角色区分是否正确

## 13.2 边界测试
1. 空用户名
2. 空密码
3. 极长用户名
4. 非法 token
5. 被禁用用户登录

## 13.3 安全测试
1. 密码是否明文落库
2. 日志中是否泄漏敏感信息
3. token 缺失是否正确拦截
4. 管理端接口是否做角色判断

---

## 14. 实现约束

Auth 模块在实现时，必须遵守以下约束：

1. `AuthController` 不写复杂业务逻辑
2. 不允许明文存储密码
3. 不允许日志中打印密码、token、secret
4. 所有接口统一返回 `ApiResponse`
5. token 解析和校验统一走安全层
6. 不允许字符串拼接 SQL
7. 修改用户表、认证接口或权限判断后，必须同步更新：
   - `schema.md`
   - `api-spec.md`
   - `module-map.md`
   - `dependency-map.md`

当前建议按以下顺序实现 Auth 模块：

### 第一步
实现 `user` 表和基础 Mapper

实现注册接口

- 密码加密存储

### 第三步
实现登录接口

### 第四步
实现当前用户接口
- token 解析
- 用户信息返回

### 第五步
补最小角色校验
- 管理员接口前置判断
