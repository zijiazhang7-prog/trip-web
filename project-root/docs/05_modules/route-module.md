# 路线规划模块设计说明（Route Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“路线规划模块”的设计目标、职责边界、输入输出、核心算法、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：
- 路线规划模块负责什么，不负责什么
- 路线规划模块如何支撑“旅游中”场景
- 路线规划模块依赖哪些数据和公共能力
- 路线规划模块采用什么图建模与选路策略
- 路线规划模块如何与设施查询、日记、前端地图页协作

---

## 2. 模块定位

### 2.1 模块目标
路线规划模块用于支持用户进入景区或校园后的导航需求，帮助用户从当前位置到目标景点或场所规划最优路线，并支持后续的多目标路径规划、时间最优策略、交通工具约束和室内导航扩展。

### 2.2 对应课程要求
课程对路线规划模块的要求非常明确，包括：
1. 用户输入目标景点或场所后，系统应规划从当前位置出发到目标点的最优旅游线路；
2. 用户输入多个目标点后，系统应规划从当前位置出发、参观多个点并返回起点的最优旅游线路；
3. 需要设计导航功能的图形界面，包括地图展示和路径展示；
4. 需要支持最短距离策略；
5. 需要支持考虑拥挤度和理想速度的最短时间策略；
6. 需要支持交通工具约束；
7. 需要提供室内导航示例；
8. 核心算法必须基于自己设计的数据结构实现。:contentReference[oaicite:3]{index=3}

### 2.3 模块在系统中的位置
路线规划模块位于用户端“旅游中”主链路的核心位置。  
用户一般会先通过推荐模块或搜索模块确定目的地，再进入路线规划模块完成：
- 单目标导航
- 多目标游览路线生成
- 路径展示
- 周边设施联动查询

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
路线规划模块负责：
1. 目的地内部地图图结构建模后的路径计算
2. 单目标最短路径规划
3. 多目标路径规划
4. 最短距离与最短时间策略切换
5. 交通工具约束下的选路
6. 路径结果组织与返回
7. 路径可视化所需数据输出
8. 路线历史记录写入

## 3.2 本模块不负责的内容
路线规划模块不负责：
1. 用户登录与权限校验（由 Auth 模块负责）
2. 目的地推荐与搜索（由 Recommend 模块负责）
3. 周边设施结果本体查询（由 Facility 模块负责）
4. 美食推荐（由 Food 模块负责）
5. 日记发布与管理（由 Diary 模块负责）
6. 地图原始数据采集与清洗（由离线数据处理与导入流程负责）

## 3.3 模块边界原则
- 路线规划模块只负责“基于图数据计算路线并输出结果”；
- 不在本模块内直接写周边设施筛选、美食排序或推荐逻辑；
- 图算法与业务编排分层实现，公共图能力优先抽到 `MapService`。

## 4. 用户场景与功能范围

## 4.1 核心用户场景
### 场景一：单目标导航
用户已进入某个景区或校园，想从当前位置前往图书馆、景点、食堂等目标点，系统返回一条最优路线。

### 场景二：多目标游览
用户希望依次参观多个点位，例如“校门 → 图书馆 → 食堂 → 体育馆 → 返回校门”，系统给出访问顺序与整体路线。

### 场景三：策略切换
用户想比较“最短距离”和“最短时间”的结果差异，或者在可选交通工具约束下选择更合适的路线。

### 场景四：室内导航示例
用户进入教学楼或博物馆，希望完成大门到电梯、楼层间、电梯到房间等室内导航。

---

## 4.2 当前阶段功能范围

### P0（当前必须优先实现）
1. 单目标路径规划
2. 地图路径展示数据输出
3. 总距离与预计时间输出
4. 路线历史记录
5. 与周边设施查询的基础联动

### P1（基础闭环后补充）
1. 多目标路径规划
2. 最短时间策略
3. 交通工具约束
4. 起点终点可视化增强
5. 路径步骤说明

### P2（后续增强）
1. 室内导航示例
2. 混合交通工具时间最优
3. 路径偏好解释
4. 与日记回顾联动
5. 更复杂的预算 / 体力 / 时间约束

## 5. 输入与输出设计

## 5.1 输入
路线规划模块的输入主要包括：

### 5.1.1 基础定位输入
- `destinationId`
- `startNodeId`
- `targetNodeId`（单目标）
- `targetNodeIds`（多目标）

### 5.1.2 路径策略输入
- `strategyType`
  - `shortest_distance`
  - `shortest_time`
- `transportType`
  - `walk`
  - `bike`
  - `cart`
  - 后续可扩展 `mixed`

### 5.1.3 多目标规划输入
- 是否返回起点 `returnToStart`
- 目标点列表顺序是否固定（后续可扩展）

---

## 5.2 输出
路线规划模块统一输出：
- `destinationId`
- `strategyType`
- `transportType`
- `totalDistance`
- `estimatedTime`
- `pathNodes`
- `pathEdges`
- `orderedTargets`（多目标时）
- `routeSummary`
- `historyId`（可选）

单目标返回重点：
- 一条路径节点序列
- 总距离
- 预计时间

多目标返回重点：
- 最终访问顺序
- 拼接后的总路径
- 总距离
- 总时间

## 6. 数据依赖

## 6.1 主要数据表
路线规划模块主要依赖以下数据表：

### 6.1.1 `map_node`
保存图中的节点，包括：
- 路口
- 建筑物
- 设施点
- 室内楼层点位

### 6.1.2 `map_edge`
保存图中的边，包括：
- 起点节点
- 终点节点
- 距离
- 理想速度
- 拥挤度系数
- 交通方式
- 边类型
- 是否双向

### 6.1.3 `destination`
用于限定当前规划所在目的地。

### 6.1.4 `route_history`
用于保存用户历史规划路线，支撑“我的路线”“路径回顾”等后续能力。`route_history` 已预留 `path_node_json` 和 `strategy_type` 等字段，适合记录路径结果。:contentReference[oaicite:4]{index=4}

### 6.1.5 `place` / `facility`（辅助）
用于把用户可理解的景点、建筑、设施映射到图节点。

## 7.1 依赖的系统模块
路线规划模块直接依赖：

### 7.1.1 `MapService`

### 7.1.3 `Auth` 模块（弱依赖）

用于保存历史路径结果。

当前后端模块结构图里已经明确：`RouteController → RoutePlanService → MapService → GraphEngine`，并通过 `MapNodeMapper / MapEdgeMapper / RouteHistoryMapper` 访问数据。:contentReference[oaicite:5]{index=5}

## 8. 核心算法设计

## 8.1 设计目标
课程要求路线规划模块必须体现：
- 最短路径算法
- 途经多点最短路径算法
- 时间最优策略
- 交通工具约束
- 室内导航建模思路  
因此本模块的核心不是“简单查表”，而是“**图建模 + 路径搜索 + 约束策略**”。:contentReference[oaicite:6]{index=6}

---

## 8.2 图建模思路

### 8.2.1 图类型
采用**有向带权图**建模。  
原因：
1. 不同方向的通行条件可能不同；
2. 交通工具限制可能导致边方向不对称；
3. 室内楼梯、电梯、单向通道等场景更适合有向图。

### 8.2.2 节点设计
节点可表示：
- 道路交叉口
- 建筑入口
- 景点位置
- 设施点位
- 室内楼层节点
- 电梯口 / 楼梯口

### 8.2.3 边设计
边可表示：
- 普通道路
- 步行通道
- 自行车道
- 电瓶车线路
- 楼梯
- 电梯连接

边上至少包含：
- 距离
- 理想速度
- 拥挤度
- 交通工具类型
- 边类型

---

## 8.3 单目标路径规划算法

### 当前推荐实现
P0 阶段优先采用：
- **Dijkstra** 实现单目标最短路径

### 原因
1. 逻辑清晰
2. 适合有向带权图
3. 便于课程答辩说明
4. 可直接支持最短距离策略

### 时间复杂度
若用优先队列实现：
- 约为 `O((V + E) log V)`

### 输出内容
- 最短路径节点序列
- 总距离
- 基础预计时间

---

## 8.4 最短时间策略

### 思路
在最短时间策略下，不再直接以 `distance` 作为边权，而是根据课程要求计算时间权重：

`realSpeed = crowdFactor * idealSpeed`  
`timeCost = distance / realSpeed`

其中：
- `crowdFactor <= 1`
- `idealSpeed` 和 `crowdFactor` 可由数据表提供

### 当前建议
- P1 阶段在 Dijkstra 基础上切换边权定义即可；
- 不需要重写整套最短路径框架。

课程已明确给出最短时间策略和拥挤度公式。:contentReference[oaicite:7]{index=7}

---

## 8.5 交通工具约束

### 思路
不同交通工具下，只允许走对应类型的边：
- `walk`：可走步行道路、通用道路
- `bike`：只能走允许自行车通行的边
- `cart`：只能走电瓶车路线或允许电瓶车通行的边

### 当前实现建议
P1 阶段先实现“单一交通工具约束”：
- 过滤不合法边
- 在合法子图上运行最短路径

### 后续增强
P2 阶段可扩展“混合交通工具最短时间”。

课程要求中已明确提出校区内步行/自行车、景区内步行/电瓶车等交通工具最短时间策略。:contentReference[oaicite:8]{index=8}

---

## 8.6 多目标路径规划算法

### 问题特点
多目标路径规划本质上属于“从当前位置出发，访问多个目标点并返回起点”的路径优化问题，课程明确将其作为考查点。:contentReference[oaicite:9]{index=9}

### 当前建议
P1 阶段采用“**启发式分解**”实现：
1. 先计算目标点之间的两两最短路径代价；
2. 再用启发式方法求访问顺序；
3. 最后拼接整体路径。

### 可选方法
- 贪心最近邻
- 状态压缩 DP（目标点较少时）
- 启发式搜索
- A* 作为局部搜索增强

### 当前项目建议
- 第一版优先做“目标点数量较少”的可演示实现；
- 不追求大规模 TSP 最优求解。

---

## 8.7 室内导航示例

### 设计目标
室内导航属于课程要求中的增强方向，重点是展示“建筑内部也能建模”，不追求高精定位。:contentReference[oaicite:10]{index=10}

### 当前建议
可选一个简化场景，例如：
- 教学楼
- 图书馆
- 博物馆

建模要点：
- 楼层节点
- 电梯节点
- 楼梯节点
- 房间节点
- 楼层间边

### 实现优先级
P2

## 9. 业务流程设计

## 9.1 单目标路径流程
1. 用户进入导航页
2. 选择目的地、当前位置和目标点
3. 前端调用单目标路径规划接口
4. `RouteController` 接收请求
5. `RoutePlanService` 组织业务逻辑
6. `MapService` 构建子图并执行最短路径
7. 返回路径节点、距离、时间、边信息
8. 前端展示路径与总结果
9. 如用户已登录，可写入 `route_history`

---

## 9.2 多目标路径流程
1. 用户选择多个目标点
2. 前端调用多目标规划接口
3. `RoutePlanService` 调用 `MapService`
4. `MapService` 先计算点对间代价，再求访问顺序
5. 拼接整体路线并返回
6. 前端展示完整行程与顺序

---

## 12. 异常与边界场景

至少需要考虑以下情况：

1. 起点不存在
2. 终点不存在
3. 起点终点不属于同一目的地
4. 节点不可达
5. 图数据缺失
6. 多目标列表为空
7. 交通工具下无合法路径
8. 拥挤度或速度字段异常
9. 楼层节点映射错误（室内导航场景）

这些场景不应直接抛出系统异常给前端，而应返回清晰的业务错误提示，并记录脱敏日志。当前代码规范也明确要求统一返回体、全局异常处理和可排错日志。:contentReference[oaicite:11]{index=11}

## 13. 测试要点

## 13.1 功能测试
1. 单目标路径是否正确
2. 多目标路径是否能返回完整路线
3. 最短距离与最短时间结果是否可切换
4. 交通工具约束是否生效
5. 路径展示数据是否完整
6. 路线历史是否正确写入

## 13.2 算法测试
1. 有向图下路径是否正确
2. 节点和边较少时结果是否正确
3. 图规模扩大后结果是否稳定
4. 拥挤度变化后时间策略是否变化
5. 不可达图场景下是否正确返回失败

## 13.3 边界测试
1. 起点等于终点
2. 多目标只有一个点
3. 图中存在回路
4. 边权为极小值或极大值
5. 某条边缺失速度字段
6. 不同交通工具切换后合法边集合变化

当前代码规范要求核心 service 至少覆盖正常输入、边界输入和异常输入三类最小测试样例。:contentReference[oaicite:12]{index=12}

## 14. 实现约束

路线规划模块在实现时，必须遵守以下约束：

1. `RouteController` 不写复杂业务逻辑
2. 路径计算逻辑统一放在 `MapService`
3. 底层图算法与业务编排分层实现
4. 不允许在 Controller 中直接拼接路径结果
5. 不允许字符串拼接 SQL
6. 路径算法必须写注释，说明：
   - 图结构
   - 算法步骤
   - 时间复杂度
   - 适用范围
7. 所有接口统一返回 `ApiResponse`
8. 路线历史写入与读取需走正式数据访问层
9. 修改图结构或路径接口后，必须同步更新：
   - `schema.md`
   - `api-spec.md`
   - `module-map.md`
   - `dependency-map.md`

这些约束与当前项目代码规范保持一致：单体后端、分层组织、公共能力抽取、统一返回体、复杂算法需说明复杂度。:contentReference[oaicite:13]{index=13}

## 15. 当前阶段实现建议

当前建议按以下顺序实现路线规划模块：

### 第一步
实现图基础能力：
- 读取 `map_node`
- 读取 `map_edge`
- 构建有向带权图

### 第二步
实现单目标最短距离路径：
- Dijkstra
- 路径回溯
- 总距离输出

### 第三步
补充时间策略：
