# 路线规划模块设计说明（Route Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“路线规划模块”的设计目标、职责边界、输入输出、核心算法、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：
- 路线规划模块负责什么，不负责什么
- 路线规划模块如何支撑“旅游中”场景
- 路线规划模块依赖哪些数据和公共能力
- 路线规划模块采用什么图建模与选路策略
- 路线规划模块如何与设施查询、日记、AI 模块协作
- 当前阶段先实现哪些能力，后续如何扩展到创新需求

---

## 2. 模块定位

### 2.1 模块目标
路线规划模块用于支持用户进入景区或校园后的导航需求，帮助用户从当前位置到目标景点或场所规划最优路线，并支持后续的多目标路径规划、时间最优策略、交通工具约束、路线回顾和室内导航扩展。

### 2.2 对应课程要求
课程对路线规划模块的要求非常明确，包括：
1. 用户输入目标景点或场所后，系统应规划从当前位置出发到目标点的最优旅游线路；
2. 用户输入多个目标点后，系统应规划从当前位置出发、参观多个点并返回起点的最优旅游线路；
3. 需要设计导航功能的图形界面，包括地图展示和路径展示；
4. 需要支持最短距离策略；
5. 需要支持考虑拥挤度和理想速度的最短时间策略；
6. 需要支持交通工具约束；
7. 需要提供室内导航示例；
8. 核心算法必须基于自己设计的数据结构实现。

### 2.3 模块在系统中的位置
路线规划模块位于用户端“旅游中”主链路的核心位置。  
用户一般会先通过推荐模块或搜索模块确定目的地，再进入路线规划模块完成：

- 单目标导航
- 多目标游览路线生成
- 路径展示
- 周边设施联动查询
- 后续路线回顾输入生成

### 2.4 当前阶段定位
当前路线规划模块的定位是：

- **P0：单目标路径规划 + 地图展示结果 + 路线历史**
- **P1：多目标路径规划 + 最短时间策略 + 交通工具约束**
- **P2：室内导航、路线回顾增强、外部地图 API 相关增强**

当前阶段路线规划仍以内部图数据和 `MapService` 为主实现，不依赖外部地图 API 才能工作。

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
路线规划模块负责：

1. 目的地内部地图图结构建模后的路径计算
2. 单目标最短路径规划
3. 多目标路径规划
4. 最短距离与最短时间策略切换
5. 交通工具约束下的选路
6. 路径结果组织与返回
7. 路径可视化所需数据输出
8. 路线历史记录写入
9. 为后续路线回顾提供结构化输入来源

## 3.2 本模块不负责的内容
路线规划模块不负责：

1. 用户登录与权限校验（由 Auth 模块负责）
2. 目的地推荐与搜索（由 Recommend 模块负责）
3. 用户偏好保存（由 UserPreference 模块负责）
4. 周边设施结果本体查询（由 Facility 模块负责）
5. 美食推荐（由 Food 模块负责）
6. 日记发布与管理（由 Diary 模块负责）
7. 路线回顾文案生成（由 AIService 后续增强承接）
8. 地图原始数据采集与清洗（由离线数据处理与导入流程负责）
9. 外部地图 API 的接入管理与密钥配置

## 3.3 模块边界原则
- 路线规划模块只负责“基于图数据计算路线并输出结果”；
- 不在本模块内直接写周边设施筛选、美食排序或推荐逻辑；
- 图算法与业务编排分层实现，公共图能力优先抽到 `MapService`；
- 当前不把外部地图 API 作为主路径算法依赖；
- 路线规划先保证可计算、可展示、可记录，再考虑解释性增强和轨迹增强。

## 4. 用户场景与功能范围

## 4.1 核心用户场景

### 场景一：单目标导航
用户已进入某个景区或校园，想从当前位置前往图书馆、景点、食堂等目标点，系统返回一条最优路线。

### 场景二：多目标游览
用户希望依次参观多个点位，例如“校门 → 图书馆 → 食堂 → 体育馆 → 返回校门”，系统给出访问顺序与整体路线。

### 场景三：策略切换
用户想比较“最短距离”和“最短时间”的结果差异，或者在可选交通工具约束下选择更合适的路线。

### 场景四：偏好增强规划（后续）
用户除最短距离外，还可能关心：
- 开放时间
- 适合游玩时长
- 性价比
- 与个人偏好的匹配程度

这些因素后续可作为规划增强维度进入模块，但当前阶段不阻塞基础路径算法实现。

### 场景五：路线回顾与日记联动（后续）
用户完成一次游览后，希望系统根据路线历史、地点完成记录或后续轨迹数据，生成路线回顾说明或为日记 / 手账提供结构化输入。

### 场景六：室内导航示例
用户进入教学楼或博物馆，希望完成大门到电梯、楼层间、电梯到房间等室内导航。

4. 路线历史记录

3. 交通工具约束
4. 起点终点可视化增强
5. 路径步骤说明

### P2（后续增强）
1. 室内导航示例
2. 混合交通工具时间最优

4. 与日记回顾联动
5. 更复杂的预算 / 体力 / 时间约束
6. 外部地图 API 接入后的轨迹回顾增强

## 8. 核心算法设计

## 8.1 设计目标
课程要求路线规划模块必须体现：

- 最短路径算法
- 途经多点最短路径算法
- 时间最优策略
- 交通工具约束
- 室内导航建模思路

因此本模块的核心不是“简单查表”，而是“**图建模 + 路径搜索 + 约束策略**”。

---

## 8.2 图建模思路

### 8.2.1 图类型
采用**有向带权图**建模。  
原因：

1. 不同方向的通行条件可能不同；
2. 交通工具限制可能导致边方向不对称；
3. 室内楼梯、电梯、单向通道等场景更适合有向图。

### 8.2.2 节点设计
节点可表示：
- 道路交叉口
- 建筑入口
- 景点位置
- 设施点位
- 室内楼层节点
- 电梯口 / 楼梯口

### 8.2.3 边设计
边可表示：
- 普通道路
- 步行通道
- 自行车道
- 电瓶车线路
- 楼梯
- 电梯连接

边上至少包含：
- 距离
- 理想速度
- 拥挤度
- 交通工具类型
- 边类型

---

## 8.3 单目标路径规划算法

### 当前推荐实现
P0 阶段优先采用：
- **Dijkstra** 实现单目标最短路径

### 原因
1. 逻辑清晰
2. 适合有向带权图
3. 便于课程答辩说明
4. 可直接支持最短距离策略

### 时间复杂度
若用优先队列实现：
- 约为 `O((V + E) log V)`

### 输出内容
- 最短路径节点序列
- 总距离
- 基础预计时间

---

## 8.4 最短时间策略

### 思路
在最短时间策略下，不再直接以 `distance` 作为边权，而是根据课程要求计算时间权重：

`realSpeed = crowdFactor * idealSpeed`  
`timeCost = distance / realSpeed`

其中：
- `crowdFactor <= 1`
- `idealSpeed` 和 `crowdFactor` 可由数据表提供

### 当前建议
- P1 阶段在 Dijkstra 基础上切换边权定义即可；
- 不需要重写整套最短路径框架。

---

## 8.5 交通工具约束

### 思路
不同交通工具下，只允许走对应类型的边：

- `walk`：可走步行道路、通用道路
- `bike`：只能走允许自行车通行的边
- `cart`：只能走电瓶车路线或允许电瓶车通行的边

### 当前实现建议
P1 阶段先实现“单一交通工具约束”：
- 过滤不合法边
- 在合法子图上运行最短路径

### 后续增强
P2 阶段可扩展“混合交通工具最短时间”。

---

## 8.6 多目标路径规划算法

### 问题特点
多目标路径规划本质上属于“从当前位置出发，访问多个目标点并返回起点”的路径优化问题。

### 当前建议
P1 阶段采用“**启发式分解**”实现：

1. 先计算目标点之间的两两最短路径代价；
2. 再用启发式方法求访问顺序；
3. 最后拼接整体路径。

### 可选方法
- 贪心最近邻
- 状态压缩 DP（目标点较少时）
- 启发式搜索
- A* 作为局部搜索增强

### 当前项目建议
- 第一版优先做“目标点数量较少”的可演示实现；
- 不追求大规模 TSP 最优求解。

---

## 8.7 规划增强维度（后续）
创新需求对路线规划提出了更贴近真实旅游场景的增强方向，后续可纳入以下因素：

- 开放时间
- 适合游玩时长
- 性价比
- 用户偏好
- 人数与同行约束

当前处理原则：

1. 这些因素先进入设计预留；
2. 不改变当前 P0 主路径算法；
3. 若后续正式实现，应同步更新：
   - `schema.md`
   - `api-spec.md`
   - `module-map.md`
   - `dependency-map.md`
   - `ADR-004-algorithm-choice.md`

## 9. 业务流程设计

## 9.1 单目标路径流程
1. 用户进入导航页
2. 选择目的地、当前位置和目标点
3. 前端调用单目标路径规划接口
4. `RouteController` 接收请求
5. `RoutePlanService` 组织业务逻辑
6. `MapService` 构建子图并执行最短路径
7. 返回路径节点、距离、时间、边信息
8. 前端展示路径与总结果
9. 如用户已登录，可写入 `route_history`

---

## 9.2 多目标路径流程
1. 用户选择多个目标点
2. 前端调用多目标规划接口
3. `RoutePlanService` 调用 `MapService`
4. `MapService` 先计算点对间代价，再求访问顺序
5. 拼接整体路线并返回
6. 前端展示完整行程与顺序

---

## 9.3 路线回顾输入生成流程（后续）
1. 用户完成一次路线规划或实际游览
2. 系统保存 `route_history`
3. Diary 或 AI 模块需要生成路线回顾时，读取路线历史和已访问点信息
4. Route 模块提供结构化路径结果
5. AIService 基于这些输入生成路线回顾说明

说明：
- 当前阶段先保留 `route_history`；
- 路线回顾文案生成属于 AI 模块职责，不属于 Route 模块本体。

## 12. 异常与边界场景

至少需要考虑以下情况：

1. 起点不存在
2. 终点不存在
3. 起点终点不属于同一目的地
4. 节点不可达
5. 图数据缺失
6. 多目标列表为空
7. 交通工具下无合法路径
8. 拥挤度或速度字段异常
9. 楼层节点映射错误（室内导航场景）
10. 外部地图 API 不可用（后续）
11. 路线历史不存在但用户请求回顾（后续）

这些场景不应直接抛出系统异常给前端，而应返回清晰的业务错误提示，并记录脱敏日志。

## 13. 测试要点

## 13.1 功能测试
1. 单目标路径是否正确
2. 多目标路径是否能返回完整路线
3. 最短距离与最短时间结果是否可切换
4. 交通工具约束是否生效
5. 路径展示数据是否完整
6. 路线历史是否正确写入
7. 路线历史是否可作为后续回顾输入来源

## 13.2 算法测试
1. 有向图下路径是否正确
2. 节点和边较少时结果是否正确
3. 图规模扩大后结果是否稳定
4. 拥挤度变化后时间策略是否变化
5. 不可达图场景下是否正确返回失败

## 13.3 边界测试
1. 起点等于终点
2. 多目标只有一个点
3. 图中存在回路
4. 边权为极小值或极大值
5. 某条边缺失速度字段
6. 不同交通工具切换后合法边集合变化
7. 开放时间 / 游玩时长约束尚未实现时是否能正确忽略这些输入（后续）

## 14. 实现约束

路线规划模块在实现时，必须遵守以下约束：

1. `RouteController` 不写复杂业务逻辑
2. 路径计算逻辑统一放在 `MapService`
3. 底层图算法与业务编排分层实现
4. 不允许在 Controller 中直接拼接路径结果
5. 不允许字符串拼接 SQL
6. 路径算法必须写注释，说明：
   - 图结构
   - 算法步骤
   - 时间复杂度
   - 适用范围
7. 所有接口统一返回 `ApiResponse`
8. 路线历史写入与读取需走正式数据访问层
9. 当前路线规划主实现不依赖外部地图 API
10. 在未确定具体地图 API 前，不在 Route 模块中直接绑定某一家地图服务 SDK
11. 修改图结构、路径接口或历史记录结构后，必须同步更新：
   - `schema.md`
   - `data-dictionary.md`
   - `api-spec.md`
   - `swagger-draft.yaml`
   - `module-map.md`
   - `dependency-map.md`
   - `ai-module.md`

## 15. 当前阶段实现建议

当前建议按以下顺序实现路线规划模块：

### 第一步
实现图基础能力：
- 读取 `map_node`
- 读取 `map_edge`
- 构建有向带权图

### 第二步
实现单目标最短距离路径：
- Dijkstra
- 路径回溯
- 总距离输出

### 第三步
补充路线历史记录：
- 保存 `path_node_json`
- 保存策略类型
- 保存距离与时间摘要

### 第四步
补充时间策略：
- 引入 `ideal_speed`
- 引入 `crowd_factor`
- 切换边权

### 第五步
实现多目标路径启发式版本

### 第六步
再考虑：
- 开放时间 / 游玩时长 / 性价比等增强输入
- 路线回顾输入结构
- 室内导航示例
- 外部地图 API 评估与接入
