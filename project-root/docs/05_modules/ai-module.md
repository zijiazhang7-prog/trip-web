# AI 模块设计说明（AI Module）

## 1. 文档用途
本文件用于说明个性化旅游系统中“AI 模块”的设计目标、职责边界、输入输出、能力范围、依赖关系、接口范围和测试要点，为后续编码、联调、测试和答辩说明提供依据。

本文件重点回答以下问题：

- AI 模块在系统里到底负责什么
- AI 模块如何为日记、推荐、多人规划、路线回顾等功能提供支撑
- AI 模块依赖哪些数据和公共能力
- AI 模块当前阶段先做什么、后做什么
- AI 模块如何避免喧宾夺主、拖垮基础主线
- 如果后续接入外部 AI API，应如何管理边界和配置

---

## 2. 模块定位

## 2.1 模块目标
AI 模块用于为系统提供统一的智能能力接入层，支持以下几类增强功能：

1. 旅游日记草稿生成
2. 图片内容摘要与标签提取
3. 手账式旅行记录增强
4. 路线回顾说明生成
5. 多人旅游规划协商说明
6. 推荐理由解释文本生成
7. 后续多模态生成能力接入

---

## 2.2 模块在系统中的位置
AI 模块不属于最小闭环里必须最先实现的业务模块，而是一个**公共增强能力层**。

它更适合作为以下模块的“能力提供方”：

- Diary 模块
- Recommend 模块
- Route 模块
- UserPreference 模块
- 后续多人规划创新功能
- 后续轨迹回顾创新功能

当前架构中，`AIService` 被放在公共能力层，而不是某个单独业务模块内部，这个定位是合理的。

---

## 2.3 当前模块角色
在当前项目里，AI 模块属于：

- **P2 创新增强模块**
- 先预留统一入口
- 不阻塞推荐—规划—查询—日记基础闭环
- 后续在基础功能稳定后逐步接入

---

## 2.4 当前模块原则
AI 模块必须遵守以下原则：

1. 不替代课程要求中的核心算法实现；
2. 不替代基础推荐排序、最短路径、图上距离计算等主线算法；
3. 不因为 AI 服务失败影响基础功能可用性；
4. 不在业务模块中分散调用外部模型；
5. 所有 AI 能力统一通过 `AIService` 封装；
6. 外部 AI API 的接入方式、密钥配置、调用限制和降级方案，应单独记录到外部 API 管理文档中。

---

## 3. 模块职责边界

## 3.1 本模块负责的内容
AI 模块负责：

1. 封装与外部 AI 能力的调用；
2. 统一组织 prompt、上下文和输出格式；
3. 为业务模块返回结构化 AI 结果；
4. 生成日记草稿、摘要、标题建议和标签；
5. 支持图片摘要与标签提取；
6. 支持手账式旅行记录的内容组织建议；
7. 生成路线回顾说明；
8. 支持多人旅游规划协商说明；
9. 生成推荐理由解释文本；
10. 管理 AI 调用失败时的兜底逻辑。

---

## 3.2 本模块不负责的内容
AI 模块不负责：

1. 用户登录与权限校验（由 Auth 模块负责）；
2. 日记最终内容的保存（由 Diary 模块负责）；
3. 推荐算法本体（由 Recommend 模块负责）；
4. 路线规划算法本体（由 Route 模块负责）；
5. 地图图结构建模和最短路径计算（由 MapService / Route 模块负责）；
6. 原始媒体文件上传（由 FileService 负责）；
7. 高德地图或其他地图 API 的直接业务封装；
8. 大模型平台本身的训练与部署；
9. 复杂 Agent 平台本体实现；
10. 社区评论治理和恶意内容过滤完整方案。

---

## 3.3 模块边界原则
- AI 模块只做“智能能力封装”，不直接接管业务主流程；
- 所有 AI 调用结果应返回给业务模块，由业务模块决定是否落库、展示或继续处理；
- 不允许把课程要求中的核心算法直接外包给 AI 黑箱；
- 不允许因为 AI 能力不稳定而影响基础功能可用性；
- AI 生成内容必须允许用户确认、编辑或放弃；
- AI 输出不得直接自动发布为正式日记、评论或公开内容。

---

## 4. 用户场景与功能范围

## 4.1 核心用户场景

### 场景一：AI 日记草稿生成
用户上传照片、选择目的地、输入少量游玩信息，系统生成一篇可编辑的日记草稿。

### 场景二：图片摘要与标签提取
用户上传图片后，系统自动提取图片内容标签，用于辅助生成标题、关键词或页面展示。

### 场景三：手账式旅行记录增强
用户在旅游过程中持续记录地点、图片、文字和评价。系统后续可将这些内容组织成一次完整旅行手账，并提供摘要、标签、内容组织建议。

### 场景四：路线回顾说明
用户完成路线规划并结束游览后，系统根据路线记录、地点完成记录或后续轨迹数据生成简短的行程回顾说明。

### 场景五：多人旅游规划协商
多个用户或多个偏好参与同一份路线 / 行程方案的规划时，AI 模块输出协商后的方案说明、理由摘要、冲突点和候选建议。

### 场景六：推荐理由解释
推荐模块返回目的地列表时，AI 模块为前几项生成可读的推荐理由说明。

---

## 4.2 当前阶段功能范围

### P0（当前不强制实现）
- 仅预留 `AIService` 能力接口和模块位置；
- 不要求 AI 功能成为验收主线阻塞项；
- 用户偏好标签输入虽然可作为 AI 输入，但其基础实现属于 UserPreference，不属于 AI 模块本体。

### P1（基础闭环后可补）
1. 手账式旅行记录基础形态与 AI 模块的数据衔接；
2. 路线回顾所需的 route_history 数据准备；
3. AI 输入所需的偏好、地点、日记、路线等上下文整理。

### P2（创新功能重点）
1. 日记草稿生成；
2. 图片标签提取；
3. 路线回顾文案生成；
4. 多人旅游规划协商；
5. 推荐理由解释；
6. AI 日记增强排版；
7. 多模态素材整合输出。

## 8. 核心设计思路

## 8.1 当前设计目标
AI 模块当前的目标不是做“通用聊天机器人”，而是做一个：

- 输入明确
- 输出可控
- 结构化返回
- 能被业务模块稳定调用
- 失败时可降级

的“任务型 AI 能力层”。

---

## 8.2 统一封装原则
所有 AI 能力调用都应通过统一服务层封装，至少做到：

1. 输入结构统一
2. prompt 模板统一管理
3. 输出格式统一
4. 错误处理统一
5. 调用失败有兜底
6. 不暴露真实 API Key
7. 不让外部模型结果直接绕过业务校验

不建议在各个业务模块里到处直接拼 prompt 和直接调用模型接口。

---

## 8.3 Prompt 设计原则
每种 AI 任务应有独立 prompt 模板，例如：

- 日记草稿生成模板
- 图片摘要模板
- 手账增强模板
- 路线回顾模板
- 多人协商模板
- 推荐解释模板

模板中应尽量包含：

- 明确任务目标
- 明确输入字段
- 明确输出 JSON 结构
- 明确语气和长度限制
- 明确不要编造事实
- 明确不要输出与用户输入无关的虚构地点、路线或评价

这些模板可统一收录到：

- `project-root/docs/08_ai/prompt-library.md`

---

## 8.4 结构化输出原则
AI 模块输出应尽量采用结构化 JSON，而不是直接返回一大段不可控文本。

例如日记生成输出示例：

```json
{
  "titleSuggestion": "春日校园漫步记",
  "contentDraft": "今天在校园里走了很久……",
  "tagList": ["校园", "图书馆", "春天"],
  "summary": "一段轻松的校园游览经历"
}
```

这样更便于：

- 前端展示
- 用户编辑
- 后端落库
- 错误处理

---

## 8.5 AI 调用失败兜底原则

AI 能力不是基础主线刚需，因此必须有失败兜底：

1. 调用失败时，不影响主业务流程；
2. 返回“生成失败，请手动编辑”类提示；
3. 不允许因为 AI 服务异常而导致日记发布页、推荐页、回顾页整体不可用；
4. 调用失败要记录可排错日志，但不得泄露敏感信息；
5. 多人协同决策生成失败时，应允许用户继续使用基础路线规划结果；
6. 外部 API 不可用时，不影响基础推荐、路径规划、设施查询和日记发布。

---

## 9. 业务流程设计

## 9.1 AI 日记草稿生成流程

1. 用户进入日记发布页
2. 上传图片，填写少量提示信息
3. 前端调用 AI 日记生成接口
4. `AIModule / AIService` 整理上下文
5. 调用外部 AI 能力
6. 返回标题建议、正文草稿、标签
7. 前端展示草稿，用户确认或修改后再调用 Diary 发布接口

---

## 9.2 图片摘要流程

1. 用户上传图片
2. 前端将图片地址传给 AI 模块
3. AI 模块生成：
    - 场景摘要
    - 标签
    - 可能的标题关键词
4. 前端展示结果或交给日记生成流程使用

---

## 9.3 手账增强流程

1. 用户在旅游过程中记录多个地点内容
2. 系统将地点记录、图片、文字和评价组织为一次旅行记录
3. 用户在手账页请求 AI 辅助整理
4. AI 模块生成手账标题、摘要、标签或内容组织建议
5. 用户确认后保存或继续编辑

---

## 9.4 路线回顾流程

1. 用户完成一次路线规划并保存历史
2. 前端在回顾页发起请求
3. AI 模块读取 `route_history` 或地点完成记录
4. 生成一段简要路线回顾说明
5. 前端展示为“本次游览回顾”

---

## 9.5 多人旅游规划协商流程

1. 用户输入多人的偏好或约束条件
2. 系统先生成候选路线 / 目的地
3. AI 模块根据不同成员偏好模拟讨论
4. 输出：
    - 最终建议
    - 理由摘要
    - 冲突点说明
    - 备选建议
5. 用户自行决定是否采纳方案

---

## 9.6 推荐理由生成流程

1. Recommend 模块生成基础推荐结果
2. 前端或后端请求 AI 推荐理由
3. AI 模块读取用户偏好和候选目的地摘要
4. 返回推荐理由和匹配偏好说明
5. 前端展示给用户

---

## 12. 异常与边界场景

至少需要考虑以下情况：

1. 外部 AI 服务调用失败
2. 图片地址无效
3. 输入文本为空
4. 路线记录不存在
5. AI 输出不符合 JSON 结构
6. 模型返回内容过长
7. 用户未登录却尝试调用需要身份上下文的功能
8. 多人输入信息不完整
9. 外部 API 超时
10. AI 返回内容与输入不相关
11. AI 返回内容包含明显虚构事实
12. 私有日记内容被误用于公开展示

这些情况都应有清晰兜底，不影响基础主线功能。

1. AI 返回慢时前端是否可接受

3. 重复调用是否能得到基本可用输出
4. 输出字段缺失时系统是否能处理
5. 外部服务不可用时基础主线是否仍然可用

1. 空图片列表

## 14. 实现约束

AI 模块在实现时，必须遵守以下约束：

1. AI 模块只做智能能力封装，不直接保存最终业务数据
2. 所有 AI 调用统一走 `AIService`
3. prompt 模板应统一管理，不分散硬编码
4. 输出尽量要求结构化 JSON
5. AI 调用失败不能拖垮主业务流程
6. 不允许把核心算法直接交给 AI 黑箱替代
7. 不允许在日志中记录敏感输入全文
8. 真实 API Key 不得进入仓库
9. 前端不应直接暴露外部模型 API Key
10. 当前 AI 能力统一通过 `AIService` 抽象；在未确定具体模型 API 前，先使用统一输入输出结构和 mock / stub 实现，不在业务模块中直接绑定具体模型厂商。
11. 修改 AI 输入输出结构后，必须同步更新：
    - `api-spec.md`
    - `swagger-draft.yaml`
    - `prompt-library.md`
    - `module-map.md`
    - `dependency-map.md`
    - `test-cases.md`

## 15. 当前阶段实现建议

当前建议按以下顺序实现 AI 模块：

### 第一步

预留 `AIService` 接口与统一调用结构。

### 第二步

先做最稳的能力：

- 日记草稿生成
- 图片标签提取

### 第三步

再补手账摘要和路线回顾文案生成。

### 第四步

最后补多人规划协商和推荐解释。

### 第五步

如果时间允许，再评估：

- 高德地图 API
- 多模态 AI 模型 API
- 轨迹记录能力
- 手账自动排版能力

### 当前重要原则

- AI 模块一定后置于基础主线；
- 不要让它阻塞推荐、路线规划、设施查询、日记发布这些必做功能；
- 外部 API 接入前，必须先明确密钥管理、调用限制、失败降级和隐私边界。
