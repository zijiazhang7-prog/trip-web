# 项目总览（Project Overview）

## 1. 项目名称
个性化旅游系统的设计与实现

---

## 2. 项目背景
当前用户在旅游活动中，通常会经历“旅游前选择目的地、旅游中规划路线与查询场所、旅游后记录和分享经历”这几个连续场景。传统旅游工具往往只能覆盖单一环节，例如只提供景点信息查询、只提供地图导航，或者只提供游记浏览，难以形成完整闭环。

本项目拟设计并实现一个面向旅游活动全流程的 Web 个性化旅游系统，围绕“旅行前推荐—旅行中规划与查询—旅行后日记记录与交流”构建完整功能链路。系统既要满足课程设计中的业务功能要求，也要体现数据结构课程的特点，将排序、查找、图、最短路径、多点路径、全文检索、压缩等数据结构与算法真正落到系统实现中。

---

## 3. 项目目标
本项目的总体目标如下：

1. 实现一个可远程访问的 Web 个性化旅游系统；
2. 覆盖旅游推荐、旅游路线规划、场所查询、周边设施查询、旅游日记等核心功能；
3. 在系统中实现并展示排序、查找、图、路径规划等核心算法；
4. 使用智能体和 AI 工具辅助需求分析、系统设计、编码、测试和文档整理；
5. 形成可用于课程验收、中期汇报与最终演示的代码、文档和测试材料。

---

## 4. 项目定位
本项目的定位为：

- **系统形态**：Web 远程系统
- **架构方式**：B/S 架构、前后端分离、单体后端、算法模块化
- **项目类型**：课程设计项目
- **团队规模**：3 人协作
- **开发方式**：文档驱动 + 模块化开发 + AI 辅助开发

---

## 5. 当前范围定位

### 5.1 当前 P0 主线范围
当前阶段优先保证以下最小闭环跑通：

#### 旅行前
- 用户注册 / 登录
- 用户偏好输入
- 目的地推荐
- 目的地搜索与筛选
- 目的地详情查看

#### 旅行中
- 单目标路线规划
- 周边设施查询
- 图上可达距离 / 时间排序
- 路线历史基础记录

#### 旅行后
- 图文日记发布
- 日记列表与详情查看
- 按目的地查看相关日记
- 文件上传与媒体访问

### 5.2 当前 P1 增强范围
在基础主线跑通后继续补充：

- 多目标路径规划
- 最短时间策略
- 美食推荐基础版
- 日记评分
- 日记标题查询与全文检索
- 管理端基础数据维护
- 数据导入流程完善

### 5.3 当前 P2 创新与增强范围
后续再逐步补充：

- AI 日记草稿生成
- 图片摘要 / 标签提取
- AI 路线回顾
- 推荐理由解释
- 多人旅游规划协商
- 更复杂的手账与多模态展示能力

- 当前 AI 能力统一通过 `AIService` 预留
- 外部地图 API 和外部模型 API 当前均未最终选型
- 当前 MVP 主链路不依赖外部地图或外部 AI 服务才能跑通

## 7. 当前创新方向（已预留，后续接入）
在完成课程要求的基础功能之外，项目当前已明确预留以下创新方向：

### 7.1 多模态日记与路线回顾
- 根据用户路线、图片与文字生成更完整的旅游日记草稿
- 支持路线回顾总结
- 支持图片摘要和标签提取
- 为后续手账式旅行记录提供扩展空间

### 7.2 多人旅游规划与协商决策
- 支持多位参与者输入偏好
- 输出协商后的候选方案与理由说明
- 由用户决定是否采纳

> 说明：当前创新方向已经纳入设计预留，但当前阶段仍不作为 P0 主链路阻塞项。

## 8. 当前模块划分（概要）

### 8.1 核心业务模块
- 认证模块（Auth）
- 用户偏好模块（UserPreference）
- 推荐模块（Recommend）
- 路线规划模块（Route）
- 场所 / 周边设施模块（Facility）
- 美食模块（Food）
- 日记模块（Diary）
- 管理模块（Admin）

### 8.2 公共能力模块
- QueryService
- RankService
- MapService
- SearchService
- FileService
- ImportService
- AIService

### 8.3 基础支撑能力
- Security / JWT
- Config / Exception / Response
- GraphEngine / IndexEngine / CompressionEngine / ImportEngine

模块之间通过统一的数据、查询、排序和算法能力协同工作，公共能力尽量抽取为统一服务层，避免重复实现。

## 9. 当前开发原则
为保证项目可推进、可协作、可交付，当前确定以下开发原则：

1. 一次只实现一个模块或一个明确子功能；
2. 优先完成最小可运行版本，再逐步增强；
3. 文档优先于实现，先明确需求与设计，再写代码；
4. 数据库、接口、模块职责变化后必须同步文档；
5. 核心算法需与课程要求保持一致，不用黑箱能力替代最终实现；
6. 优先保证“推荐—规划—查询—日记”这条主线闭环跑通，再补 P1 / P2 能力；
7. 外部服务统一通过抽象服务层接入，不直接散落在业务模块中。
