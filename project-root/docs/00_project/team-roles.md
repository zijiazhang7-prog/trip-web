# 团队角色与分工（Team Roles）

## 1. 文件用途
本文件用于明确小组成员在项目中的角色定位、责任边界、协作方式和交付责任，避免出现任务重叠、责任空白或互相等待的问题。

## 2. 小组基本信息
- 项目名称：个性化旅游系统的设计与实现
- 小组人数：3 人
- 项目类型：数据结构课程设计
- 开发方式：文档驱动 + 模块化开发 + AI 辅助开发

## 3. 分工原则
本项目的分工遵循以下原则：

1. **主责明确**：每个核心模块都必须有唯一主负责人。
2. **协作补位**：一个模块可以多人参与，但必须明确谁主导、谁支持。
3. **文档同步**：负责人不仅负责代码，也负责相关文档同步。
4. **模块闭环**：每个人负责的工作应尽量形成相对完整的小闭环，而不是只负责零碎片段。
5. **阶段动态调整**：在项目不同阶段，成员重心可以变化，但主责不应频繁变化。
6. **先主线后增强**：优先保障推荐—规划—查询—日记这条主线跑通，再考虑增强功能。

## 4. 当前建议角色划分

### 成员 A：后端主责 / 推荐与接口文档主导
**角色定位：**
后端主责成员，负责系统主要业务接口和推荐相关能力的落地。

**主要职责：**
- 负责后端工程骨架搭建
- 负责认证、推荐、部分公共服务层实现
- 负责 API 文档初稿与维护
- 参与数据库设计和接口联调
- 协助统一返回体、异常处理和后端规范落地

**当前重点模块建议：**
- Auth 模块
- Recommend 模块
- API 设计与接口联调
- 公共服务层中的 QueryService / RankService

---

### 成员 B：算法主责 / 路线规划与数据处理主导
**角色定位：**
算法与数据主责成员，负责地图建模、路径规划、数据导入和算法相关实现。

**主要职责：**
- 负责路线规划模块的核心算法实现
- 负责图结构设计、最短路径、多点路径等算法实现
- 负责地图节点、边、设施等数据结构设计
- 负责数据清洗、导入脚本和样例数据准备
- 协助测试阶段的算法正确性验证

**当前重点模块建议：**
- Route 模块
- Facility 模块中的可达距离计算
- 数据导入脚本
- 公共服务层中的 MapService / SearchService（如参与全文检索）

---

### 成员 C：前端主责 / 日记模块与测试交付主导
**角色定位：**
前端与交付主责成员，负责用户界面、日记模块、测试整理和答辩展示支持。

**主要职责：**
- 负责前端页面骨架与页面联调
- 负责日记模块和部分 AI 创新功能的前端交互
- 负责测试文档、用户手册、演示脚本和交付材料整理
- 协助日记相关接口联调和媒体展示
- 协助周报、进度和答辩材料整理

**当前重点模块建议：**
- Diary 模块
- 前端页面整体骨架
- 测试文档与交付文档
- AI 模块中的前端交互部分

## 5. 当前阶段任务分配建议

### 5.1 文档阶段
在尚未正式开发前，建议三人优先按以下方向推进：

- 成员 A：需求收敛、接口草案、后端规则整理
- 成员 B：算法方案、数据结构方案、数据来源与导入方案
- 成员 C：前端页面结构、日记交互设计、测试与交付模板整理

### 5.2 开发阶段
进入正式开发后，建议按以下方式推进：

- 成员 A：后端主链路接口与推荐模块
- 成员 B：路线规划、设施查询与数据导入
- 成员 C：前端主页面、日记模块、测试与演示材料

### 5.3 联调阶段
联调时建议按以下方式协作：

- 成员 A：接口修正、联调问题定位
- 成员 B：算法结果校验、路径与数据问题修正
- 成员 C：页面联调、交互修正、用户操作路径验证

## 7. 协作规则

### 7.1 Git 协作
1. 不直接在主分支上随意修改核心内容。
2. 代码、接口、数据库结构修改前应先确认当前文档版本。
3. 提交前尽量保证：
   - 代码可运行
   - 变更说明清楚
   - 涉及文档已同步或已明确 TODO

### 7.2 文档协作
1. 每个文档应有主维护人。
2. 他人可补充，但不应在未沟通的情况下大幅改动他人主文档结构。
3. 需求、架构、数据、接口类文档修改后，需同步告知相关成员。

### 7.3 AI 工具协作
1. 使用 AI 工具前应先明确当前任务边界。
2. 不同成员生成的代码和文档应回到仓库规则统一校验。
3. AI 输出不能直接视为最终事实，必须以项目文档和已确认方案为准。
4. 涉及数据库、API、模块职责变更时，必须先同步文档再扩散实现。

## 8. 组会建议
建议每周至少进行一次小组同步，内容包括：

- 本周完成情况
- 当前阻塞问题
- 文档是否同步
- 下周计划
- 是否需要调整分工或优先级

组会结束后建议同步更新：
- `docs/00_project/progress.md`
- `docs/00_project/risk-log.md`
- 必要时更新相关模块文档

## 9. 当前阶段特别提醒
由于项目目前仍处于文档完善与方案收敛阶段，当前更重要的是：

1. 先把项目文档和边界收敛清楚；
2. 先明确最小可运行闭环；
3. 先确定基础功能实现顺序；
4. 不要过早被创新需求和实现细节拖乱节奏。
