# 团队角色与分工（Team Roles）

## 1. 文件用途
本文件用于明确小组成员在项目中的角色定位、责任边界、协作方式、当前阶段分工和交付责任，避免出现任务重叠、责任空白、多人互相等待或文档与实现脱节的问题。

本文件主要回答以下问题：

- 三个成员当前分别主负责什么
- 每个模块由谁主导、谁协作
- 每类文档由谁主维护
- 当前阶段应该先做哪些事情、后做哪些事情
- 开发、联调、测试和交付阶段如何协作

---

## 4. 分工原则
本项目的分工遵循以下原则：

1. **主责明确**：每个核心模块必须有唯一主负责人。
2. **协作补位**：一个模块可以多人参与，但必须明确谁主导、谁支持。
3. **文档同步**：负责人不仅负责代码，也负责相关文档同步。
4. **模块闭环**：每个人尽量围绕若干模块形成相对完整的小闭环，而不是只负责零碎片段。
5. **先主线后增强**：优先保障“推荐—规划—查询—日记”这条主线跑通，再考虑增强功能。
6. **公共能力优先抽取**：查询、排序、图建模、文件、检索、导入等能力优先按公共服务层管理，不重复实现。
7. **阶段可调整，但主责不频繁漂移**：不同阶段重心可以变化，但同一模块不应频繁换主负责人。
8. **AI 只辅助，不替代责任归属**：使用 AI 工具不改变模块归属和负责人责任。

## 5. 当前建议角色划分

## 5.1 成员 A：后端主责 / 认证、偏好、推荐与接口文档主导

### 角色定位
后端主责成员，负责当前 P0 主线中的认证、用户偏好、推荐相关能力，以及后端接口规范和公共查询/排序能力的落地。

### 主要职责
- 负责后端工程骨架与主链路接口的组织
- 负责 Auth、UserPreference、Recommend 相关能力实现
- 负责 QueryService / RankService 等公共能力主导
- 负责 API 文档、Swagger 文档和接口联调口径维护
- 协助统一返回体、异常处理、后端规范和权限基础能力落地
- 协助 Admin 的后端接口框架预留

### 当前重点模块建议
- Auth
- UserPreference
- Recommend
- QueryService
- RankService
- API 文档与 Swagger
- 统一返回体 / 错误码 / 异常处理

---

## 5.2 成员 B：算法与数据主责 / 路线规划、设施查询与导入主导

### 角色定位
算法与数据主责成员，负责图建模、路径规划、设施可达性计算、数据导入与样例数据准备。

### 主要职责
- 负责 Route 模块的核心算法实现
- 负责 Facility 模块中的图上可达距离 / 时间能力
- 负责 MapService 与图结构设计
- 负责地图节点、地图边、场所、设施等数据组织
- 负责数据清洗、导入脚本、样例数据准备和导入联调
- 协助测试阶段的算法正确性验证

### 当前重点模块建议
- Route
- Facility
- MapService
- ImportService
- 数据导入脚本与基础样例数据
- Route / Facility 相关测试支持

---

## 5.3 成员 C：前端与交付主责 / 日记模块、页面联调与测试整理主导

### 角色定位
前端与交付主责成员，负责当前 P0 主线中的页面骨架、日记模块、文件上传联调、测试整理和演示交付材料支持。

- 负责 Diary 模块主链路相关实现与页面交互
- 协助 FileService 在页面端的上传接入与媒体展示
- 负责测试计划、测试用例、测试记录、缺陷记录整理
- 负责用户手册、演示脚本、周报和交付材料整理
- 后续负责 Food、AI 前端交互和演示页补充

### 当前重点模块建议
- Diary
- 前端页面整体骨架
- File 上传前端接入
- 测试文档与交付文档
- 演示链路整理

## 9. 当前阶段协作方式

## 9.1 开发前
在模块正式开工前，必须先确认：

1. 当前模块的前置依赖是否已完成；
2. 当前模块的接口是否已冻结到可开发程度；
3. 当前模块相关表结构是否已明确；
4. 当前模块文档是否已经有可执行版本；
5. 当前模块是否属于 P0 / P1 / P2 的正确优先级。

---

## 9.2 开发中
开发阶段建议按以下方式协作：

- 成员 A 负责接口、公共返回和后端规范收口；
- 成员 B 负责图能力、样例数据和算法正确性收口；
- 成员 C 负责页面可用性、主链路联调和测试记录收口；
- 模块变更时，主负责人必须同步更新相关文档；
- 涉及公共能力的变更，必须先同步相关协作人再修改。

---

## 9.3 联调阶段
联调时建议按以下方式协作：

- **成员 A**
  - 接口字段修正
  - 统一返回与异常问题定位
  - 权限问题定位

- **成员 B**
  - 路径结果校验
  - 图数据问题定位
  - 设施可达性问题定位
  - 导入数据问题定位

- **成员 C**
  - 页面联调
  - 用户操作链路验证
  - 日记与上传交互修正
  - 测试记录同步

---

## 9.4 测试与回归阶段
测试阶段建议：

- 成员 A 主看：Auth / UserPreference / Recommend / API 正确性
- 成员 B 主看：Route / Facility / MapService / 导入数据正确性
- 成员 C 主看：Diary / File / 页面联调 / 测试记录与报告
- 全组共同负责：
  - P0 主线回归
  - 演示链路完整性验证
  - 高优先级 bug 修复确认

## 10. 责任边界说明

### 10.1 代码责任
谁主负责模块，谁先负责：
- 提供可运行版本
- 保证接口与文档一致
- 修复本模块高优先级问题
- 协调该模块联调和测试问题

### 10.2 文档责任
谁主维护文档，谁先负责：
- 更新文档正文
- 保证文档与代码一致
- 在组会或群里同步关键变化

### 10.3 不允许出现的情况
1. 模块无人主负责
2. 代码改了但文档不改
3. 接口改了但不通知相关人
4. 把 P2 创新功能偷偷提升为当前主线阻塞项
5. 依赖没完成就跨模块硬写

---

## 11. 协作规则

## 11.1 Git 协作
1. 不直接在主分支随意修改核心内容；
2. 代码、接口、数据库结构修改前先确认当前文档版本；
3. 提交前尽量保证：
   - 代码可运行
   - 变更说明清楚
   - 涉及文档已同步或已明确 TODO；
4. 涉及公共能力和接口字段的改动，优先小步提交，避免一次性改太多。

## 11.2 文档协作
1. 每个文档应有主维护人；
2. 他人可补充，但不应在未沟通的情况下大幅改动他人主文档结构；
3. 需求、架构、数据、接口、测试类文档修改后，应同步告知相关成员；
4. 以项目文档和已确认 ADR 为准，不以口头说法替代正式文档。

## 11.3 AI 工具协作
1. 使用 AI 工具前先明确当前任务边界；
2. AI 输出不能直接视为最终事实，必须回到项目文档和已确认方案上校验；
3. 不同成员生成的代码和文档必须回到统一仓库规则校验；
4. 涉及数据库、API、模块职责、优先级变更时，必须先同步文档再扩散实现。

## 12. 组会建议
建议每周至少进行一次小组同步，内容包括：

- 本周完成情况
- 当前阻塞问题
- 哪些模块已进入开发 / 联调 / 测试
- 文档是否同步
- 下周计划
- 是否需要调整优先级或分工重心

组会结束后建议同步更新：
- `project-root/docs/00_project/progress.md`
- `project-root/docs/00_project/risk-log.md`
- 必要时更新模块文档、接口文档和测试文档
