# 风险记录（Risk Log）

## 1. 文件用途
本文件用于记录项目推进过程中识别到的主要风险、风险影响、应对措施、责任人和当前状态，帮助小组在开发前和开发过程中及时发现问题、减少返工、避免进度失控。

## 2. 使用原则
1. 本文件记录的是“对项目有影响的不确定因素”，不是日常琐碎问题清单。
2. 每条风险都应尽量写清楚：
   - 风险是什么
   - 会影响什么
   - 当前严重程度如何
   - 准备怎么应对
   - 谁负责跟进
3. 风险状态建议统一为：
   - `未处理`
   - `跟进中`
   - `已缓解`
   - `已关闭`
4. 每周组会或阶段检查时至少更新一次。
5. 已关闭风险不删除，保留记录，便于回顾。

## 3. 风险等级建议
### 概率等级
- 低：短期内不太可能发生
- 中：有一定可能发生
- 高：很可能发生或已经出现苗头

### 影响等级
- 低：影响较小，可快速修复
- 中：会影响部分模块、进度或联调
- 高：会影响核心功能、整体排期或最终验收

## 4. 风险记录模板

| 风险ID | 发现日期 | 风险标题 | 风险描述 | 概率 | 影响 | 当前状态 | 应对措施 | 责任人 | 预计复查时间 | 备注 |
|---|---|---|---|---|---|---|---|---|---|---|
| R-001 | YYYY-MM-DD | 示例：创新需求迟迟未确认 | 创新需求文档未及时完成，可能影响模块设计、数据库设计和接口设计收敛 | 高 | 高 | 跟进中 | 先完成受影响小的文档，创新部分留“待确认”占位 | 待填写 | YYYY-MM-DD | 初期重点风险 |

## 5. 当前阶段建议重点关注的风险类型
当前项目尚未正式开发，建议优先关注以下类型的风险：

1. **需求风险**
   - 需求边界不清
   - 创新需求迟迟未定
   - 需求频繁变动

2. **设计风险**
   - 模块边界不清
   - 数据库设计返工
   - 接口设计与模块设计不一致

3. **技术风险**
   - 技术栈不熟悉
   - 算法实现难度高于预期
   - AI 工具生成代码质量不稳定

4. **协作风险**
   - 分工不清
   - 文档和代码不同步
   - Git 协作不熟练导致冲突

5. **进度风险**
   - 前期文档拖太久
   - 后期联调集中爆雷
   - 测试和答辩准备时间不足

## 6. 填写建议
- 风险标题尽量短，便于快速扫描；
- 风险描述尽量写成“原因 + 影响”的形式；
- 应对措施尽量写成可执行动作，不要只写“加强沟通”“注意一下”；
- 责任人应明确到某一个人，不写“全组”。

## 7. 更新时机
建议在以下场景更新本文件：
- 每周组会后
- 需求发生明显变化后
- 新增技术难点后
- 联调出现系统性问题后
- 测试阶段发现重复性缺陷后
