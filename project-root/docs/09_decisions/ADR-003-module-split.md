# ADR-003：系统模块划分与边界定义

- 状态：Accepted
- 日期：2026-04-23
- 决策人：项目小组
- 关联文档：
  - `docs/02_architecture/module-map.md`
  - `docs/02_architecture/dependency-map.md`
  - `docs/05_modules/*`

---

## 1. 背景

课程要求在概要设计阶段明确总体结构、主模块与子模块划分，并定义模块之间的接口关系。  
当前项目已确定采用 B/S、前后端分离、单体后端、算法模块化路线，因此需要进一步冻结模块边界，避免后续出现：

- 一个模块做太多事
- 控制器直接写业务逻辑
- 跨模块随意调用
- 查询、排序、图能力重复实现

---

## 2. 决策问题

需要确定：

1. 系统主模块有哪些
2. 公共能力是否单独抽层
3. 各模块的职责边界如何划分
4. 哪些模块属于基础主线，哪些属于增强层

---

## 3. 决策结果

系统划分为两类模块：

### 3.1 核心业务模块
- Auth
- Recommend
- Route
- Facility
- Food
- Diary
- Admin

### 3.2 公共能力模块
- QueryService
- RankService
- MapService
- SearchService
- FileService
- ImportService
- AIService

---

## 4. 各模块职责定义

### 4.1 Auth
职责：
- 用户注册
- 用户登录
- 获取当前用户
- 角色与登录态校验

不负责：
- 推荐逻辑
- 日记业务
- 复杂权限中心

### 4.2 Recommend
职责：
- 目的地推荐
- 目的地搜索
- 筛选与排序
- Top-K 输出

不负责：
- 用户登录
- 路径规划
- 文本检索增强
- 文件处理

### 4.3 Route
职责：
- 单目标路径规划
- 多目标路径规划（后续增强）
- 最短距离 / 最短时间策略
- 路径结果输出

不负责：
- 目的地基础查询
- 设施列表直接维护
- AI 总结

### 4.4 Facility
职责：
- 周边设施查询
- 设施类别过滤
- 按图上可达距离 / 时间排序

不负责：
- 地图建模底层实现
- 复杂推荐逻辑

### 4.5 Food
职责：
- 美食推荐
- 菜系过滤
- 按热度 / 评分 / 距离排序
- 模糊查询

不负责：
- 路径规划
- 文件上传
- 日记逻辑

### 4.6 Diary
职责：
- 日记发布
- 日记列表与详情
- 日记媒体关联
- 日记评分
- 目的地相关日记查看

不负责：
- 用户登录
- 地图路径规划
- AI 直接调用细节实现

### 4.7 Admin
职责：
- 目的地维护
- 场所维护
- 设施维护
- 美食维护
- 地图节点与边维护
- 日记管理端查看
- 导入入口

不负责：
- 前台推荐逻辑
- 路径算法实现
- AI 算法实现

---

## 5. 公共能力模块职责定义

### 5.1 QueryService
职责：
- 统一查询入口
- 封装多模块复用的查询逻辑
- 供 Recommend / Facility / Food / Admin 调用

### 5.2 RankService
职责：
- 排序
- Top-K
- 多条件二次排序

### 5.3 MapService
职责：
- 图建模
- 图结构读取
- 最短路径与多点路径底层能力
- 可达距离 / 时间计算

### 5.4 SearchService
职责：
- 文本检索
- 模糊检索
- 倒排索引或类似能力封装

### 5.5 FileService
职责：
- 文件上传
- 文件访问
- 媒体资源路径管理

### 5.6 ImportService
职责：
- 数据导入
- 导入批次执行
- 标准化导入流程承接

### 5.7 AIService
职责：
- AI 增强能力统一接入
- 当前作为 P2 预留层
- 不阻塞基础主线

---

## 6. 模块之间的边界规则

1. Controller 不直接跨模块访问数据库
2. 推荐、设施、美食模块应尽量复用 QueryService 和 RankService
3. 路径相关计算必须统一走 MapService
4. Diary 的媒体处理统一走 FileService
5. AI 能力不得直接散落在 Recommend / Diary / Route 中，统一走 AIService
6. Admin 只负责维护数据与状态，不直接实现复杂前台业务逻辑

---

## 7. 当前优先级

### P0
- Auth
- Recommend
- Route（先单目标）
- Facility
- Diary（基础版）
- QueryService
- RankService
- MapService
- FileService

### P1
- Food
- Admin
- ImportService
- SearchService

### P2
- AIService
- 创新功能扩展

---

## 8. 决策理由

1. 模块拆分要服务于课程主线，而不是追求“看起来复杂”
2. 推荐、设施、美食天然共享查询与排序能力，适合抽公共层
3. 路径规划和场所查询都依赖图能力，MapService 必须单独抽出
4. 日记与媒体处理应分离，避免 Diary 模块职责过重
5. Admin 应作为后台支撑模块，而不是前台业务逻辑的重复实现
6. AI 暂时作为独立增强层，更符合当前阶段节奏

---

## 9. 影响范围

本决策将影响：

- 包结构组织
- 模块文档写法
- API 分组
- 测试分组
- 代码生成提示词
- 后续联调顺序

如后续模块边界发生实质变化，应通过新的 ADR 更新。
