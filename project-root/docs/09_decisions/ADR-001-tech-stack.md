## 1. 背景

本项目为《个性化旅游系统的设计与实现》课程设计。课程要求系统覆盖旅游推荐、旅游路线规划、场所查询、旅游日记交流等基本功能，强调核心算法必须基于自己设计的数据结构实现，并建议做 Web/App 远程系统，不做单机版；同时要求最终提供真实可用、可演示的系统。  
此外，本项目由 3 人小组完成，时间有限，需要同时兼顾需求分析、架构设计、算法实现、编码、测试、文档整理与中期 / 期末验收展示。

结合当前项目阶段，技术路线必须满足以下约束：

1. 能支撑“推荐—规划—查询—日记”主链路最小闭环；
2. 能支持多用户、数据持久化、数据导入和管理端基础维护；
3. 能让 3 人小组在课程周期内可控推进；
4. 能让 AI 工具较稳定地理解、生成和修改代码；
5. 能为后续 AI 增强能力和外部服务接入预留空间，但不把这些增强项做成当前主链路硬依赖。

---

## 2. 决策问题

需要确定以下关键技术与系统形态：

1. 系统采用 Web 远程系统还是桌面单机程序
2. 系统采用 B/S 还是 C/S 架构
3. 前后端是否分离
4. 后端采用单体还是微服务
5. 前端采用什么技术栈
6. 后端采用什么技术栈
7. 数据持久化采用什么数据库
8. 媒体资源与导入文件如何存储
9. 数据导入与清洗采用什么工具
10. AI 能力如何接入
11. 地图与外部服务能力如何处理

---

## 3. 备选方案

### 方案 A：桌面单机程序

#### 优点
- 本地开发简单
- 部署形式直接
- 不需要处理前后端分离

#### 缺点
- 不符合课程更推荐的远程访问方向
- 不利于多用户、统一数据管理和后续演示
- 不利于管理端、数据持久化和接口联调
- 与当前已形成的项目文档和模块设计方向不一致

---

### 方案 B：B/S Web 系统，前后端分离，单体后端

#### 优点
- 符合课程建议的 Web 远程访问形态
- 适合多用户、统一数据管理和管理端
- 前后端可并行开发
- 更利于接口文档、测试和答辩展示
- 对 3 人小组更可控
- 更适合当前项目的模块化和文档驱动开发方式
- 更利于 AI 工具结合本地文档进行协作开发

#### 缺点
- 前后端与接口联调成本高于单机
- 需要额外处理认证、跨域、文件上传等基础问题
- 需要同时维护前端、后端、数据库与文档

---

### 方案 C：B/S Web 系统，微服务后端

#### 优点
- 模块拆分更细
- 服务边界理论上更清晰
- 架构形式更接近大规模生产系统

#### 缺点
- 对 3 人课程设计项目明显过重
- 运维、部署、联调与故障排查成本高
- 会把时间花在工程复杂度上，而不是核心算法与功能实现上
- 与当前“先完成最小闭环，再逐步增强”的项目策略不一致

---

## 4. 决策结果

本项目采用以下技术路线。

### 4.1 系统形态
- Web 远程系统
- B/S 架构
- 前后端分离

### 4.2 前端技术栈
- Vue 3
- Vite
- Element Plus
- Vue Router
- Pinia
- Axios

### 4.3 后端技术栈
- Java 17
- Spring Boot 3
- MyBatis-Plus
- Maven
- Spring Security + JWT

### 4.4 数据与存储
- MySQL 8 作为结构化数据持久化数据库
- 本地文件存储 / 统一文件目录作为图片、视频、导入文件等非结构化资源存储

### 4.5 数据处理与导入
- Python
- pandas
- CSV / JSON 标准化数据文件
- ImportService 负责导入流程编排

### 4.6 AI 协作与增强能力
- Claude Code / OpenClaw / ChatGPT / Codex 作为 AI 协作工具
- AI 业务能力统一封装在 `AIService`
- AI 能力当前作为增强能力预留，不阻塞基础主线实现与验收

### 4.7 地图与外部服务策略
- 当前路线规划、设施查询统一基于内部 `MapService` 与自建图数据实现
- 当前不把外部地图 API 作为主路径算法依赖
- 外部地图服务和外部 AI 服务只作为后续导航增强、轨迹记录、路线回顾和多模态增强的预留能力
- 所有外部服务通过统一封装层接入，不直接散落在业务模块中

### 4.8 后端组织方式
- 单体后端
- 算法模块化封装
- 不采用微服务

---

## 5. 决策理由

### 5.1 为什么选择 Web + B/S + 前后端分离
课程和课堂讨论都强调尽量做成 Web/App 远程系统，而不是单机安装版。  
当前项目采用 Web 远程系统 + B/S 架构 + 前后端分离，更适合实现：

- 多用户访问
- 用户端与管理端统一入口
- 浏览器直接演示与验收
- 前后端并行协作
- 接口先行与文档驱动开发

### 5.2 为什么选择单体后端
当前项目是 3 人协作的课程设计，优先目标是跑通最小主链路并完成课程要求中的核心算法实现。  
单体后端更适合当前阶段：

- 架构复杂度可控
- 部署简单
- 调试链路短
- 联调成本低
- 更利于把时间集中在推荐、路径规划、查询和日记等核心功能上

### 5.3 为什么选择 Vue 3 + Element Plus
前端需要同时覆盖用户端页面和管理端页面，并支持推荐页、导航页、设施页、美食页、日记页和管理页。  
Vue 3 + Element Plus 具备以下优势：

- 社区成熟
- 页面搭建速度快
- 资料丰富
- 易于 AI 工具生成和修改
- 对课程项目的管理端和表单页尤其友好

### 5.4 为什么选择 Java 17 + Spring Boot 3 + MyBatis-Plus
后端需要承载：

- 用户认证与权限校验
- 推荐与查询逻辑
- 路线规划与图算法调度
- 周边设施与美食查询
- 日记保存、检索与文件管理
- 管理端数据维护
- 数据导入与基础支撑能力

Java + Spring Boot + MyBatis-Plus 在资料、社区、规范、文档和 AI 生成支持上都相对成熟，适合当前项目的稳定推进。

### 5.5 为什么选择 MySQL 8 + 文件存储
项目需要同时处理两类数据：

1. 结构化业务数据：用户、目的地、场所、地图节点与边、设施、美食、日记、评分等；
2. 非结构化资源：图片、视频、导入文件、地图原始 JSON / CSV 等。

因此采用：

- MySQL 8 处理结构化业务数据
- 文件存储处理媒体资源与附件

这种方式最符合当前课程项目复杂度，也与现有数据设计文档保持一致。

### 5.6 为什么选择 Python + pandas 做数据处理
项目需要做真实数据采集、清洗、转换和导入。  
相比把全部预处理逻辑都放进 Java，Python + pandas 更适合：

- 数据清洗
- CSV / JSON 转换
- 批量处理
- 原始数据预处理脚本

这样可以降低后端主应用复杂度，让导入链路更清晰。

### 5.7 为什么 AI 能力统一走 `AIService`
当前项目已经在模块、接口和架构层把 AI 能力定义为增强能力，而不是基础主线的硬依赖。  
统一封装到 `AIService` 的好处是：

- 避免 AI 调用逻辑散落到多个模块
- 便于后续切换模型或平台
- 便于 mock / stub
- 便于降级处理
- 便于把创新需求和基础主线解耦

### 5.8 为什么当前不把外部地图 API 作为基础依赖
当前路径规划模块的核心考查点是：

- 图建模
- 最短路径
- 多点路径
- 自己设计的数据结构和算法实现

因此，当前主路径算法必须基于内部 `MapService` 和自建图数据完成。  
外部地图 API 可以作为后续路线展示、轨迹记录和回顾增强能力的补充，但不能替代当前主路径算法实现。

## 6. 影响范围

本决策将直接影响以下内容：

1. 项目目录结构与仓库组织方式
2. 前后端工程初始化方式
3. 模块拆分方式
4. API 文档设计方式
5. 数据库与文件存储设计
6. 导入脚本与数据准备方式
7. AI 能力的接入方式
8. 外部服务的接入边界
9. 协作、联调、测试与验收方式

---

## 7. 代价与风险

### 7.1 代价
- 需要同时维护前端、后端、数据库、导入脚本和文档
- 接口联调工作量较大
- 文件上传、JWT 鉴权、异常处理等基础能力需要额外实现
- 前后端分离会增加早期对接口定义一致性的要求

### 7.2 风险
1. 前后端接口变动不同步，会导致联调反复返工
2. 若过早接入外部地图 API 或外部 AI 服务，可能分散主线开发精力
3. 若把增强功能误当成基础依赖，可能拖慢 5.10 前 MVP 主链路实现
4. 若后端分层不清或公共能力不抽离，后续模块会快速变乱
5. 若数据导入规范不清，真实数据准备会拖慢开发

---

## 8. 配套约束

为保证本 ADR 真正落地，项目约定：

1. 当前主链路开发必须围绕“推荐—规划—查询—日记”最小闭环推进；
2. 不得擅自把单体后端改成微服务；
3. 不得在业务模块中直接绑定具体外部地图服务 SDK 或 AI 模型厂商；
4. 外部服务统一通过抽象服务层封装；
5. 若技术栈发生实质变化，必须同步更新：
   - `tech-stack.md`
   - `architecture.md`
   - `dependency-map.md`
   - `external-integrations.md`
   - 相关 ADR

---

## 9. 当前结论

本项目正式采用：

- Web 远程系统
- B/S 架构
- 前后端分离
- 单体后端
- Vue 3 + Spring Boot 3 + MyBatis-Plus + MySQL 8
- Python + pandas 负责离线数据处理
- MySQL + 文件存储双存储结构
- `AIService` 统一承接 AI 增强能力
- `MapService` 统一承接内部图建模和路径规划能力
- 外部服务只做预留，不做当前主链路硬依赖
