# ADR-001：技术栈与系统形态选择

- 状态：Accepted
- 日期：2026-04-23
- 决策人：项目小组
- 关联文档：
  - `docs/02_architecture/tech-stack.md`
  - `docs/02_architecture/architecture.md`

---

## 1. 背景

本项目为《个性化旅游系统的设计与实现》课程设计。课程要求系统应覆盖旅游推荐、路线规划、场所查询、旅游日记等基本功能，并强调核心算法必须基于自己设计的数据结构实现；同时建议做 Web/App 远程系统，不做单机版，且最终提供真实可用系统。  
此外，本项目由 3 人小组完成，时间有限，需兼顾文档、算法、编码、测试和验收展示。

---

## 2. 决策问题

需要确定以下关键技术与系统形态：

1. 系统采用 Web 还是桌面单机
2. 架构采用 B/S 还是 C/S
3. 后端采用单体还是微服务
4. 前后端是否分离
5. 数据持久化采用什么数据库
6. 数据导入与清洗采用什么工具
7. AI 能力如何接入

---

## 3. 备选方案

### 方案 A：桌面单机程序
优点：
- 本地开发简单
- 无需前后端分离

缺点：
- 不符合课程更推荐的远程访问方向
- 不利于多用户、数据持久化、管理端和后续演示
- 与当前团队技术路线不一致

### 方案 B：B/S Web 系统，前后端分离，单体后端
优点：
- 符合课程建议的 Web 远程访问形态
- 适合多用户、统一数据管理和管理端
- 前后端可并行开发
- 更利于接口文档、测试和答辩展示
- 对 3 人小组更可控

缺点：
- 前后端与接口联调成本高于单机
- 需要额外处理认证、跨域、文件上传等基础问题

### 方案 C：B/S Web 系统，微服务后端
优点：
- 架构“看起来更大”
- 模块拆分更细

缺点：
- 对 3 人课程设计项目过重
- 运维与联调成本高
- 容易把时间花在工程复杂度上，而不是核心算法与功能上

---

## 4. 决策结果

本项目采用以下技术路线：

### 4.1 系统形态
- Web 远程系统
- B/S 架构
- 前后端分离

### 4.2 前端
- Vue 3
- Element Plus
- Axios
- Vue Router
- Pinia（如需要状态管理）

### 4.3 后端
- Java 17
- Spring Boot 3
- MyBatis-Plus
- Spring Security + JWT（登录态与权限控制）

### 4.4 数据层
- MySQL 8 作为结构化数据持久化数据库
- 文件存储用于图片、视频等媒体资源

### 4.5 数据处理与导入
- Python + pandas
- CSV / JSON 标准化数据文件
- ImportService 负责导入流程

### 4.6 AI 能力接入
- AI 能力统一封装在 `AIService`
- 当前作为增强能力保留
- 不阻塞基础主线实现与验收

### 4.7 后端组织方式
- 单体后端
- 算法模块化封装
- 不采用微服务

---

## 5. 决策理由

1. 课程更推荐 Web/App 远程系统，且不建议做单机版。
2. 课程要求多用户、数据持久化、批量导入、真实可用系统，B/S + 数据库更适合。
3. 3 人小组在当前时间和能力边界下，单体后端比微服务更容易落地。
4. 前后端分离更利于接口先行、文档驱动开发与并行协作。
5. Java + Spring Boot + MySQL 的资料、社区和 AI 生成支持相对成熟。
6. Python 更适合数据采集、清洗和导入脚本，不必强行用 Java 完成全部数据预处理。
7. 将 AI 能力统一封装而非直接散落在业务模块中，更利于后续扩展和降级处理。

---

## 6. 影响范围

本决策将直接影响：

- 项目目录结构
- 模块划分方式
- API 文档设计
- 数据库设计
- 开发环境准备
- 协作与联调方式
- 测试与验收方式

---

## 7. 代价与风险

### 7.1 代价
- 需要同时维护前端、后端、数据库和文档
- 接口联调工作量较大
- 文件上传和 JWT 权限处理需要额外实现

### 7.2 风险
- 前后端联调阶段可能暴露较多字段不一致问题
- 数据准备不足会直接影响推荐、路径和查询模块演示
- 若过早投入 AI 增强，可能拖慢基础主线

---

## 8. 当前约束下的执行原则

1. 先跑通基础主线，再补 AI 与增强能力
2. 先完成最小可演示版本，不追求复杂部署
3. 所有新增技术选型都需说明“是否真的必要”
4. 若后续技术栈有变更，必须新建 ADR 说明原因
