## 1. 背景

课程要求所有基础功能后列出的核心算法必须基于自己设计的数据结构并自己编程实现，而且这是考查重点。  
当前项目的基础主线已经明确为：

- 旅游推荐
- 路线规划
- 场所 / 周边设施查询
- 旅游日记管理与交流

同时，项目当前开发顺序也已经明确：

- `QueryService / RankService / MapService` 属于 P0 公共能力
- `SearchService` 属于 P1 增强能力
- `AIService` 属于 P2 增强能力

因此，需要冻结一版**与当前主链路和模块优先级一致**的核心算法与数据结构选型，作为后续详细设计、编码与测试的统一依据。

---

## 2. 决策问题

需要确定：

1. 推荐与查询使用什么数据结构与算法
2. 路径规划使用什么图结构与最短路径算法
3. 周边设施为什么不能按直线距离排序
4. 日记当前阶段的排序、精确查找、全文检索和压缩分别如何定位
5. 多目标路径、最短时间、AI 增强当前如何定位
6. 哪些算法属于当前 P0 必做，哪些属于后续增强

---

## 3. 决策结果总览

### 3.1 推荐模块
- 基础查找：按名称 / 类别 / 关键字 / 类型检索
- 基础排序：按热度、评分排序
- Top-K：优先输出前 K 个结果，不以完全排序作为首选方案
- 二次排序：先召回候选集，再排序
- 用户偏好：作为当前推荐输入之一，但不改变推荐的基础算法主线

### 3.2 路线规划模块
- 图结构：有向带权图
- 单目标路径：Dijkstra 为当前主实现
- 最短时间：通过边权变换实现，定位为 P1
- 多目标路径：启发式 / 近似求解，定位为 P1
- A*：作为可扩展方案，不作为当前主线必做算法

### 3.3 场所 / 周边设施查询模块
- 设施查询结果按图上可达距离 / 时间排序
- 不使用直线距离作为最终排序依据

### 3.4 日记模块
- 当前主线：按时间 / 热度等基础字段排序
- 标题精确查询：P1
- 全文检索：P1，统一归入 `SearchService`
- 无损压缩：P1 / 后续增强
- AI 草稿、图片摘要、路线回顾：P2 增强，不作为基础算法实现本体

---

## 4. 具体算法选择

## 4.1 推荐与搜索

### 当前选择
- 检索：按名称、类别、关键字、类型做候选召回
- 排序：按热度、评分做排序
- Top-K：优先输出前 K 个结果，不以全量完全排序作为第一实现方案
- 二次排序：先检索缩小范围，再排序
- 偏好参与：当前以“偏好输入参与简单打分 / 结果组织”为主，不引入复杂画像学习

### 选择理由
1. 课程明确要求推荐算法基础为排序算法，并强调用户通常只看前若干结果；
2. 课程也明确要求查询结果有多项时可继续排序；
3. “召回 + 排序 + Top-K”更符合当前 Recommend 模块文档和接口设计；
4. 当前项目规模和阶段更适合可解释、可控、便于答辩说明的规则型方案。

### 当前建议的数据结构
- 数组 / 列表：保存候选结果集
- 哈希结构：辅助精确分类与条件过滤
- 堆（优先队列）：实现 Top-K
- 可选索引结构：为后续检索增强预留

### 当前不选择
- 复杂机器学习推荐模型
- 强依赖外部推荐服务的黑箱实现
- 当前直接由 AI 决定推荐结果

---

## 4.2 路线规划

### 当前选择
- 图模型：有向带权图
- 顶点：路口、建筑物、设施点、室内点位等
- 边：道路段，带距离、理想速度、拥挤度、交通方式等权重信息
- 单目标最短路径：Dijkstra
- 最短时间：在 Dijkstra 基础上切换边权定义
- 可扩展路径：A* 作为增强方案

### 选择理由
1. 课程要求中已明确包含最短路径和多点路径；
2. 课程对最短时间策略、交通工具约束、室内导航也给出了扩展要求；
3. 当前 Route 模块设计已经明确“单目标先稳住、最短时间和多目标后补”；
4. Dijkstra 更适合当前阶段先落地、先演示、先验证；
5. A* 可以保留为增强方案，但不必在第一阶段强行实现。

### 当前建议的数据结构
- 图：有向带权图
- 邻接表 / 邻接结构：保存边关系
- 优先队列：支撑 Dijkstra
- 路径前驱记录结构：用于路径回溯

### 当前不选择
- 把外部地图服务当作主路径算法实现
- 复杂实时交通预测模型
- 一开始就追求最优多目标精确算法

---

## 4.3 多目标路径规划

### 当前选择
- 多目标路径作为 P1 增强
- 优先采用启发式 / 近似方法
- 当前优先做“目标点数量较少”的可演示版本
- 必要时加入剪枝条件限制搜索规模

### 可选思路
- 贪心最近邻
- 启发式搜索
- 状态压缩 DP（目标点数量较少时）
- 局部优化 / 路径拼接

### 选择理由
1. 课程明确把多点路径列为更复杂的问题；
2. 其组合规模增长快，不适合在当前阶段优先攻克；
3. 当前项目更需要先把单目标路径和设施可达性跑稳。

---

## 4.4 最短时间与交通工具策略

### 当前选择
- 最短时间策略作为 P1 增强
- 在图结构不变的前提下，通过改变边权实现
- 交通工具约束作为 P1 / P2 分阶段增强

### 基础思路
- `realSpeed = crowdFactor * idealSpeed`
- `timeCost = distance / realSpeed`

### 选择理由
1. 课程明确要求最短时间、拥挤度、理想速度、交通工具约束；
2. 这些内容适合在当前图结构稳定后逐步补充；
3. 不应在单目标最短距离尚未稳定前就把时间策略混入第一实现阶段。

---

## 4.5 场所查询与周边设施

### 当前选择
- 查询：按设施类别过滤
- 排序：按图上可达距离 / 时间排序
- 能力依赖：`QueryService + RankService + MapService`

### 选择理由
1. 课程明确指出不能按直线距离做最终排序；
2. 场所 / 周边设施查询的“远近”必须与真实步行路径相关；
3. 该模块天然依赖图能力，不能退化成纯数据库列表排序。

### 当前建议的数据结构
- 设施结果列表
- 图节点与设施映射关系
- 距离 / 时间结果缓存结构（可选）

---

## 4.6 日记排序、查找、检索与压缩

### 当前选择
- 当前主线：按时间 / 热度做基础排序
- 标题精确查询：P1
- 全文检索：P1，由 `SearchService` 统一承接
- 压缩存储：P1 / 后续增强
- AI 日记草稿、图片摘要、路线回顾：P2 增强

### 选择理由
1. 课程最终要求日记支持精确查询、全文检索和无损压缩；
2. 但当前 Diary 模块首先要跑通“发布—浏览—详情—按目的地查看”主线；
3. 当前依赖图也明确：Diary 主线不强依赖 `SearchService`，全文检索后补；
4. 压缩应作为增强能力，不阻塞基础日记功能落地；
5. AI 内容生成不能替代基础日记算法与数据结构实现。

### 当前建议的数据结构
- 列表 / 数组：日记结果组织
- 哈希结构：标题精确查找（后续）
- 文本索引结构：全文检索（后续）
- 压缩结构：无损压缩（后续）

---

## 5. 数据结构选择原则

1. 所有核心算法都必须能清楚对应到底层数据结构；
2. 地图问题统一使用图结构建模；
3. 推荐、排序、Top-K 必须能解释为什么不用完全排序；
4. 检索能力要区分：
   - 精确查找
   - 模糊查询
   - 全文检索
5. 外部 AI 结果不能直接作为基础算法实现本体；
6. 当前阶段优先选择：
   - 易于实现
   - 易于解释
   - 易于测试
   - 易于答辩说明
   的方案。

---

## 6. 与模块关系

- Recommend 依赖 `QueryService + RankService`
- Route 依赖 `MapService + 图结构`
- Facility 依赖 `QueryService + RankService + MapService`
- Diary 检索增强依赖 `SearchService`
- Diary 压缩增强可作为后续独立增强能力
- UserPreference 为 Recommend 和 AIService 提供输入，但不改变当前推荐算法主线
- AI 仅作为增强层，不替代基础算法实现

---

## 7. 当前边界

### 当前纳入 P0
- 推荐的召回 + 排序 + Top-K
- 单目标最短路径
- 设施按图上可达距离 / 时间排序
- 日记基础排序
- 路线图结果组织与返回

### 当前纳入 P1
- 多目标路径规划
- 最短时间增强
- 标题精确查询
- 日记全文检索
- 日记压缩存储
- 更复杂的偏好参与推荐
- 交通工具约束

### 当前纳入 P2
- AI 推荐理由解释
- AI 路线回顾
- AI 日记草稿
- 图片摘要 / 标签提取
- 多人协同规划摘要
- 多模态内容生成

---

## 8. 风险与应对

### 风险
1. 多目标路径规划复杂度高，容易拖慢主线开发
2. 数据量不足会影响推荐、Top-K 和检索演示效果
3. 图数据不真实会影响路径与设施查询可信度
4. 全文检索与压缩实现时间可能不足
5. 过早引入 AI 能力会分散算法主线实现精力

### 应对
1. 先稳定单目标路径
2. 先准备高质量样例数据与图数据
3. 多目标 / 检索 / 压缩 / 时间策略后置
4. AI 能力统一留在 P2，不阻塞基础主线
5. 所有增强算法必须能独立后接，不推翻当前主实现

---

## 9. 结论

当前阶段，项目的核心算法路线应以：

- 推荐的“召回 + 排序 + Top-K”
- 路线规划的“有向带权图 + 单目标 Dijkstra”
- 场所查询的“图上可达距离 / 时间排序”
- 日记的“基础排序 + 后续精确查找 / 全文检索 / 压缩增强”

为主线推进。

AI 增强能力、外部服务能力和更复杂的路径 / 检索 / 压缩方案，将在基础主线稳定后逐步扩展，不作为当前阶段的算法阻塞项。

## 10. 后续更新规则

1. 若核心算法主实现发生变化，必须更新本文件
2. 若某项 P1 算法被提前纳入当前主线，应同步更新本文件与 `ADR-002`
3. 若模块边界变化导致算法归属变化，应同步更新本文件与 `ADR-003`
4. 若路线规划从单目标扩展到多目标正式实现，应补充多目标算法落地版本说明
5. 若全文检索、压缩、AI 增强能力正式进入实现阶段，应补充各自的详细算法说明
