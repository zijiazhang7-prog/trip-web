# ai-usage.md

## 1. 文件用途

本文件用于说明本项目中 AI 的使用原则、适用范围、协作方式、质量要求与风险边界。

目标：

1. 让组员知道哪些环节可以让 AI 深度参与
2. 让组员知道哪些内容必须人工确认
3. 让 AI 产出能稳定沉淀为项目资产
4. 避免“AI 写了很多，但项目越来越乱”的情况

本文件不替代项目规则文件。  
AI 在本项目中的所有行为，仍必须优先遵守：

1. `project-root/README.md`
2. `project-root/AGENTS.md`
3. `project-root/coding-rules.md`
4. `project-root/security-rules.md`
5. 当前模块相关文档
6. 当前数据库与 API 定义文档

---

## 2. AI 在本项目中的定位

AI 在本项目中的定位是：

- 辅助分析
- 辅助设计
- 辅助实现
- 辅助测试
- 辅助整理文档
- 辅助任务拆解与协作

AI 不是：

- 最终决策者
- 最终验收者
- 最终正确性的保证者
- 可以替代人工 review 的黑箱工具

一句话原则：

> AI 负责提效，人负责拍板。

---

## 3. AI 可参与的环节

## 3.1 需求分析阶段

AI 可参与：

- 课设讲义转项目需求
- 用户场景拆解
- 功能清单整理
- MVP 范围收敛
- 创新点发散与筛选
- 课程设计报告相关文案初稿

要求：

1. AI 输出后必须人工确认是否符合讲义要求
2. 不允许让 AI 擅自扩展无关的大型功能
3. 需求文档必须最终落地到 `project-root/docs/01_requirements/`

---

## 3.2 系统设计阶段

AI 可参与：

- 总体架构方案整理
- 模块划分
- 模块依赖关系分析
- 数据结构与算法方案梳理
- 数据库表设计初稿
- API 定义初稿
- UI/UX 线框稿说明

要求：

1. 所有设计稿必须与项目实际技术栈一致
2. 核心算法必须能解释清楚“为什么这样选”
3. 数据库与 API 设计必须最终落地到 `project-root/docs/02_architecture/`、`project-root/docs/03_data/`、`project-root/docs/04_api/`

---

## 3.3 编码实现阶段

AI 可参与：

- 初始化前端工程骨架
- 初始化后端工程骨架
- 单模块最小实现
- DTO / VO / Entity / Mapper / Controller / Service 骨架生成
- 前端页面骨架生成
- 小范围重构
- bug 定位与修复
- 代码审查
- 单元测试补充

要求：

1. 不允许一次性生成整个系统
2. 必须按模块渐进开发
3. 先读 docs，再动代码
4. 所有代码改动都必须人工 review
5. 修改代码后，要同步检查相关 docs 是否需要更新

---

## 3.4 测试与验收阶段

AI 可参与：

- 测试计划初稿
- 测试用例设计
- 黑盒/白盒/单元测试检查项整理
- bug 复盘
- 验收清单生成
- 演示脚本整理

要求：

1. 测试计划和测试用例必须结合当前实现状态
2. 不允许只写形式化测试文档而不验证实际功能
3. 测试结果必须落地到 `project-root/docs/06_testing/`

---

## 3.5 答辩与交付阶段

AI 可参与：

- README 整理
- 项目计划书整理
- 用户使用说明初稿
- 项目亮点提炼
- 答辩讲稿生成
- 老师提问预测
- PPT 文案润色

要求：

1. 所有最终交付材料必须人工改写和确认
2. 不允许直接把 AI 原文不加筛选地交上去
3. 表述必须贴合你们真实实现情况，不夸大

---

## 4. AI 不适合直接替代人工的内容

以下内容不能只依赖 AI，必须人工重点确认：

1. 课程要求理解是否准确
2. 创新点是否真实可做
3. 核心算法是否正确实现
4. 数据库表结构是否合理
5. API 是否与前后端真实联调一致
6. 路线规划、图建模、Top-K 等核心逻辑是否正确
7. 安全相关逻辑是否符合要求
8. 最终答辩材料是否与真实项目一致
9. 最终提交版代码是否可运行
10. 最终提交版文档是否完整

---

## 5. AI 使用的基本工作流

## 5.1 文档类任务工作流

1. 先确认任务属于哪个文档类别
2. 先让 AI 读取已有资料
3. 让 AI 生成结构化初稿
4. 人工筛选与改写
5. 写入正式 docs 目录
6. 如影响其他文档，继续同步更新

---

## 5.2 代码类任务工作流

1. 明确当前只做哪个模块
2. 先让 AI 阅读：
   - `project-root/README.md`
   - `project-root/AGENTS.md`
   - 本地规则文件
   - 当前模块文档
   - 当前模块代码
3. 先让 AI 输出“修改计划”
4. 再允许 AI 执行代码修改
5. 人工 review 改动
6. 运行测试或最小验证
7. 同步更新 docs

---

## 5.3 bug 修复类工作流

1. 提供报错信息
2. 提供触发步骤
3. 提供预期结果
4. 让 AI 先定位原因
5. 先要求最小修复方案
6. 再实施修改
7. 修复后要求 AI 说明：
   - 根因
   - 修改文件
   - 验证方法

---

## 5.4 答辩材料类工作流

1. 先整理真实实现情况
2. 再让 AI 生成讲稿/PPT 文案
3. 人工删掉不真实、不贴合项目的表述
4. 最终按自己的表达习惯再润色一次

---

## 6. AI 输出的落地要求

AI 生成的内容不能长期停留在聊天记录里。  
只要内容有长期价值，就必须沉淀为项目资产。

建议落地方式：

- 需求分析类 → `project-root/docs/01_requirements/`
- 架构设计类 → `project-root/docs/02_architecture/`
- 数据与数据库类 → `project-root/docs/03_data/`
- API 类 → `project-root/docs/04_api/`
- 模块说明类 → `project-root/docs/05_modules/`
- 测试类 → `project-root/docs/06_testing/`
- 交付类 → `project-root/docs/07_delivery/`
- prompt / AI 协作类 → `project-root/docs/08_ai/`
- 决策类 → `project-root/docs/09_decisions/`

一句话：

> 重要内容必须进仓库，不能只留在对话框里。

---

## 7. AI 输出的质量要求

所有 AI 输出都至少应满足以下标准：

### 7.1 可理解
- 语言清楚
- 结构明确
- 术语统一

### 7.2 可执行
- 给出具体步骤
- 给出具体文件
- 给出具体模块或命令
- 不只讲空泛原则

### 7.3 可验证
- 能说清怎么验收
- 能说明如何测试
- 能指出风险与待确认项

### 7.4 可维护
- 能沉淀为正式文档
- 能被其他组员复用
- 能被后续 AI 继续读取

---

## 8. AI 使用时的禁止事项

1. 禁止一次性让 AI 生成整个系统
2. 禁止不看项目文档就直接改代码
3. 禁止把 AI 原文直接当最终交付材料
4. 禁止让 AI 擅自推翻已确定的项目决策
5. 禁止多个 AI 同时无约束地改同一模块
6. 禁止把外部通用规范直接替代本地规则
7. 禁止对高风险改动不做人工 review
8. 禁止把未验证的 AI 输出当成事实
9. 禁止在 prompt 中暴露真实密钥、密码、token
10. 禁止在项目中堆积大量一次性无复用 prompt

---

## 9. 不同工具在本项目中的建议定位

### Codex CLI
主要负责：

- 阅读本地仓库与 docs
- 做最小代码修改
- 做代码审查与 bug 修复
- 跑命令、联调、补测试

### GPT
主要负责：

- 需求分析
- 架构设计说明
- 文档生成与润色
- 项目计划与答辩材料整理
- 第二意见

### OpenClaw / 调度型 agent
主要负责：

- 任务拆解
- 多步流程组织
- prompt 与工具串联
- 项目级协作支持

## 10. 组员协作时的 AI 使用原则

### 10.1 公共 prompt 与个人 prompt 分离
- 稳定、高频、可复用的 prompt 进入 `prompt-library.md`
- 临时、实验性 prompt 不直接进入公共库

### 10.2 公共结论必须沉淀
只要是组内共识，就不能只存在聊天记录中。  
必须落地到：
- docs
- prompt-library
- decisions
- 任务清单

### 10.3 同一模块尽量由同一人主导
可以让 AI 辅助多人协作，但一个模块最好始终有明确负责人。  
否则很容易出现：
- prompt 风格不一致
- 代码风格不一致
- 文档与实现不一致

---

## 11. 如何记录 AI 参与情况

建议对以下内容做简单记录：

1. AI 参与了哪些阶段
2. 使用了哪些工具
3. 哪些内容是 AI 生成初稿
4. 哪些内容经过了人工改写
5. 哪些功能是 AI 辅助实现的

记录方式可以是：

- 写入项目计划书
- 写入 README
- 写入课程设计报告相关章节
- 写入 `project-root/docs/08_ai/` 说明文件

目的不是“炫工具”，而是让项目过程更可说明、可回溯。

---

## 12. 当前阶段推荐优先使用的 AI 场景

当前最值得优先使用 AI 的环节：

1. 需求梳理与项目计划整理
2. 系统架构与模块依赖分析
3. 数据结构与数据库设计整理
4. API 文档整理
5. 工程骨架初始化
6. 单模块实现
7. 代码审查与 bug 修复
8. 测试计划与测试用例整理
9. README、项目报告、答辩讲稿整理

当前不建议最优先把时间花在：

1. 过度花哨的 prompt 工程
2. 同时试很多工具
3. 反复切换模型
4. 让 AI 直接写完整系统
5. 追求“全自动开发”

---

## 13. 一句话原则

> AI 在本项目中的作用，是让流程更快、更清楚、更可执行；不是替代你们理解项目、实现项目和解释项目。
