## 1. 文件用途

本文件用于统一管理“个性化旅游系统”课程设计项目的常用提示词，供以下场景复用：

- 项目管理与协作
- 需求分析与文档编写
- 系统设计与架构设计
- 代码生成与代码修改
- 代码审查与调试
- 测试与验收
- 答辩与交付材料整理

本文件面向以下工具使用：

- Codex CLI
- GPT
- Claude / 其他 coding agent
- OpenClaw / 其他任务调度型智能体

## 2. 使用规则

### 2.1 使用顺序

每次使用提示词时，建议遵循以下顺序：

1. 先贴“总控提示词”
2. 再贴“具体任务提示词”
3. 最后补充“当前限制条件”

### 2.2 默认限制条件模板

建议在每次具体任务后追加一段限制条件：

```text
当前限制条件：
- 只允许修改指定目录
- 不要一次性改太多文件
- 不要擅自引入大型新依赖
- 不要擅自修改数据库表结构
- 只做最小可运行版本
- 若发现文档与代码冲突，先列出冲突点，不要静默偏离
````

### 2.3 默认输出格式

除非任务另有说明，否则统一要求 AI 按以下格式输出：

```text
一、任务理解
二、任务拆解
三、执行结果
四、验收标准
五、风险与待确认项
六、下一步建议
```

## 8. 代码审查与调试类提示词

### 8.1 代码审查

```text
请对当前改动进行代码审查。

审查重点：
1. 是否符合当前项目结构规范
2. 是否有明显的空指针、异常吞掉、命名混乱、重复逻辑
3. Controller 是否写了过多业务逻辑
4. DTO / VO / Entity 是否混用
5. 是否有 SQL 注入、硬编码密钥、日志泄露敏感信息等风险
6. 是否和 docs 中的设计不一致

输出格式：
- 主要问题
- 次要问题
- 建议修改项
- 可以暂时接受的问题
不要直接修改代码，先只审查。
```

### 8.2 bug 定位与修复

```text
我现在遇到一个 bug，请你帮助定位并修复。

请按这个顺序工作：
1. 先阅读相关代码
2. 分析最可能的原因
3. 给出最小修复方案
4. 再实施修改
5. 最后说明：
   - 根因是什么
   - 你改了哪些文件
   - 怎么验证修复是否成功

补充信息：
- 报错信息：<粘贴报错>
- 触发步骤：<描述操作步骤>
- 预期结果：<你本来希望发生什么>
```

### 8.3 git diff 审查

```text
请基于当前工作区的改动，对 git diff 做一次 review。

重点检查：
1. 是否改动范围过大
2. 是否混入无关修改
3. 是否有格式化噪音
4. 是否有遗漏的文档更新
5. 是否适合作为一次独立提交

输出：
- 建议保留的改动
- 建议拆分出去的改动
- 建议补充的内容
不要直接修改。
```

### 8.4 小范围重构

```text
请对 <模块名> 做一次小范围重构。
```

```text
5. 测试文档是否需要补充

输出格式：
- 一致的部分
```

## 12. 使用建议

### 12.1 日常使用推荐顺序

适合日常开发的顺序：

1. 项目总控提示词
2. 具体任务提示词
3. 当前限制条件
4. 再补充模块名、报错信息、目标文件路径

### 12.2 适合 Codex CLI 的特点

Codex CLI 更适合：

- 先读仓库和文档
- 在限制条件下做最小改动
- 做代码审查、bug 修复、diff review
- 做前后端骨架初始化
- 做小范围模块实现

### 12.3 使用提醒

- 一次只做一个明确任务
- 不要把多个大任务混在一个 prompt 里
- 先让 AI 分析，再让 AI 修改
- 重要改动前先要求它列“修改计划”
- 重要改动后让它列“修改了哪些文件”
