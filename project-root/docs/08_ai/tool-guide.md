- 作用：写每个工具怎么用、适合做什么。

## 1. 文件用途

本文件用于说明本项目中不同 AI 工具的定位、适用场景、输入要求、输出要求和使用边界。

目标：
1. 让组员知道“什么任务优先用什么工具”
2. 减少工具混用带来的重复劳动
3. 保证 AI 输出更稳定、更可复用
4. 避免把所有任务都丢给同一种工具处理

---

## 2. 总体原则

### 2.1 工具不是越多越好
本项目不追求“同时使用很多 AI 工具”，而是追求：
- 分工清晰
- 上下文统一
- 输出可落地
- 结果可检查

### 2.2 先看任务类型，再选工具
优先按任务性质选工具，而不是按个人习惯选工具。

### 2.3 本地规则优先
无论使用哪个 AI 工具，都必须优先遵守：
1. `project-root/README.md`
2. `project-root/AGENTS.md`
3. `project-root/docs/coding-rules.md`
4. `project-root/docs/security-rules.md`
5. 当前模块相关文档

### 2.4 AI 输出不是最终结果
所有 AI 输出都必须经过人工确认，尤其是：
- 核心算法
- 数据库设计
- 接口定义
- 安全相关逻辑
- 最终提交材料

## 3. 项目中推荐使用的工具分工

## 3.1 Codex CLI

### 定位
本项目中的主力本地 coding agent。

### 更适合做的任务
- 阅读本地仓库和 docs 文档
- 初始化前后端工程骨架
- 按要求实现单个模块
- 做最小范围代码修改
- 根据报错修 bug
- 做代码审查
- 对 git diff 做 review
- 按本地规则执行命令、联调、格式化和测试

### 不适合优先做的任务
- 大范围开放式需求发散
- 很长的课程报告成稿润色
- 太早期、太模糊的问题定义
- 没有上下文时直接让它“做完整系统”

### 推荐使用方式
1. 先让它读项目上下文
2. 先让它分析，不立即改代码
3. 先要求它给出最小修改计划
4. 再让它执行代码修改
5. 修改后要求它说明：
   - 改了哪些文件
   - 为什么这么改
   - 如何验证

### 推荐搭配
- 代码实现：Codex CLI
- 代码审查：Codex CLI + 人工复核
- 文档校对：Codex CLI + GPT 第二意见

---

## 3.2 GPT

### 定位
本项目中的通用分析与文档辅助工具。

### 更适合做的任务
- 需求拆解
- 架构方案对比
- 项目计划书整理
- README / 报告 / PPT 文案生成与润色
- 解释算法思路
- 帮助提炼项目亮点
- 给出第二意见
- 帮助梳理老师可能提问的问题

### 不适合优先做的任务
- 在没有代码上下文的情况下直接大改仓库
- 高风险本地自动化修改
- 替代项目本地规则做裁决

### 推荐使用方式
1. 适合在问题还比较模糊时先用 GPT 整理思路
2. 适合在项目文档阶段做结构化输出
3. 适合作为 Codex CLI 输出后的复核工具

### 推荐搭配
- 需求分析：GPT 主力
- 文档写作：GPT 主力
- 架构与算法解释：GPT 主力
- 代码修改：GPT 辅助，不作唯一执行工具

---

## 3.3 OpenClaw / 任务调度型 agent

### 定位
项目级任务调度与协作型工具。

### 更适合做的任务
- 任务拆解
- 多步骤工作流组织
- 管理 prompt 和 skill 的调用逻辑
- 整理 AI 使用记录
- 串联多个上下文文件
- 辅助协作流程执行

### 不适合优先做的任务
- 替代本地 coding agent 写大量代码
- 在缺少稳定 skill 时承担主开发任务
- 直接决定项目级关键方案

### 推荐使用方式
1. 用于组织任务，而不是盲目产出代码
2. 用于做“调度层”，不是直接替代所有工具
3. 适合配合 prompt-library、tool-guide、context-files 一起使用

---

## 3.4 其他 coding agent（如后续接入）

### 定位
专项补充工具。

### 适合做的任务
- 某类特定代码生成
- 某类专项重构
- 补充测试代码
- 做第二套 review 视角

### 使用原则
- 不作为默认主力
- 只有在已验证效果不错时才纳入公共流程
- 必须遵守本地规则文件

## 4. 按任务类型推荐工具

| 任务类型 | 优先工具 | 次选工具 | 说明 |
|---|---|---|---|
| 任务拆解 | GPT / OpenClaw | Codex CLI | 更适合先做结构化整理 |

| 数据结构与算法说明 | GPT | Codex CLI | 先讲清，再落代码 |
| 数据库设计 | GPT | Codex CLI | 文档先行，代码后补 |
| API 设计 | GPT | Codex CLI | 先定规范，再实现 |
| 初始化前端骨架 | Codex CLI | 其他 coding agent | 更适合本地工程改动 |
| 初始化后端骨架 | Codex CLI | 其他 coding agent | 更适合本地工程改动 |
| 单模块实现 | Codex CLI | 其他 coding agent | 要求最小改动、可执行 |
| bug 修复 | Codex CLI | GPT | 先定位再改 |
| 代码审查 | Codex CLI | GPT | 双重审查更稳 |
| 测试计划/测试用例 | GPT | Codex CLI | 先生成结构，再补实现 |

## 5. 推荐工作流

## 5.1 文档类任务工作流
1. 先用 GPT 梳理结构
2. 输出 Markdown 初稿
3. 人工检查
4. 如需与代码对齐，再让 Codex CLI 检查一致性
5. 定稿后写入 docs

## 5.2 代码类任务工作流
1. 先让 Codex CLI 阅读上下文文件
2. 先分析，不立即动代码
3. 给出最小修改计划
4. 实施修改
5. 跑格式化、静态检查和测试
6. 人工 review
7. 同步 docs

## 5.3 bug 类任务工作流
1. 提供报错信息和触发步骤
2. 让 Codex CLI 定位原因
3. 先给最小修复方案
4. 修复并验证
5. 再用 GPT 检查是否还有隐藏风险

## 6. 使用每种工具前应准备的输入

### Codex CLI 需要优先给的内容
- 当前任务目标
- 限制条件
- 当前模块名
- 相关 docs 路径
- 具体要改的目录范围
- 是否允许执行命令

### GPT 需要优先给的内容
- 当前问题背景
- 当前阶段（需求/架构/开发/测试/答辩）
- 希望输出的文体
- 目标长度
- 现有材料

### OpenClaw 需要优先给的内容
- 当前项目状态
- 任务链路
- 可调用工具范围
- 是否需要多步执行
- 输出要落到哪里

## 7. 高风险场景提醒

以下场景必须人工重点复核：

1. 核心算法实现
2. 路线规划与图建模逻辑
3. 数据库表结构变更
4. 认证、权限、JWT、安全相关逻辑
5. 文件上传与媒体处理
6. 涉及删除、大范围重构、批量重命名的改动
7. 最终提交版文档和答辩材料

---

## 8. 工具使用边界

### 不建议做的事
1. 不要多个 AI 同时在同一模块上直接并行写代码
2. 不要让 AI 跳过 docs 直接改实现
3. 不要把一次性临时 prompt 当作长期规则
4. 不要把外部规范网站当作唯一执行规则
5. 不要把“AI 说可以”当作最终结论

### 建议坚持的事
1. 所有共识落地为 Markdown
2. 所有高频 prompt 进入 prompt-library
3. 所有工具分工写清楚
4. 所有项目级规则写进仓库
5. 重要决策写进 `project-root/docs/09_decisions/`

## 9. 本项目当前推荐默认组合

### 默认推荐
- 本地代码实现：Codex CLI
- 需求/文档/汇报：GPT
- 任务拆解/调度：OpenClaw（如实际启用）
- 最终决策：人工

### 一句话版本
- **Codex CLI 负责“做”**
- **GPT 负责“想、讲、整理”**
- **OpenClaw 负责“串流程”**
- **人负责“拍板和验收”**
