# 数据来源说明（Data Source）

## 1. 文件用途
本文件用于说明本项目的数据来源、数据类型、采集方式、清洗方式、导入方式和使用边界，为以下工作提供依据：

- 数据库设计
- 数据导入与初始化
- 推荐、查询、路径规划、日记等功能的数据准备
- 测试数据准备
- 后续数据维护与更新

本文件关注的是“数据从哪里来、怎么进入系统、当前阶段怎么用”，不替代字段级数据字典和数据库表结构说明。

相关文档：

- `project-root/docs/03_data/er-model.md`
- `project-root/docs/03_data/data-dictionary.md`

## 2. 数据来源设计目标

本项目的数据来源设计目标如下：

1. 尽量使用真实数据或接近真实场景的数据；
2. 满足课程对数据规模的基本要求；
3. 支持推荐、查询、路线规划、周边设施、美食、日记等核心功能；
4. 区分“真实业务数据”“样例数据”“测试数据”“系统运行数据”；
5. 支持批量导入和后续增量维护。

根据课程要求，系统应尽量接近真实景区和校园场景，建议爬取真实地图数据；同时需要满足景区/校园数量、建筑物数量、设施数量、边数和用户数等数据规模要求。:contentReference[oaicite:3]{index=3}

## 3. 数据来源总体分类

本项目当前将数据来源分为 4 类：

### 3.1 基础静态数据
指系统运行前预先准备好的相对稳定的数据，包括：
- 目的地基础信息
- 场所 / 建筑物信息
- 服务设施信息
- 地图节点与边
- 美食基础信息

### 3.2 半结构化外部数据
指从网页、公开资料、地图文件或整理文档中获取，再经过清洗转换后导入系统的数据，包括：
- 景点介绍
- 校园建筑分布
- 设施类别信息
- 道路图数据
- 美食信息
- 评分与评论摘要

### 3.3 样例与测试数据
指为了前期联调、演示和测试而人工构造的数据，包括：
- 小规模目的地样例
- 小规模地图样例
- 测试用户
- 测试日记
- 测试评分

### 3.4 运行时业务数据
指系统运行过程中由用户或管理员产生的数据，包括：
- 用户账号信息
- 用户偏好
- 路线记录
- 日记内容
- 日记媒体
- 日记评分
- 管理端维护数据

## 4.5 地图节点与边数据（最关键）

### 数据内容
- 地图节点

- 边距离
- 理想速度
- 拥挤度

### 具体数据来源

#### 首选来源 1：OpenStreetMap 原始区域数据
适合采集：
- 道路
- 路口
- 建筑物
- 步行通道
- 周边设施 POI
- 区域内空间关系

具体可用方式：
- 下载目标区域的 OSM 原始数据
- 提取道路与节点
- 转换为 `map_node` / `map_edge` 导入格式

OpenStreetMap 官方提供开放地图数据；其生态中也有区域数据下载与搜索能力。:contentReference[oaicite:11]{index=11}

#### 首选来源 2：学校 / 景区导览图人工建模
适合场景：
- 校园内部步行道路
- 景区内部游览路线
- 地下车库或室内导航简化场景

做法：
- 从导览图上抽取关键节点
- 将节点定义为：
  - 交叉口
  - 建筑出入口
  - 设施点
- 将边定义为：
  - 道路
  - 通道
  - 楼梯
  - 电梯
- 再人工补充：
  - 距离
  - 拥挤度
  - 理想速度

#### 备选来源 3：高德 / 百度开放平台地理服务
适合补充：
- 地理编码
- 坐标校准
- POI 地址解析
- 周边位置搜索

说明：
- 如果你们需要对人工整理的地点名称做经纬度映射，这类平台更方便做辅助定位与校准。:contentReference[oaicite:12]{index=12}

### 当前阶段建议
- 不要一开始就追求 200+ 边的大地图全自动导入；
- 先选 1~2 个核心目的地，手工 + 半自动结合建出高质量有向图；
- 先保证：
  - 单目标路径规划
  - 周边设施可达距离
  - 地图展示
  能完整跑通；
- 再逐步扩展边数和目的地数量。

课程和课堂讨论都明确提出：地图应尽量接近真实景区和校园，建议爬取真实地图数据，并按有向图建模。:contentReference[oaicite:13]{index=13} :contentReference[oaicite:14]{index=14}

## 6. 数据采集与清洗原则

### 6.1 真实性原则
尽量优先使用真实公开数据、真实地图信息和真实导览资料；无法直接获取的字段，可在合理范围内人工补充或采用规则生成，但应在文档中说明。

### 6.2 结构化原则
所有最终进入系统的数据，必须转换为统一的结构化格式，例如：
- CSV
- JSON
- SQL 初始化脚本

### 6.3 最小可用原则
前期不追求一次把所有数据做满，优先保证：
- 有代表性的真实目的地
- 可运行的地图数据
- 可演示的日记和美食数据
- 能支撑推荐、规划、查询、日记主线闭环

### 6.4 可追溯原则
每类核心数据最好记录：
- 来源类别
- 采集日期
- 清洗方式
- 导入批次

## 7. 合规与使用边界说明

### 7.1 当前使用边界
本项目当前主要用于课程设计、系统实现、功能演示和测试验证，不用于商业化发布。

### 7.2 数据使用原则
1. 优先使用公开可获取的信息；
2. 对于爬取或整理的外部数据，应只保留项目所需字段；
3. 不在仓库中提交真实密钥、敏感账号信息和隐私数据；
4. 用户运行时数据在演示阶段以测试账号和样例内容为主。

### 7.3 媒体资源说明
图片、视频等资源优先使用：
- 小组自制测试素材
- 可公开使用的演示素材
- 系统运行中用户自行上传的内容
