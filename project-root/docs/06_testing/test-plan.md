# 测试计划（Test Plan）

## 1. 文档用途
本文件用于说明个性化旅游系统项目的测试目标、测试范围、测试方法、测试环境、测试数据、分工、进度安排、进入/退出准则和测试产出物，为后续测试执行、缺陷修复、验收准备和答辩展示提供依据。

本文件关注的是：
- 为什么测
- 测什么
- 不测什么
- 怎么测
- 什么时候测
- 谁来测
- 测完以什么标准判断当前阶段通过

本文件是 `docs/06_testing/` 目录中的总控文档，后续应与以下文档配套使用：
- `test-cases.md`
- `bug-log.md`
- `test-report.md`
- `performance-notes.md`

---

## 2. 文档定位与当前版本说明

### 2.1 当前版本定位
当前版本为**课程设计开发前期的初版测试计划**，主要作用是：
1. 提前明确测试范围和策略；
2. 避免所有功能做完后才临时想“怎么测”；
3. 为后续 `test-cases.md`、`bug-log.md` 和 `test-report.md` 提供上位约束；
4. 服务于课程设计最终提交中“应用范例执行结果及测试情况说明”的材料准备。

### 2.2 当前适用阶段
- 文档完善阶段
- 开发阶段
- 联调阶段
- 验收准备阶段

### 2.3 后续更新原则
随着模块逐步落地，本文件应从“计划版”不断补充为“执行版”，并同步更新测试范围、测试进度和退出准则。

---

## 3. 测试目标

本项目测试的总体目标如下：

1. 验证个性化旅游系统的核心主线功能可用；
2. 验证推荐、路线规划、周边设施查询、旅游日记、美食推荐等功能满足预期；
3. 验证前后端、数据库、文件上传、导入流程之间的联动正确；
4. 验证核心算法在典型输入、边界输入和异常输入下结果正确；
5. 验证系统在课程验收场景下能够稳定演示；
6. 发现并记录功能缺陷、逻辑缺陷、接口缺陷、数据缺陷和安全性问题；
7. 为后续答辩提供测试依据和缺陷修复依据。

当前测试计划默认不覆盖以下范围：

5. 未进入当前版本范围的创新需求细节。

## 7. 测试类型与测试方法

## 7.1 黑盒测试
目的：
- 从用户视角验证输入与输出是否符合预期
- 不关注代码内部实现

适用对象：
- 推荐结果
- 搜索结果
- 路线规划结果
- 周边设施结果
- 日记发布 / 浏览流程
- 美食查询流程
- 管理端基本操作

重点：
- 功能正确性
- 页面交互正确性
- 接口结果是否符合预期
- 错误提示是否清晰

---

## 7.2 白盒测试
目的：
- 从程序内部逻辑视角验证分支、路径和边界处理是否正确

适用对象：
- 排序逻辑
- 图算法逻辑
- 检索逻辑
- 评分聚合逻辑
- 状态判断逻辑
- 异常处理逻辑

重点：
- 分支覆盖
- 路径覆盖
- 极端输入处理
- 逻辑完整性

---

## 7.3 单元测试
目的：
- 验证核心 service、算法和工具逻辑的最小正确性

适用对象：
- RecommendService
- RoutePlanService
- FacilityService
- FoodService
- DiaryService
- MapService
- RankService
- SearchService（若实现）
- ImportService（若实现）

重点：
- 正常输入样例
- 边界输入样例
- 异常输入样例

---

## 7.4 集成测试 / 联调测试
目的：
- 验证多个模块组合后是否能够协同工作

适用对象：
- 前端 + 后端 + 数据库
- 文件上传 + 日记发布
- 路线规划 + 周边设施查询
- 推荐 + 目的地详情 + 日记入口
- 导入流程 + 前台展示

重点：
- 数据流是否打通
- 接口字段是否一致
- 前后端状态是否一致
- 数据库是否正确持久化

---

## 7.5 回归测试
目的：
- 修复问题后验证旧功能未被破坏

适用对象：
- 所有修复过 bug 的模块
- 涉及公共能力修改的模块
- 涉及表结构或接口字段变更的模块

重点：
- 原问题是否消失
- 是否引入新问题
- 是否破坏既有功能

---

## 7.6 安全性基础测试
目的：
- 验证项目级安全规范中的关键点是否被满足

适用对象：
- 登录与权限接口
- 文件上传
- 外部接口调用
- 全局异常处理
- 参数校验

重点：
- 未登录访问是否被拦截
- token 无效是否正确处理
- 管理端接口是否做角色校验
- 文件类型、大小、数量限制是否生效
- 是否存在明显 SQL 拼接和异常泄漏问题

---

## 8. 测试策略

## 8.1 整体策略
本项目采用“**先单元，后联调；先核心主线，后增强模块；先功能正确，后体验优化**”的测试策略。

### 具体原则
1. 每个核心模块先完成最小单元测试；
2. 前置依赖完成后再做模块联调测试；
3. 主线功能优先测试；
4. bug 修复后必须做回归测试；
5. 每次较大改动后，同步更新测试记录。

---

## 8.2 按模块推进的测试策略
### 第一阶段：基础与依赖测试
- 数据库连接
- 建表
- 导入基础数据
- Auth 基础流程
- 统一返回体与错误码
- 文件上传

### 第二阶段：核心业务模块测试
- Recommend
- Route（单目标）
- Facility
- Diary 基础版

### 第三阶段：增强与扩展测试
- Food
- Route 多目标
- Diary 检索与评分
- Admin
- AI 功能（若实现）

---

## 8.3 缺陷优先级处理策略
建议把 bug 分成三类：

### P0 缺陷
会阻塞核心主线演示或导致系统无法使用的问题，例如：
- 登录失败
- 路线规划结果明显错误
- 日记无法发布
- 接口报错导致页面打不开

### P1 缺陷
不阻塞主线，但影响功能完整性或结果正确性的问题，例如：
- 某类筛选不生效
- 某排序结果异常
- 管理端某项操作失效

### P2 缺陷
不影响核心功能，但影响体验或非核心增强功能的问题，例如：
- 文案不一致
- UI 展示细节问题
- 某些边缘交互异常

## 10. 测试数据准备

## 10.1 基础数据要求
测试数据应至少覆盖：
- 10 个测试用户
- 2~5 个真实样例目的地
- 每个目的地 20 个以上场所 / 建筑点
- 若干设施点
- 至少 1~2 个可完整导航的地图样例
- 20~50 篇样例日记
- 少量美食样例数据

## 10.2 数据来源
测试数据来源包括：
- 初始化导入数据
- 小组人工构造样例数据
- 运行时生成数据
- 文件上传测试素材

## 10.3 数据准备原则
1. 优先使用结构清晰的小样本真实数据；
2. 先保证主线演示可跑通；
3. 再逐步扩展到更接近课程要求的数据规模；
4. 测试账号、测试日记、测试图片应便于复用。

## 12. 进入准则（Entry Criteria）

以下条件满足后，可正式进入当前阶段测试：

1. 对应模块代码已提交并可运行；
2. 相关数据库表结构已落地；
3. 相关接口已可被调用；
4. 当前模块前置依赖已完成；
5. 已准备至少一组可复现测试数据；
6. 对应模块文档已基本齐全。

---

## 13. 退出准则（Exit Criteria）

以下条件满足后，可认为当前阶段测试基本通过：

1. 当前阶段 P0 测试项全部执行；
2. P0 缺陷已修复或已有明确解释与兜底；
3. 核心主线可完整演示；
4. 主要接口返回稳定；
5. 数据持久化结果正确；
6. 测试记录和 bug 记录已更新。

## 18. 风险与应对

## 18.1 主要测试风险
1. 需求和接口仍在变动，导致测试用例反复修改；
2. 基础数据不足，导致推荐、导航、设施查询测不充分；
3. AI 功能不稳定，影响增强测试结果；
4. 多人并行开发导致联调阶段 bug 集中爆发；
5. 测试工作被拖到最后，来不及回归和整理报告。

## 18.2 应对建议
1. 先以 P0 主线功能建立最小测试集；
2. 模块一落地就做最小测试，不等全部开发完成；
3. bug 修复后立即做回归；
4. AI 功能单独看作增强项，不阻塞基础主线验收；
5. 每周同步测试进度和主要缺陷。
